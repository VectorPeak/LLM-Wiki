# 工程支架设计

## 方法定位

工程支架设计是为智能体构造外部工作系统的方法。它把“让模型更努力”改写为“让环境更清晰、反馈更及时、错误更可恢复、完成标准更可验证”。

## 基本结构

1. 地图：短入口文件指向更深的真实信息来源。
2. 状态：进度日志、计划、任务清单和 git 历史保存当前工作。
3. 约束：架构边界、lint、测试和类型系统限制不良路径。
4. 验证：端到端测试、浏览器自动化、日志和指标给出事实反馈。
5. 交接：每轮结束时留下下一轮可读的结构化状态。
6. 垃圾回收：定期清理漂移、重复、陈旧文档和技术债。

## 操作原则

- 先找最小可行支架，不要一开始堆复杂多智能体系统。
- 每个支架组件都应回答一个明确问题：它弥补了模型哪项稳定性缺口？
- 当新模型能力提升时，重新检查哪些支架仍然承重。
- 主观标准也应尽量转化为可评估准则。

## 对 LLMWiki 的应用

LLMWiki 的支架包括 `Schema`、入口页、来源页、索引、hot cache、日志和 lint 报告。后续 `/Ingest` 不应只产出摘要，而应持续加强这个支架，使知识库越来越容易被后续 agent 读取和维护。
