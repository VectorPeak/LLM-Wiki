# 如何让长程编码智能体稳定推进复杂任务

## 问题

长程编码智能体经常需要跨多个小时、多个会话和多个上下文窗口推进复杂任务。如何降低一口气做太多、遗忘上下文、过早宣布完成、自评偏宽松和架构漂移这些风险？

## 简答

不要只依赖模型记忆和长提示词。应把任务环境设计成智能体可读、可验证、可交接的工程系统：用短入口文档做地图，用结构化计划和功能清单定义目标，用 git、日志和进度文件保存状态，用测试、浏览器自动化和可观测性提供反馈，用 lint 和架构规则强制不变量，用生成器-评估器循环补足自评缺陷。

## 详细解释

长程智能体的核心矛盾是：任务状态比上下文窗口大，且模型会在长期运行中产生漂移。解决方法不是把所有信息塞进一次对话，而是把关键状态外化。

从 Anthropic 的两篇文章看，最小可行支架包括 initializer agent、coding agent、功能清单、进度文件、git 历史、启动脚本和端到端验证。更复杂的任务可以引入 planner、generator、evaluator，把规划、实现和质量判断分离。

从 OpenAI 的 Codex 实践看，代码仓库本身应成为记录系统。`AGENTS.md` 不是百科全书，而是地图；真正的知识应落在结构化文档、执行计划、架构规则、质量报告、测试和可观测性里。

## 仍需补充

- 如何把这套支架迁移到非代码任务，例如论文阅读、个人知识管理和业务研究。
- LLMWiki 应采用单 agent 摄取，还是引入独立 lint/evaluator agent。
- 对不同模型能力，哪些支架仍然必要，哪些会成为过度工程。
