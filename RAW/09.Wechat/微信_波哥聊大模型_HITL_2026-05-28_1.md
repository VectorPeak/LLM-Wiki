## 0x01. Human in the Loop 的工程现实
> 发布日期：2026-05-26  
> 原文链接：[Human in the Loop 的工程现实](https://mp.weixin.qq.com/s/S-AegmF2c_nXO_fXU4bHgQ)

做 AI 落地这几年，我的体感是大家对"自动化"的执念正在退烧。前几年评价一个系统好不好，常见的标准是人工干预率有多低——越接近全自动越显得本事大。现在再看那批吹得最狠的全自动方案，活下来的没几个。反倒是老老实实把人安排在关键节点上的产品，跑得又稳又久。这个转向背后的逻辑，就是今天要拆的 Human in the Loop(下面简称 HITL)。

### 它为什么是刚需，而不是过渡方案
很多人把 HITL 理解成模型还不够强时的拐杖，等模型够强了就能扔掉。这个判断我不认同。HITL 要解决的几个问题，是结构性的，模型变强并不会让它们消失。

第一个结构性问题是 **校准(calibration)失效**。大模型的输出概率和它实际的正确率长期对不上。一个模型告诉你它有 95% 的把握，真实正确率可能只有 70%。更麻烦的是，模型在它最该露怯的地方往往最自信——分布外(out-of-distribution)的输入、训练数据里稀疏的长尾场景，恰恰是它一边出错一边给高置信度的重灾区。你没法靠模型自报的置信度来决定哪些该信、哪些该查，这个缺口必须由外部判断来补，人就是那个外部判断。

第二个是 **长尾的不可穷尽**。医疗、金融、法律这些领域，真正咬人的从来都是那 1% 的边角案例。这些案例数量少、形态杂、还在不断冒新的，你没法提前把它们全塞进训练集。模型在这些场景上的表现天然不稳定，而这些场景往往又是后果最严重的。把人放在这条尾巴上，是目前唯一可行的兜底手段。

第三个是 **责任主体的法律要求**。欧盟《AI 法案》对高风险系统的人工监督是写进条款的强制要求，监管者要的是一个能为决策负责、能随时叫停的自然人。再准的模型也无法承担法律责任，这一环不是技术能优化掉的，是制度层面的硬约束。

至于价值层面，HITL 真正的杠杆在数据飞轮上。模型上线后跑出来的真实输入、加上人对它输出的每一次纠正，构成了远比离线标注集珍贵的信号——它带着分布漂移的信息、带着真实用户的偏好、带着模型当前最薄弱环节的定位。RLHF 把 GPT 调成 ChatGPT，DPO、RLAIF 这些后来的路线本质都是在更高效地把人的判断蒸馏进模型。谁的飞轮转得快，谁的模型迭代就快，这是现在产品层面最硬的壁垒。

### Claude Code、Codex 这批产品把 HITL 做到了什么程度
AI 编程是 HITL 设计最成熟、也最值得拆解的领域，因为它把"人机协作"的颗粒度做得极细。这里单独展开讲。

**早期的补全式交互(Copilot** **范式)**，HITL 的粒度是 token 级和行级的。模型在光标处给出灰色的建议补全，你按 Tab 接受、继续打字就拒绝。这套设计的高明之处在于把"人的批准"无缝嵌进了打字这个动作里——你不需要额外停下来审核，接受和拒绝就是你正常编码节奏的一部分。代价是它只能处理局部建议，管不了跨文件的大改动。

**Agent 式的编程工具(Claude Code、Codex** **CLI** **这类)面对的是一个量级更难的问题**：模型现在要自主读文件、改多个文件、跑命令、装依赖、执行测试。这时候人的批准不能再嵌进打字里了，因为模型的动作已经脱离了你的键盘。它们的解法很值得研究，我拆几个关键设计：

其一是 **操作分级与权限门**。这类工具把动作分成不同风险等级：读文件、跑只读命令这种基本无害，可以放开；写文件、删文件、跑 shell 命令、装包这种有副作用、可能不可逆的，默认要停下来等你确认。Claude Code 的权限系统就是围绕这个建的——你可以预先授权某类安全操作(比如允许它随便读文件、跑测试)，把审批的注意力省下来集中在真正危险的动作上。这直接对应后面要讲的"自动化自满"难题：不是每一步都问，只在该问的地方问。

其二是 **diff 作为人机交接的界面**。模型改代码不会直接覆盖，而是先生成 diff 给你看——红绿对照，改了哪几行清清楚楚。这个设计解决的是 HITL 里最隐蔽的成本：人理解模型意图的成本。你不需要读懂模型的全部推理，扫一眼 diff 就能判断这个改动对不对。把模型的"想法"压缩成一个人能在几秒内消化的视觉差异，是这套交互的精髓。

其三是 **计划-执行分离与中断能力**。复杂任务上，这类 Agent 会先列出执行计划给你过目，再分步执行；执行过程中你随时能打断、能纠偏、能让它回退。Codex 在云端跑长任务时也是类似思路，把一个长链条切成可观测、可干预的若干段，而不是黑箱跑完给你个结果。这背后是对一个核心矛盾的处理：Agent 自主性越高，人能插手的窗口越少，所以必须主动在链条上预留出可中断的节点。

其四，也是最容易被忽略的，是 **沙箱与回滚兜底**。再好的审批机制也有看走眼的时候，所以这类工具普遍跑在受限的执行环境里、依赖 Git 这种版本控制做安全网。哪怕一个危险操作漏过了人的审核，还有"撤销"这层保险。HITL 不是只靠人这一道防线，它是人工审批 + 权限隔离 + 可回滚的纵深防御。

把这几条放一起看，你会发现 Claude Code、Codex 这批产品真正解决得好的，是 **在高自主性和可控性之间找到了一个动态平衡**：安全的事不打扰你，危险的事拦下来问你，看走眼了还能退回去。这套范式现在正被整个 Agent 行业借鉴，做企业级 Agent 的团队基本都在抄这套作业——审批节点、操作分级、沙箱执行，已经成了 Agent 安全设计的事实标准。

### 其他领域各家在做什么
跳出编程看，HITL 在别的方向也铺得很开，只是形态不同。

做基础模型的 OpenAI、Anthropic，重头戏在对齐环节，养着大量标注和评测团队给模型输出排序、挑刺。Anthropic 的 Constitutional AI 想用一套成文原则加少量人工反馈让模型学会自我约束，这是在试图降低对人工标注的绝对依赖，但人定原则、人来校验这两环依然在。

客服和内容审核走的是置信度路由那一套：模型有把握的自动处理，灰色地带转人工。内容审核尤其依赖这套机制，海量内容机器先过滤，模糊的丢给人工复核队列。

国内这边，阿里、百度、字节的大模型平台和 Agent 产品都内置了类似的审批与审核机制，金融、医疗这些强监管行业的落地方案里，"AI 初筛 + 专家复核"几乎是标配，一些做垂直行业大模型的公司直接把这个流程当核心卖点在卖。

### 真正难的地方在哪
概念人人会讲，落地全是细节，每个细节都能让项目翻车。

**自动化自满(automation complacency)是头号杀手。** 当模型大多数时候都对，人会逐渐放松，最终退化成无脑点"同意"。你本想用人兜底，结果人成了橡皮图章，监督形同虚设。航空和医疗领域早就为此付出过代价——系统太可靠，反而让监督它的人丧失了警觉。Claude Code 的预授权机制其实就是在对抗这个问题：与其让人面对一长串确认提示逐渐麻木，不如把安全操作放开，让人的注意力集中在真正高风险的少数动作上。怎么设计交互让人保持真实判断而非机械确认，是这个领域最核心的难题。

**介入时机的把握没有通用解。** 介入太密，AI 的效率优势归零，人累得想砸电脑；介入太疏，兜底失效，风险照漏。这个度高度依赖对业务的理解。更底层的难点又回到了校准——你得知道模型这个置信度该信几分，才能定阈值，而校准本身就是个未解的硬骨头。

**人机交接的认知成本被严重低估。** 模型甩给人一个半成品，人得先逆向理解模型怎么想的、错在哪、怎么补，这个成本有时高过自己从头做。前面说的 diff 界面就是针对这个痛点的解法，但 diff 只适用于代码，换到别的场景——一段法律意见、一份诊断建议——怎么把模型的判断压缩成人能快速接住的形式，仍然是个开放问题。

**人本身就是噪声源。** 我们默认人的判断是金标准，可人会疲劳、有情绪、标注口径因人而异、同一个人不同时段都可能给出不同结果。加了人不等于变客观，可能只是把模型的偏见换成了人的偏见，还更难追溯。

**成本与规模的剪刀差。** 人是最贵、最难弹性扩张的资源。业务量一涨，待审项指数级膨胀，要么砸钱堆人，要么调高自动化比例放松把关，两头都疼。用最少的人工守住最关键的风险，是所有 HITL 系统永恒的优化命题。

### 求职者可以动手做哪些项目
国内从业者想往这个方向走，光懂概念没用，得有拿得出手的东西。下面按难度排，挑适合自己的做。

**入门：带审核回路的生成工具。** 做一个 AI 生成文案/邮件的应用，重点是审核界面——模型出草稿，用户可改、可批、可打回，所有编辑动作落库。难度不高，但能体现你对 HITL 核心交互的理解，面试好讲。

**进阶：置信度路由分类系统。** 做一个文本分类器(工单分类、内容审核都行)，亮点是引入置信度阈值：高置信自动处理，低置信进人工队列。重点研究置信度怎么校准、阈值定在哪 ROI 最高。这个项目直击"何时该让人介入"这个核心命题，比单纯做分类器有深度得多。能把校准曲线、阈值与人工成本的权衡曲线画出来，就很能打。

**进阶：数据飞轮 Demo。** 做一个能从人工反馈持续学习的闭环：收集用户每次的纠正，定期用这些数据微调或更新模型，再用指标证明模型确实变好了。这对应 HITL 最有价值的一环，做出来很加分，关键是要有前后对比的硬数据。

**高阶：带审批节点的 Agent(最对口)。** 用 LangGraph、AutoGen 这类框架做一个多步任务 Agent，在敏感操作前插人工审批节点——把 Claude Code 那套思路自己实现一遍：操作分级、计划展示、等待确认、处理否决、支持回滚。能把"哪些操作要审批、计划怎么呈现给人、人否决了怎么优雅处理"这几个问题想透并实现出来，含金量很高，几乎是直接对标现在最热的岗位。

**算法向加分项：RLHF** **/ DPO 小复现。** 在小模型小数据集上把 RLHF 或 DPO 流程跑通，能讲清楚人类偏好数据怎么转化成模型能力。做模型对齐的团队很看重这个理解。

几句实在话。项目贵精不贵多，一个能完整讲透"为什么介入、在哪介入、介入后数据怎么用"的项目，胜过五个浅尝辄止的 Demo。一定要有数据：加了人工环节后准确率提升多少、错误率降多少、人工成本几何，这些数字面试官最买账。把踩过的坑讲出来——你怎么发现自动化自满、怎么调的阈值、diff 怎么设计的——这些经验比代码本身更能体现水平。

### 最后
AI 这波浪潮里有个常见的误读，以为终点是把人彻底替换掉。在一线做落地的人慢慢都看清了，至少可见的将来，更靠谱的形态是人和 AI 分工——模型负责快和广，人负责准和稳，中间用精心设计的交接机制把两者缝起来。Claude Code 这批产品的价值，恰恰在于把这个缝合做得足够优雅，让协作的摩擦小到可以忽略。

HITL 是一种成熟的工程态度：承认能力有边界，然后在边界处把人的位置安排好。能把这件事想透、做扎实的人，未来几年不缺机会。
