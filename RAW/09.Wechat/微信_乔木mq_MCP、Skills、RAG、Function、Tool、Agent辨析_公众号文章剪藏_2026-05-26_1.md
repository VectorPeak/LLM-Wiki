## 一. MCP、Skills、RAG、Function、Tool、Agent辨析

> 发布日期：2026-03-26  
> 原文链接：[MCP、Skills、RAG、Function、Tool、Agent辨析](https://mp.weixin.qq.com/s/Br6kFaWfKZyiC7hvZXXynQ)

RAG 是让模型博学，Function/Tool 是让模型手长，Skills 是让模型懂行，Agent 是让模型自立，而 MCP 则是让它们能轻松互联的通用语言。
 1. RAG (Retrieval-Augmented Generation，检索增强生成)
 (1)概念：RAG 是一种架构模式，旨在解决大模型“知识滞后”和“胡说八道”的问题。它的核心逻辑是：在模型回答问题之前，先去外部知识库(如数据库、文档、互联网)里“查资料”，把查到的相关信息作为背景材料喂给模型，让模型基于这些真实资料生成答案。
 (2)起源：由 Facebook AI Research (现 Meta AI) 在 2020 年正式提出论文。它是为了解决纯参数化模型(只靠训练数据记忆)无法获取最新信息且难以追溯来源的痛点。
 (3)通俗理解：就像开卷考试。模型不再是死记硬背(训练数据)，而是允许它先翻书(检索)，再根据书上的内容写答案(生成)。
 2. Function / Tool / Skills (函数/工具/技能)
 这三个词在很多时候是混用的，但在细微处有区别：
 (1)Function (函数)：最底层的代码概念。指一段具体的、可执行的代码逻辑，比如“计算加法”、“查询天气 API”。在大模型语境下，特指模型可以调用的具体代码接口。
 (2)Tool (工具)：更偏向工程化的称呼。指被封装好、带有描述信息(告诉模型这个函数是干嘛的、参数是什么)的 Function。模型通过“工具调用”机制来使用它。
 (3)Skills (技能)：更偏向 Agent(智能体)语境的高层概念。一个“技能”可能由多个 Tool 组合而成，或者代表一种复杂的能力(如“搜索并总结新闻”)。它强调的是模型能“做什么事”，而不仅仅是调什么函数。
 (4)起源：Function Calling 功能最早由 OpenAI 在 2023 年推出，随后 Google、Anthropic 等纷纷跟进。Skills 的概念则随着 AutoGPT 等 Agent 框架的兴起而流行。
 (5)通俗理解：Function 是具体的“扳手”；Tool 是贴了标签、放在工具箱里随时待命的“扳手”；Skills 是“修车”这项能力，可能需要用到扳手、千斤顶等多个 Tool。
 3. Agent (智能体)
 (1)概念：Agent 是一个能够感知环境、进行推理、规划任务，并主动调用工具(Tools/Skills)来达成目标的自主系统。它不仅仅是回答问题，而是能“做事”。
 (2)起源：概念源自人工智能早期研究，但在大模型时代(2023 年后)爆发。ReAct (Reasoning + Acting) 框架的提出是其现代形态的重要里程碑。
 (3)通俗理解：如果说大模型是“大脑”，那 Agent 就是“大脑 + 手脚”。大脑负责思考，手脚(工具)负责执行。Agent 能自己决定先迈哪只脚，遇到障碍怎么绕开。
 4. MCP (Model Context Protocol，模型上下文协议)
 (1)概念：MCP 是一个开放的标准化协议，旨在统一大模型与外部数据源、工具之间的连接方式。它被称为“AI 界的 USB-C 接口”。
 (2)起源：由 Anthropic 公司在 2024 年底正式推出并开源。目的是解决之前每个模型连接每个工具都需要单独写代码(M×N 问题)的碎片化困境。
 (3)通俗理解：以前想让你的 AI 连上谷歌硬盘、本地数据库、Slack，得分别写三套不同的代码。有了 MCP，就像有了通用的 USB-C 口，只要设备支持 MCP 协议，插上就能用，不用重复造轮子。

#### 区别与联系
 1. RAG vs Function/Tool：一个是“读”，一个是“做”
 (1)核心区别：RAG 侧重于“获取信息”(Read-Only)，目的是丰富模型的上下文，让它回答更准确；Function/Tool 侧重于“执行动作”(Write/Action)，目的是让模型去操作外部系统，比如发邮件、改数据库、下单买东西。
 (2)联系：一个强大的 Agent 通常同时具备两者。它先用 RAG 查阅公司文档了解政策，再用 Tool 去执行报销流程。
 2. Function vs Tool vs Skills：粒度与视角的不同
 (1)粒度差异：Function 是代码级的最小单元；Tool 是应用级的封装单元(包含元数据)；Skills 是业务级的能力单元(可能是多个 Tool 的编排)。
 (2)使用场景：开发者在写代码时关注 Function；在配置模型提示词时关注 Tool 的描述；在设计 Agent 任务时关注 Skills 的规划。
 (3)演变趋势：随着 Agent 越来越复杂，单纯的 Function Calling 正在向更高级的 Skills 编排演进，强调多步骤任务的自动拆解与执行。
 3. Agent vs 其他所有：它是“指挥官”
 (1)包含关系：Agent 是顶层概念。一个 Agent 内部通常会集成 RAG 模块(用于查资料)、Tool/Skills 模块(用于干活)。
 (2)主动性差异：RAG 和 Function Calling 通常是被动的(用户问一句，它查一下或调一下)；而 Agent 具有主动性，它可以自主规划多步任务(例如：“帮我策划一次旅行”，Agent 会自动拆解为查机票、订酒店、排日程等一系列动作，并循环执行直到完成)。
 4. MCP vs 其他所有：它是“连接器”标准
 (1)定位不同：RAG、Function、Agent 都是技术架构或功能模式；而 MCP 是一个通信协议标准。
 (2)作用对象：MCP 不直接产生智能，它是为了让 Agent 更方便、标准化地连接到 RAG 所需的数据源和 Function 所在的工具服务器。
 (3)价值点：在没有 MCP 之前，构建 Agent 连接外部工具需要大量定制开发(胶水代码)；有了 MCP，这部分工作被标准化了，开发者可以专注于 Agent 的逻辑本身。

#### 核心对比表
 (1)RAG：解决“不知道”的问题。关键词：检索、知识库、上下文、准确性。
 (2)Function/Tool：解决“做不到”的问题。关键词：API 调用、执行、操作、参数。
 (3)Skills：解决“复杂任务”的问题。关键词：能力封装、多步编排、业务语义。
 (4)Agent：解决“自主性”的问题。关键词：规划、推理、循环、目标导向。
 (5)MCP：解决“连接难”的问题。关键词：标准化协议、通用接口、解耦、生态。

#### 实战中的协作流程
 想象你要构建一个“自动客服 Agent”：
 (1)底层连接：你使用 MCP 协议，快速接入了公司的知识库服务器(提供 RAG 数据)和工单系统服务器(提供 Tool)。
 (2)知识增强：当用户提问时，Agent 触发 RAG 流程，通过 MCP 从知识库检索最新产品政策。
 (3)能力调用：如果用户需要退款，Agent 识别意图，调用“退款 Skill”，该 Skill 内部通过 MCP 调用工单系统的“创建工单 Tool”。
 (4)自主规划：如果退款失败，Agent 自主决定下一步是“查询用户等级”还是“转接人工”，并循环执行，直到问题解决。

#### 未来趋势
 (1)标准化：MCP 这类协议将成为基础设施，让工具连接像插 U 盘一样简单。
 (2)技能化：Function Calling 将逐渐演变为更语义化的 Skills 市场，开发者可以直接订阅“搜索技能”、“绘图技能”。
 (3)代理化：单纯的问答机器人将消失，取而代之的是能独立闭环完成任务的 Agent，RAG 和 Tool 将成为 Agent 的标配器官。
