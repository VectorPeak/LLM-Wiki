## 0x01. Agent 工程师需要了解的开发技术栈——FastAPI
> 发布日期：2026-05-27  
> 原文链接：[Agent 工程师需要了解的开发技术栈——FastAPI](https://mp.weixin.qq.com/s/FiQ62lFah4rCfva0UjXF4g)

这是一期新连载，记录Agent工程师需要了解的开发技术栈

### 写在前面：这篇文章想讲清楚什么
 无论你是要给 Agent 暴露一组工具接口，还是要搭一个对外提供能力的服务，FastAPI 几乎都是 Python 世界里的默认选择。但很多人用它时只是"照葫芦画瓢"，并不清楚一个请求进来后框架到底做了什么。

 这篇文章只围绕一条主线展开：

 **一个 HTTP 请求从进入到返回，FastAPI 是如何组织代码、处理路由、并把它需要的资源准备好送进来的？** 我们会顺着这条线，依次回答下面几个递进的问题——每一个问题都建立在上一个之上：

 代码怎么组织？—— APIRouter 与主应用的关系

 一条路由是怎么"诞生"的？—— @router.post() 装饰器背后做了什么

 路由需要的资源怎么送进来？—— Depends() 依赖注入与依赖链

 这些资源活多久？——依赖的作用域(每请求 / 单例)

 函数参数里的值到底从哪来？——Path、Query、Body、BaseModel、Depends 的区别

 框架没帮我封装的东西怎么拿？—— Request 对象与 app.state

 把以上全部串成一个能跑的小项目

 读完你应该能建立这样一个心智模型： **路由负责"声明需要什么"，依赖系统负责"准备好并注入"，而 FastAPI 把这一切自动编排起来。**### 1. 代码怎么组织：APIRouter 与主应用的关系**要解决的问题**：项目一大，所有路由堆在一个文件里就没法维护了。怎么按业务拆分？

 最朴素的写法是把所有接口都写在 app 上：
```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/users")
async def list_users(): ...

@app.post("/orders")
async def create_order(): ...
### ……几十上百个接口全堆在这里
```
这显然不可持续。FastAPI 的解法是 APIRouter ——你可以把它理解成一个 **迷你的、可拆卸的 FastAPI 应用**。它自己不能跑，但能把一组相关路由打包起来，最后统一"插"到主应用上。

 于是项目就能按业务模块拆开：
```text
FastAPI app (主应用)
   ├── include_router(users_router,  prefix="/users")
   ├── include_router(orders_router, prefix="/orders")
   └── include_router(items_router,  prefix="/items")
```
每个模块写在自己的文件里：
```python
### routers/users.py
from fastapi import APIRouter

router = APIRouter(
    prefix="/users",        # 这组路由的统一前缀
    tags=["users"],         # 在 /docs 文档里分组显示
)

@router.get("/")
asyncdef list_users():
    return [{"id": 1, "name": "Alice"}]

@router.get("/{user_id}")
asyncdef get_user(user_id: int):
    return {"id": user_id}
```
再到主应用里"插"上去：
```python
### main.py
from fastapi import FastAPI
from routers import users

app = FastAPI()
app.include_router(users.router)   # 挂载子路由
```
#### 关于 prefix 的叠加
 prefix 可以在 APIRouter() 和 include_router() 两处分别设置，最终路径是它们的拼接。这一点初学时很容易搞混，看一张表就清楚了：

| APIRouter() 里 | include_router() 里 | 函数上的路径 | 最终访问路径 |
| :--- | :--- | :--- | :--- |
| prefix="/users" | — | /{user_id} | /users/{user_id} |
| — | prefix="/api/v1" | /{user_id} | /api/v1/{user_id} |
| prefix="/users" | prefix="/api/v1" | /{user_id} | /api/v1/users/{user_id} |

 prefix 不能以 / 结尾，写 /users 而不是 /users/。

 **承上启下**：现在我们知道路由分散在各个 router 里、由 prefix 拼出最终路径。但每一条路由——比如那个 @router.get("/{user_id}") ——挂上去的瞬间，框架到底替我们做了什么？这就是下一节。

### 2. 一条路由怎么"诞生"：装饰器背后做了什么
 **要解决的问题**： @router.post("/items") 看着只是登记了个地址，但它显然做了不止这些——参数校验、文档、序列化都没让我手写，是谁干的？

 答案是： **这个装饰器在你启动应用时就把这些活全干了。**当 Python 执行到 @router.post("/items") 这一行时，依次发生：

- **读取函数签名**：根据你写的类型注解，推断每个参数该从哪里取值(路径？查询串？请求体？依赖？)。

- **打包成一条路由**：把"HTTP 方法 + 路径 + 处理函数 + 响应模型"封装成一个 APIRoute 对象。

- **基于 Pydantic 生成校验逻辑**：请求进来时自动校验、不合法就返回 422。

- **登记进路由表**：加入 router.routes，等待挂载。

- **生成 OpenAPI 文档**： /docs 和 /redoc 里的交互式文档就来自这里。

 也就是说，你写的这一个装饰器，等价于"注册路由 + 写参数校验 + 写序列化 + 写接口文档"的全部工作量：
```python
from fastapi import APIRouter, status
from pydantic import BaseModel

router = APIRouter()

class ItemIn(BaseModel):
    name: str
    price: float

class ItemOut(BaseModel):
    id: int
    name: str
    price: float

@router.post(
    "/items",
    response_model=ItemOut,               # 声明返回模型 → 自动序列化 + 字段过滤
    status_code=status.HTTP_201_CREATED,  # 成功返回 201
    summary="创建一个商品",                # 文档里的接口标题
)
asyncdef create_item(item: ItemIn):
    # 进到函数体时，item 已经是校验通过的 ItemIn 实例
    return {"id": 1, **item.model_dump()}
```
注意那个参数 item: ItemIn ——框架是怎么知道要从请求体里把 JSON 解析成 ItemIn 再塞进来的？更进一步，如果我需要的不是请求里的数据，而是一个数据库连接、一个当前登录用户，又该怎么"塞"？

 这就触及了 FastAPI 最核心的机制：依赖注入。

#### 2.3 资源怎么送进来：Depends() 依赖注入与依赖链
 **要解决的问题**：路由函数经常需要一些"不在请求里、但处理时必须有"的东西——数据库会话、当前用户、配置。这些怎么优雅地拿到，又不让每个函数都重复一遍获取逻辑？

 FastAPI 的思路非常漂亮： **函数只负责"声明我需要什么"，框架负责"准备好并送进来"。** 这就是依赖注入。

 你用 Depends(某个可调用对象) 来声明依赖，框架会在请求到达时自动调用它、拿到返回值，再注入到你的参数里。

#### 先看最简单的例子
 把"分页参数"这种多个接口都要用的逻辑抽成一个依赖：
```python
from fastapi import Depends, APIRouter

router = APIRouter()

def pagination_params(skip: int = 0, limit: int = 10):
    return {"skip": skip, "limit": limit}

@router.get("/items")
async def list_items(page: dict = Depends(pagination_params)):
    return {"page": page}
```
访问 /items?skip=20&limit=5，框架先调用 pagination_params，把结果注入到 page。逻辑只写一次，所有接口复用。

#### 关键：依赖可以依赖别的依赖(依赖链)
 真正强大的地方在于， **一个依赖自己也能声明依赖**，从而串成一条链。最经典的就是鉴权：先从请求头拿 token，再用 token 换出用户。
```python
from fastapi import Depends, Header, HTTPException

### 第 1 层：从请求头取出 token
def get_token(authorization: str = Header(...)):
    ifnot authorization.startswith("Bearer "):
        raise HTTPException(status_code=401, detail="认证头格式错误")
    return authorization.removeprefix("Bearer ")

### 第 2 层：依赖上一层，用 token 解析出当前用户
def get_current_user(token: str = Depends(get_token)):
    if token != "secret-token":
        raise HTTPException(status_code=401, detail="token 无效")
    return {"username": "alice"}

### 路由：只声明"我要当前用户"，完全不关心 token 怎么来的
@router.get("/me")
asyncdef read_me(user: dict = Depends(get_current_user)):
    return user
```
框架的求解顺序是 **自底向上**： get_token → get_current_user → 路由函数。任何一层抛异常，整条链立即中断、直接返回错误。路由函数因此变得极其干净——它对"token 如何解析"一无所知，只是声明"给我当前用户"。

 同一个请求里，相同依赖默认只求解 **一次**(结果被缓存)。需要每次重新调用时用 Depends(dep, use_cache=False)。**承上启下**：依赖帮我们把资源送了进来。但随之而来一个问题——这个数据库会话，是每个请求新建一个，还是全程共用一个？答案不同，行为天差地别。这就是"作用域"。

#### 2.4 这些资源活多久：依赖的作用域
 **要解决的问题**：数据库会话应该"每个请求一个，用完就关"；而机器学习模型这种重资源，应该"整个程序只加载一次"。同样是依赖，怎么控制它们的生命周期？

 FastAPI 依赖默认是**「每请求」作用域**：每来一个请求就重新求解一次。理解这一点，两类典型需求就都能对上号了。

#### 每请求一个：用 yield 管理生命周期
 数据库会话是最典型的例子——每个请求开一个独立会话，请求结束自动关闭：
```python
def get_db():
    db = SessionLocal()      # 每个请求都新建
    try:
        yield db             # 把会话注入给路由用
    finally:
        db.close()           # 请求结束后必定执行清理
```
诀窍在 yield：它之前的代码是"准备"，之后的代码是"清理"。框架会在请求处理完成后自动回来执行 finally，所以你永远不用担心忘记关连接。

#### 全局只要一个：单例作用域
 如果资源很"重"(配置、连接池、ML 模型)，希望整个进程只创建一次，可以用 lru_cache 做成单例：
```python
from functools import lru_cache

@lru_cache               # 整个进程只构造一次，之后永远返回同一个
def get_settings():
    return Settings()    # 假设读取环境变量的配置对象

@router.get("/config")
async def read_config(settings: Settings = Depends(get_settings)):
    return {"app_name": settings.app_name}
```
(更适合全局重资源的方式是挂到 app.state，第 6 节会讲。)

#### 对照表
| 作用域 | 怎么实现 | 典型用途 |
| :--- | :--- | :--- |
| 每请求新建 | 普通函数 / yield 依赖 | 数据库会话、当前用户 |
| 单请求内缓存 | 默认行为 | 同一请求多处复用同一结果 |
| 全局单例 | @lru_cache / app.state | 配置、ML 模型、连接池 |

 **承上启下**：到这里，"资源怎么来、活多久"都清楚了。回过头看路由函数的参数列表——里面既有来自请求体的 item，又有来自依赖的 user 和 db，还可能有路径里的 id 、问号后的 page。它们长得都差不多，框架怎么区分？这是下一节要彻底厘清的。

#### 2.5 参数的值到底从哪来：五种来源的区别
 **要解决的问题**：路由函数的参数看上去都一样，但有的来自 URL 路径，有的来自查询串，有的来自请求体，有的根本不来自请求。怎么区分、怎么声明？

 FastAPI 通过「类型注解 + 默认值标记」来判断每个参数从请求的哪个部分取值。一张表先建立全局印象：

| 标记 | 取值来源 | 典型场景 |
| :--- | :--- | :--- |
| Path | URL 路径里的占位符 | /users/{user_id} 里的 user_id |
| Query | URL 问号后的查询串 | ?page=2&size=10 |
| Body | 请求体里的单个标量 | 把一个孤立的数值放进 JSON 体 |
| BaseModel | 请求体里的结构化对象 | 整个 JSON 反序列化成模型 |
| Depends | 依赖求解的返回值 | 当前用户、数据库会话 |

 把它们放在一个函数里对比，区别一目了然：
```python
from fastapi import APIRouter, Path, Query, Body, Depends
from pydantic import BaseModel

router = APIRouter()

class Item(BaseModel):
    name: str
    price: float

def get_current_user():
    return {"username": "alice"}

@router.put("/items/{item_id}")
asyncdef update_item(
    item_id: int = Path(..., gt=0),          # 来自路径 /items/123
    notify: bool = Query(False),             # 来自查询串 ?notify=true
    item: Item = ...,                        # 来自请求体(整个 JSON 对象)
    discount: float = Body(0.0, ge=0, le=1), # 来自请求体的单个字段 discount
    user: dict = Depends(get_current_user),  # 来自依赖，而非请求本身
):
    return {
        "item_id": item_id,
        "notify": notify,
        "item": item,
        "discount": discount,
        "updated_by": user["username"],
    }
```
一句话记忆：

 **Path**：路径里的"坑位"，必填。

 **Query**：问号后的"过滤条件"，常带默认值。

 **BaseModel / Body**：装在请求体里的"数据"——前者是整个对象，后者是单个值。

 **Depends**： **不来自请求**，是框架替你准备好的东西。

 **承上启下**：前五节覆盖了 FastAPI 帮你自动处理的几乎所有情况。但偶尔你需要"绕过"这套封装，直接摸到底层的原始请求，或者读取一个全局共享的资源。这就要请出 Request 和 app.state。

#### 2.6 框架没封装的怎么拿：Request 对象与 app.state
 **要解决的问题**：有时我想要的东西没法用参数声明——比如客户端 IP、原始请求体、或者一个所有请求共享的全局连接池。这些去哪拿？

#### Request：底层原始请求
 在参数里声明一个 Request，就能拿到 Starlette 的原始请求对象，访问那些没被显式声明的信息：
```python
from fastapi import Request, APIRouter

router = APIRouter()

@router.get("/inspect")
asyncdef inspect(request: Request):
    return {
        "method": request.method,                     # 请求方法
        "url": str(request.url),                       # 完整 URL
        "client_host": request.client.host,            # 客户端 IP
        "headers": dict(request.headers),              # 请求头
        "query_params": dict(request.query_params),    # 查询参数
        # await request.json()  读取原始请求体
        # request.state         本次请求生命周期内的临时存储
    }
```
它常用于中间件、读取原始 body、或在中间件里写 request.state 、再到路由里读出来。

#### app.state：整个应用共享的容器
 app.state 挂在应用实例上， **整个程序生命周期都存在、所有请求共享** ——正好用来放第 4 节提到的全局单例资源。最佳实践是配合 lifespan (启动/关闭钩子)：
```python
from contextlib import asynccontextmanager
from fastapi import FastAPI, Request

@asynccontextmanager
asyncdef lifespan(app: FastAPI):
    # 启动时：创建全局资源
    app.state.db_pool = await create_db_pool()
    yield
    # 关闭时：释放
    await app.state.db_pool.close()

app = FastAPI(lifespan=lifespan)

@router.get("/ping-db")
asyncdef ping_db(request: Request):
    pool = request.app.state.db_pool   # 通过 request.app.state 访问
    return {"ok": True}
```
一句话区分： request.state 活在 **单个请求**里，请求结束就没了； app.state 活在**整个应用** 里，人人共享。

#### 2.7 全部串起来：一个能跑的迷你项目
 到这里，所有零件都讲完了。现在用一个「带鉴权的笔记 API」把它们组装成整体，你会看到前六节的每个概念各就各位：

 APIRouter + prefix 叠加 → 路由组织(第 1 节)

 装饰器自动校验、序列化、生成文档(第 2 节)

 **一个鉴权依赖**： get_token → get_current_user 的依赖链(第 3 节)

 **一个数据库会话依赖**： yield 实现的每请求会话(第 3、4 节)

 Path / Query / Body / BaseModel / Depends 五种参数同台演出(第 5 节)

 app.state + lifespan + Request 管理与读取全局资源(第 6 节)

 为了让你 **不装数据库就能直接跑通**，下面用内存字典模拟数据库，写法刻意贴近真实的 SQLAlchemy SessionLocal，之后替换成真库几乎零改动。

#### 项目结构
```text
mini_notes/
├── main.py            # 主应用：lifespan + 挂载路由
├── database.py        # 模拟数据库 + 会话依赖
├── dependencies.py    # 鉴权依赖(依赖链)
└── routers/
    └── notes.py       # 笔记路由
```
#### database.py — 数据库会话依赖(第 4 节的"每请求作用域")
```python
### database.py
"""内存字典模拟数据库；真实项目替换为 SQLAlchemy 的 SessionLocal 即可。"""

_FAKE_DB: dict[int, dict] = {}
_COUNTER = {"id": 0}

class FakeSession:
    def __init__(self):
        self.store = _FAKE_DB

    def add_note(self, owner: str, content: str) -> dict:
        _COUNTER["id"] += 1
        note = {"id": _COUNTER["id"], "owner": owner, "content": content}
        self.store[note["id"]] = note
        return note

    def list_notes(self, owner: str, skip: int, limit: int) -> list[dict]:
        mine = [n for n in self.store.values() if n["owner"] == owner]
        return mine[skip: skip + limit]

    def close(self):
        pass   # 真实场景这里归还连接到连接池

def get_db():
    db = FakeSession()
    try:
        yield db        # 注入给路由
    finally:
        db.close()      # 请求结束自动清理
```
#### dependencies.py — 鉴权依赖(第 3 节的"依赖链")
```python
### dependencies.py
from fastapi import Header, HTTPException, Depends, status

_FAKE_TOKENS = {
    "token-alice": {"username": "alice"},
    "token-bob": {"username": "bob"},
}

def get_token(authorization: str = Header(...)) -> str:
    ifnot authorization.startswith("Bearer "):
        raise HTTPException(status.HTTP_401_UNAUTHORIZED, "认证头应为 'Bearer <token>'")
    return authorization.removeprefix("Bearer ")

def get_current_user(token: str = Depends(get_token)) -> dict:
    user = _FAKE_TOKENS.get(token)
    if user isNone:
        raise HTTPException(status.HTTP_401_UNAUTHORIZED, "无效的 token")
    return user
```
#### routers/notes.py — 业务路由(第 1、2、5 节同台)
```python
### routers/notes.py
```

上一篇我们拆解了 FastAPI——而你可能已经注意到，那些路由函数几乎都写成async def。这不是巧合：现代 Python 服务端框架(FastAPI、各类 LLM SDK、Agent 框架)几乎都建立在asyncio之上。一个 Agent 在等待大模型返回、等待工具调用、等待数据库响应时，如果用同步方式只能干等着，异步则能让它在等待的间隙去干别的活。

但asyncio是 Python 里出了名的"看着会用、其实没懂"的部分。很多人能照着写async def和await，却说不清协程到底是什么、事件循环在背后干了什么、为什么有时候明明await了却没有并发。

- 读完你应该建立这样一个心智模型：async def定义的不是"立刻执行的函数"，而是一份"可以被暂停和恢复的任务说明书"；事件循环是那个不断在多份说明书之间切换的单线程调度员；await则是协程主动交出控制权的时刻。
- 1、为什么需要异步：async/await 的本质

> 要解决的问题：程序大量时间花在"等待"上——等网络、等磁盘、等数据库。同步代码在等待时整个线程都被阻塞，什么也干不了。能不能在等待的间隙去干别的事？

先看一个直观的对比。假设要请求三个网址，每个耗时 1 秒(其中绝大部分是等待网络)：

同步版本的问题在于：那三段sleep(1)其实都是"干等"，本可以重叠进行，却被串成了一条线。异步要做的，就是在某个任务进入等待时，把控制权交出去，让别的任务先跑。

#### async def 定义的是什么？
关键认知：async def定义的函数，调用它并不会执行函数体。它返回的是一个"协程对象"——你可以把它理解成一份"待执行的任务说明书"，必须交给事件循环(或用await)才会真正跑起来。

对比普通函数：

- 普通函数 def f()

- 协程函数 async def f()

| 对比项 | 普通函数 def f() | 协程函数 async def f() |
| :--- | :--- | :--- |
| 调用 f() 时 | 立即执行函数体 | 不执行，返回一个协程对象 |
| 拿到的东西 | 函数的返回值 | 协程对象(说明书) |
| 怎么真正执行 | 调用即执行 | 交给事件循环 / 被 await |
| 能否中途暂停 | 不能 | 能，在 await 处暂停并交出控制权 |

> 一句话：async def写出说明书，await标记"这里可以暂停、可以交出控制权"。

承上启下：我们知道了协程对象是一份待执行的说明书、await是暂停点。但说明书本身不会动——必须有人不停地在多份说明书之间切换、推进它们。这个"调度员"就是事件循环。

### 2. 谁来驱动协程：事件循环如何调度
> 要解决的问题：协程对象只是"说明书"，谁来真正一步步推进它？多个协程同时存在时，谁决定先跑哪个、什么时候切换？

答案是事件循环(event loop)。它是asyncio的心脏，本质是一个单线程的无限循环，干的事情可以概括为：

- 维护一个"待运行的任务"队列；

- 从队列里取一个协程，推进它，直到它遇到await暂停(交出控制权)；

- 把控制权拿回来，去推进队列里的下一个协程；
- 当某个被等待的操作(如网络、定时器)完成了，把对应协程重新放回队列；
- 如此往复，直到所有任务完成。
- 这里有个最容易误解的点：事件循环是单线程的。它的并发不是"同时执行多段代码"，而是**"在一段代码等待时，切去执行另一段"**——靠的是任务间快速切换，而非真正的并行。所以异步特别擅长I/O密集(大量时间在等待)的场景，对CPU 密集(一直在算)的场景几乎没有帮助(第 7 节会展开)。
- 打个比方：一个厨师(单线程)同时做三道菜。他不是长出三只手同时炒，而是——把第一道菜放进烤箱(开始等待)后，立刻去切第二道菜的配料，再去腌第三道；哪个计时器响了(等待结束)就回去处理哪个。整个过程只有一个人，但因为没把时间浪费在干等上，三道菜几乎同时完成。事件循环就是这个厨师，await就是"把菜放进烤箱、转身去忙别的"那一刻。

可以亲眼看到这种切换：

三个协程几乎同时"开始"，然后按等待时间长短先后"结束"——这正是事件循环在它们让出控制权的间隙里来回切换的结果。

承上启下：事件循环我们看明白了，但上面已经偷偷用了asyncio.run和asyncio.gather。它们到底各自负责什么？还有个常被混淆的create_task，三者怎么选？下一节讲清楚。

### 3. 怎么真正用起来：run / create_task / gather 的区别
> 要解决的问题：手头有几个协程，想让它们真正跑起来、甚至并发跑。asyncio.run、create_task、gather长得都像"运行协程"，到底分别在什么时候用？

一句话先记住分工：run是总开关，create_task是"扔进后台跑"，gather是"并发跑一批并等齐结果"。

#### asyncio.run：程序的总入口
它负责创建事件循环、运行你的顶层协程、结束后清理循环。整个程序通常只调用它一次，作为异步世界的入口。

> 不能在已经运行的事件循环里再调asyncio.run(比如在 Jupyter 或另一个协程里)，那会报错。

#### asyncio.create_task：把协程交给事件循环"立刻调度"
create_task把一个协程包装成Task并立即提交给事件循环排队执行——它不会等任务完成，而是马上返回，让你能继续干别的。这是实现并发的基础。

对比一下——如果直接await work("A", 2)再await work("B", 1)，就变成顺序执行、共 3 秒；用create_task让它们先一起跑起来，才有并发。

#### asyncio.gather：并发运行一批，并等它们全部完成
gather是最常用的并发工具：把多个协程一起并发运行，等全部完成后按顺序返回结果列表。

#### 三者怎么选
- 什么时候用

- asyncio.run(coro)

| 工具 | 作用 | 什么时候用 |
| :--- | :--- | :--- |
| asyncio.run(coro) | 创建并运行事件循环 | 程序唯一的异步入口，全程调一次 |
| asyncio.create_task(coro) | 把协程提交为后台 Task，立即返回 | 想让任务先跑起来、自己继续干别的；需要精细控制单个任务 |
| asyncio.gather(*coros) | 并发运行多个并等齐结果 | 一批任务要并发、且都要拿到结果(最常用) |

> 要解决的问题：普通for每取一个元素都是同步的、with的进入和退出也是同步的。但如果"取下一个元素"或"打开/关闭资源"本身就是个需要等待的异步操作呢？

#### async with：异步上下文管理器
普通with用于自动管理资源的获取与释放(如开关文件)。但有些资源的获取或释放本身是异步的——比如建立一个异步数据库连接、获取一个异步锁。这时用async with：

它和普通with的唯一区别：进入/退出钩子是aenter/aexit，且可以await。最常见于数据库连接池、HTTP 客户端会话(如aiohttp.ClientSession)。

普通for适合一次性就在内存里的序列。但如果数据是一条一条异步到达的——分页 API、数据库游标、消息流、大模型的流式输出——每取下一条都要等待，就该用async for：

> 直观区分：async with管"异步的资源开关"，async for管"异步的逐条取值"。

承上启下：上面async for能工作，是因为它背后的对象实现了某种"异步迭代协议"。async with靠aenter/aexit，那async for靠什么？这就是下一节的aiter/anext。

### 5. 底层协议：aiter/anext实现异步迭代器
> 要解决的问题：async for凭什么能逐条异步取值？如果我想自己造一个能被async for遍历的对象，要实现哪些方法？

回顾一下同步世界：一个对象能被普通for遍历，是因为它实现了迭代器协议——iter(返回迭代器)和next(返回下一个值，没有了就抛StopIteration)。

异步世界完全对应，只是把两个方法换成异步版：

- 同步迭代器协议

- 异步迭代器协议

| 协议 | 同步迭代器协议 | 异步迭代器协议 | 作用 |
| :--- | :--- | :--- | :--- |
| 返回迭代器 | iter | aiter | 返回迭代器对象本身 |
| 取下一个值 | next | anext(async def) | 异步返回下一个值；结束时抛 StopAsyncIteration |

承上启下：手写aiter/anext这一整个类，是不是有点啰嗦？就像同步世界里我们更常用生成器(yield)来简洁地造迭代器，异步世界也有对应的"语法糖"——异步生成器。

### 6. async generator：async def + yield

有。就像同步世界用def+yield写生成器来代替手写迭代器类，异步世界用async def+yield写异步生成器——它自动帮你实现了aiter/anext，代码瞬间清爽。

### 7. 选型：协程 vs 线程 vs 进程
> 要解决的问题：协程、线程、进程都能"同时处理多个任务"，但用错了不仅不会变快，反而更慢更复杂。面对一个具体任务，到底该选哪个？

要选对，先得理解一个关键背景：Python 有GIL(全局解释器锁)，它导致同一时刻只有一个线程在执行 Python 字节码。这条约束直接决定了三者的适用边界。

#### 三者本质对比
- 协程(asyncio)

- 线程(threading)

| 维度 | 协程(asyncio) | 线程(threading) | 进程(multiprocessing) |
| :--- | :--- | :--- | :--- |
| 并发方式 | 单线程内任务切换 | 多线程，受 GIL 限制 | 多进程，各自独立解释器 |
| 能否真正并行 | 否(单线程) | CPU 计算上不能(GIL) | 能(每个进程独立 GIL) |
| 切换开销 | 极小(用户态切换) | 较小(系统线程切换) | 大(进程创建、跨进程通信) |
| 内存 | 共享，无需序列化 | 共享，需注意线程安全 | 独立，数据需序列化传递 |
| 适合场景 | I/O 密集且任务量大 | I/O 密集(含同步阻塞库) | CPU 密集(计算重) |

- 任务大部分时间在算(图像处理、数值计算、加解密、大规模数据变换)→ 用进程。因为 GIL 的存在，多线程在纯计算上无法真正并行；多进程才能吃满多核 CPU。

- 想用协程，但不得不调用一个只有同步阻塞版本的库(某些老旧的数据库驱动)→ 用线程做兜底，把阻塞调用丢到线程里(asyncio.to_thread)，别让它卡住事件循环。
- 一个判断口诀
- 等得多就用协程，算得多就用进程，被同步库卡住了就用线程兜底。

举个具体场景帮你定位：一个 Agent 要"并发请求 100 个网页、然后对抓回来的内容做一次重型的本地语义计算"——前半段是 I/O 密集，用asyncio.gather并发抓取；后半段是 CPU 密集，把计算丢进进程池(ProcessPoolExecutor)。这种"异步 I/O + 进程池计算"的组合，是实战里非常常见的搭配。

### 收尾：回到那条主线
回到开头那个问题——一个协程从被定义、被调度、到能与其他协程并发协作，完整链路是怎样的？现在可以一口气说清楚了：

- async def定义协程函数，调用它得到的是一份"任务说明书"(协程对象)，并不立即执行；

- await在说明书里标出"可暂停点"——遇到它就把控制权交还给调度员；

- 事件循环这个单线程调度员，不断在多份说明书之间切换，趁某个在等待时去推进别的；
- asyncio.run启动整套机制，create_task把协程扔进后台调度，gather并发跑一批并等齐结果；
- 异步场景下的逐条取值用async for、资源开关用async with，它们背后是aiter/anext和aenter/aexit协议；
- 想简洁地造异步数据流，用async def + yield写异步生成器，边等边产出；
- 最后记住边界：异步只在I/O密集时发力，CPU 密集要交给多进程，被同步库卡住则用线程兜底。
- 把这条链路走通，你再看 FastAPI 里那些async def路由、Agent 框架里那些流式async for，就不再是"照着写"，而是清楚每一个async、每一个await背后，那个单线程调度员正在做什么。
- 一句话记忆

- async def / await

| 知识点 | 一句话记忆 |
| :--- | :--- |
| async def / await | 写说明书 + 标记可暂停点，调用不立即执行 |
| 事件循环 | 单线程调度员，在协程让出控制权时切去跑别的 |
| run / create_task / gather | 总开关 / 扔后台 / 并发跑一批等齐 |
| async with / async for | 异步资源开关 / 异步逐条取值 |
| aiter / anext | 异步迭代器协议，async for 的底层 |
| async generator | async def + yield，边等边产出，流式场景首选 |
| 协程 vs 线程 vs 进程 | 等得多用协程，算得多用进程，被同步库卡住用线程 |
