## 0x01. 百度面试官追问：你说 RAG 准确率提升了 20%，评测集在哪？指标怎么算的？
> 发布日期：2026-05-26  
> 原文链接：[百度面试官追问：你说 RAG 准确率提升了 20%，评测集在哪？指标怎么算的？](https://mp.weixin.qq.com/s/zb24AT7JEGyLpiIyu3iOKA)

大家好，我是墨圆。

前几天有个读者跟我复盘他在百度的面试，前面聊得都挺顺。

🙋‍♂️他介绍自己的 RAG 项目时说，系统做了混合检索、Rerank、Prompt 约束，最后“准确率提升了 20%”。

👨‍💻面试官听到这句话，马上停住了：“你这个 20% 是怎么算出来的？”

🙋‍♂️他说主要是业务方试用后感觉回答更准了，badcase 少了很多。

👨‍💻面试官继续追问：“那你们有固定评测集吗？每次改 chunk size、改 embedding、改 top_k，都是在同一批问题上测的吗？”

🙋‍♂️他说没有特别固定，就是拿几个典型问题试一下。

👨‍💻面试官又问：“那你怎么知道这次提升不是刚好对这几个问题有效？检索指标和生成指标分开看了吗？Recall@K、MRR、Faithfulness 分别是多少？”

🙋‍♂️他彻底卡住了。

👨‍💻面试官最后说了一句：“你说 RAG 准确率提升，这句话本身没有意义。没有评测集、没有指标拆分、没有 badcase 归因，你所谓的优化就是凭感觉调参。”

这个问题特别典型。很多人做 RAG 项目，喜欢讲“效果变好了”“召回更准了”“幻觉少了”。但真正一被追问：好多少？怎么量？是哪一层变好了？有没有固定测试集？有没有回归测试？立刻就说不清楚。RAG 不是调 Prompt 的玄学项目。它是一个工程系统。既然是系统，就必须能评测、能对比、能复现。今天这篇就来聊一个很多人忽略，但面试和真实上线都绕不开的问题：RAG 系统到底应该怎么评测？

简要回答RAG 评测不能只看“大模型最终回答得好不好”，而要拆开看。因为 RAG 不是一个单点模型，而是一条完整链路：用户问题进来之后，要经过 query 理解、检索、重排、上下文拼装和最终生成。最后答案错了，可能是资料根本没找回来，也可能是找回来了但排得太靠后，还可能是资料已经进了 Prompt，大模型却没有用对。所以更合理的评测方式，是把它分成检索评测、生成评测和端到端评测三层。检索层看正确资料有没有进入候选集，常用 Recall@K、MRR、Precision@K；生成层看模型有没有基于资料说对，常用 Faithfulness、Answer Relevance、Completeness；端到端层再看真实用户体验，比如首字延迟、引用可用性、用户满意度、人工转接率和调用成本。真正成熟的 RAG 评测，一定要有一套固定评测集。每条样本至少要包含用户问题、标准答案、正确文档片段和问题类型标签。以后每次改 Embedding、改切分、改 Rerank、改 Prompt，都在同一批样本上跑一遍。只有这样，你才能知道这次优化到底提升了哪一层，牺牲了什么指标，又引入了哪些新的 badcase。面试里只说“我们优化后准确率提升了”是不够的。你要能说清楚：评测集怎么构造，指标怎么拆分，提升发生在哪一层，badcase 是怎么归因的。否则这个“提升”，在面试官眼里就是一句没有证据的主观感受。

详细解析为什么不能只看最终回答很多人评测 RAG 的方式很简单：拿几个问题问系统，看答案顺不顺眼。这个动作当然有价值，但它只能算体验检查，不能算真正的评测。原因很简单，RAG 的错误不是一个黑盒错误。比如用户问：“生鲜商品签收后超过 24 小时还能申请坏果赔付吗？”系统最后回答：“可以申请赔付。”这个答案可能是错的，但到底为什么错？可能是检索阶段没有搜到“生鲜售后时效”那段规则；也可能是搜到了，但排在第 8 位，而 Prompt 只塞了前 5 条；也可能是资料里写得很清楚“24 小时内申请”，但模型生成时把“内”这个限制看丢了；还可能是新旧两版规则同时被召回，模型混用了旧规则。如果你只看最终答案，就只能说“答错了”。但你不知道该改哪里。这就像修一条流水线。最后出来的零件不合格，问题可能在原料、模具、传送带，也可能在最后一道质检。你不能只盯着成品说“不行”，你得知道它是哪一段开始跑偏的。RAG 评测也是一样，必须把链路拆开看。

检索评测：先确认资料有没有找对RAG 的第一道生死线是检索。如果正确资料没被召回，后面大模型再强也只能靠猜。所以第一层评测一定是检索评测，它回答的是一个非常朴素的问题：系统有没有把该找的资料找回来，并且排在足够靠前的位置。这里最常见的指标是 Recall@K。比如 Recall@5 = 0.82，意思是在所有测试问题里，有 82% 的问题能在前 5 条检索结果中找到正确资料。这个指标决定了 RAG 的上限，因为正确资料不在候选集里，后面就不可能稳定答对。MRR 则更进一步，它不只看有没有找到，还看正确资料排得靠不靠前。正确文档排第 1，得分是 1；排第 2，得分是 0.5；排第 5，得分是 0.2。这个指标特别适合判断 Rerank 有没有价值，因为同样进入 Top 10，排第 1 和排第 10 对最终回答的影响完全不一样。还有一个容易被忽略的指标是 Precision@K。它看的是前 K 条里有多少是真的相关。很多系统 Recall 不低，但 Precision 很差，召回列表里混进了大量“看起来相关、其实没用”的片段。大模型拿到这种上下文，就像一个学生考试时桌上摊了十份资料，其中九份都是干扰项，答错并不奇怪。如果场景里不止一个正确文档，而且相关性有强弱，还可以看 NDCG。比如一条文档直接回答问题，另一条只是背景说明，两者都相关，但权重不同。NDCG 就适合衡量这种排序质量。面试里不需要把公式背得多复杂，但要把作用说清楚：Recall@K 看“有没有找回来”，MRR 看“排得靠不靠前”，Precision@K 看“噪声多不多”，NDCG 看“排序顺序是否合理”。这几个指标一拆开，优化方向就会清楚很多。如果 Recall@10 很低，先别急着调 Prompt，要回到 Embedding、BM25、query rewrite 和 chunking；如果 Recall@10 很高但 MRR 低，说明资料找到了但排得靠后，要看 Rerank；如果检索指标都不错，最终答案仍然错，那问题大概率已经不在检索层了。

生成评测：资料找对了，不代表模型说对了很多人以为检索指标高，RAG 就稳了。这个理解只对了一半。资料找对，只说明你把答案放到了模型面前。但模型有没有真正用它，有没有乱补，有没有漏掉限制条件，还要单独评测。RAG 里最危险的情况，往往不是模型完全答错，而是它大部分内容都对，只在关键条件上多说了一句、漏说了一句，用户还很难发现。生成阶段最核心的是忠实性，也就是 Faithfulness。它看回答里的关键断言，是否都能被检索资料支持。比如资料里写“换新申请需在签收后 7 天内提交”，模型回答“15 天内都可以申请”，这就是不忠实。Answer Relevance 看的是回答有没有真正回应用户问题。用户问“还能不能申请”，模型讲了一大段“申请入口在哪里”，虽然内容可能都来自资料，但没有正面回答问题，相关性就不够。Completeness 看的是有没有漏掉关键条件。比如标准答案应该是“可以申请，但必须在 24 小时内上传照片，并保留商品包装”，模型只回答“可以申请”，这就不完整。这三个指标要组合起来看。一个回答可能忠实但不完整，它没有编，但漏了条件；也可能完整但不忠实，看起来面面俱到，里面有几句却是模型自己补的；还可能资料没错、回答也没编，但没有回答用户真正问的点。实际项目里，一开始不必把评测系统做得特别复杂。可以先让人工标注员按 0/1 或 1-5 分给这些维度打分。样本量也不需要一上来就几千条，先认真做 100 条，也比完全凭感觉调参强得多。更工程化一点，可以用 LLM-as-judge 做初筛，让另一个模型根据“问题、检索资料、模型回答、标准答案”给出评分和理由；但在金融、医疗、法律这类高风险场景里，关键样本最好还是人工复核。

评测集怎么建：不要拿几个顺手问题糊弄自己指标有了，下一步就是评测集。没有固定评测集，所有优化都会变成“这次看起来好了”。今天你拿 10 个问题试，明天换 10 个问题试，最后发现系统好像提升了，但这个提升到底来自模型，还是来自问题变简单了，根本说不清楚。一套最基础的 RAG 评测集，每条样本至少要有用户问题、标准答案、正确文档片段和问题标签。比如用户问题是“生鲜商品签收后超过 24 小时还能申请坏果赔付吗”，标准答案可以写成“通常不能。坏果赔付需在签收后 24 小时内提交，并上传清晰照片”。正确文档片段要指向具体的doc_id和chunk_id，比如doc_售后规则_v3_chunk_18。问题标签可以标成“售后、时效限制、否定问题、高风险”。这里最关键的是正确文档片段。没有这个字段，你就无法做检索评测，只能看最终回答。很多团队所谓的 RAG 评测，其实只有标准答案，没有 gold docs。这样最后只能判断“答没答对”，却不知道“资料有没有找对”。评测集最好来自真实用户问题，而不是自己坐在电脑前拍脑袋编。真实用户的表达非常乱：有人问“坏果怎么赔”，有人问“葡萄烂了还能退不”，有人问“昨天晚上签收今天晚上发现坏了算超时吗”，还有人会把订单信息、情绪抱怨和业务问题混在一起。这些真实表达，才是检验系统的好材料。当然，早期没有线上日志时，也可以让业务同事基于核心文档写问题。每个重要规则写 3 到 5 个问法，覆盖口语表达、专业表达、否定问题和边界条件问题。这样很快就能攒出第一版 100 到 300 条评测集。它不一定完美，但足够让系统从“凭感觉”进入“有依据”。

评测集要分层，不然总分会骗人只准备一堆问题还不够，最好给问题打标签。因为不同类型的问题，对 RAG 的要求完全不一样。“会员积分多久过期？”这种事实直查题，答案通常在一段文档里直接出现，检索和生成都比较简单。“超过 24 小时还能赔吗？”这种条件判断题，就要求模型识别时间限制、否定关系和适用条件，难度明显更高。“企业客户退保证金需要哪些材料，线上提交后多久到账？”这种多跳组合题，可能需要同时召回材料规则和到账规则。还有一种很容易被忽略的拒答题，文档里没有答案，系统应该说“根据现有资料无法回答”，而不是硬编一个。如果你只看总体准确率，很容易被平均数骗了。假设评测集里 80% 都是事实直查题，系统整体准确率可能很好看。但一上线，用户问的全是条件判断、多跳组合和边界问题，系统就开始翻车。更合理的评测报告应该按问题类型拆开看。你可能会发现，事实直查准确率 92%，条件判断只有 63%，多跳组合只有 48%，拒答类只有 35%。这个时候，优化方向就很明确了：不是系统整体“不行”，而是某几类能力明显短板。这也是面试官真正想听的东西。不是一个漂亮的总分，而是你能不能看懂这个分数背后的结构。

badcase 归因：错了不可怕，不知道为什么错才可怕评测真正有价值的地方，不是得出一个分数，而是帮你找到下一步该改什么。所以每个 badcase 都要归因。一个常见的做法是把错误分成几类：如果正确文档没有进 Top K，就是召回失败，要回到切分、Embedding、BM25、query rewrite 上排查；如果正确文档召回了，但排得太靠后，没有进入最终 Prompt，就是排序失败，要看 Rerank 和 top_k；如果正确文档已经进了 Prompt，但模型没有用对，可能是上下文太长、关键片段放在中间、Prompt 约束不够，或者模型被噪声干扰；如果模型基于资料乱补、漏条件、答非所问，那就是生成失败，要看引用标注、忠实性检测、输出约束和拒答策略。这套归因比单纯说“答错了”有用得多。比如你发现 60% 的 badcase 都是召回失败，那就别再反复改 Prompt 了，先救检索。如果 70% 的 badcase 是“资料已经进入 Prompt 但模型没用对”，那继续换 Embedding 意义也不大，应该去看上下文排序、引用约束和生成检测。很多团队优化 RAG 最大的问题，就是不做归因。看见错题就想当然地调一个参数，今天改 top_k，明天改 chunk size，后天换 Prompt，最后系统变得越来越复杂，但质量并没有稳定提升。badcase 归因的价值就在于，它能把优化从“瞎试”变成“对症下药”。

每次优化都要做 A/B，不要边调边忘RAG 优化很容易陷入一个陷阱：今天改 chunk size，明天换 Rerank，后天改 Prompt，最后发现效果好像变了，但不知道是哪一步导致的。所以评测一定要有版本意识。每次改动尽量只改一个主要变量，然后在同一套评测集上跑 baseline 和新版本。比如 v1 是 BGE-M3 加 Top20，没有 Rerank；v2 是 BGE-M3 加 Top50，再接 BGE-Reranker；v3 在 v2 基础上增加引用约束 Prompt。每个版本都记录检索指标、生成指标、延迟和成本。这样你才能看出真实变化。Rerank 可能让 MRR 从 0.61 提到 0.78，但延迟增加 600ms；引用约束可能让 Faithfulness 从 0.82 提到 0.91，但 Completeness 略微下降；chunk size 变大可能让 Recall@5 上升，但 Precision@5 下降，因为噪声变多了。这些结果不能简单用“好”或“不好”概括，它们本质上都是工程权衡。真正的优化不是让某一个指标无限变大，而是在准确性、忠实性、延迟和成本之间找到业务能接受的平衡点。

线上评测不能替代离线评测有人会说，既然最终用户满意最重要，那我直接看线上点赞率不就行了？不行。线上反馈当然重要，但它不能替代离线评测。因为线上数据太噪了。用户不点赞，不一定是答案错了，可能是界面不好用；用户点赞，也不一定代表答案完全正确，可能只是看起来顺眼。更麻烦的是，线上问题分布每天都在变，你很难判断指标变化到底来自系统优化，还是来自用户问题变简单了。离线评测的价值在于稳定、可复现。每次改动都在同一批问题上测，才能判断系统本身有没有进步。更合理的做法是两者结合：离线评测负责版本上线前的质量门禁，线上评测负责发现新问题、收集新 query、更新评测集。比如系统上线后，可以把用户点踩、人工客服接管、用户追问“你没回答我的问题”的样本捞出来，人工标注后补进评测集。这样评测集会随着业务不断进化，而不是永远停留在第一版。

面试里怎么回答这道题如果面试官问：“你们 RAG 系统怎么评测？”不要只说“我们看准确率”。 可以这样答：“我们会把 RAG 评测拆成检索、生成和端到端三层。检索层看 Recall@K、MRR、Precision@K，判断正确文档有没有召回、排得靠不靠前、噪声多不多；生成层看 Faithfulness、Answer Relevance、Completeness，判断模型有没有基于资料回答、有没有编造、有没有漏掉关键条件；端到端层再看用户满意度、首字延迟、成本和人工转接率。评测集会固定下来，每条样本包含 query、标准答案、gold docs 和问题标签。每次改 Embedding、chunk size、Rerank 或 Prompt，都在同一批样本上做 A/B，对 badcase 做召回失败、排序失败、上下文使用失败、生成失败四类归因。”这段回答的好处是，它没有停在概念层面，而是把“怎么测、测什么、怎么用结果指导优化”讲完整了。面试官真正想听的不是某个指标名，而是你有没有把 RAG 当成一个可度量、可迭代、可复现的工程系统。

写在最后很多人做 RAG，前期最兴奋的是接模型、接向量库、接 Rerank。但真正决定系统能不能长期迭代的，往往是评测体系。没有评测，你每次优化都像在黑屋子里搬家具。你觉得位置变好了，但灯一开，可能只是把沙发堵到了门口。RAG 项目也是一样。你可以一开始没有很完美的评测集，但不能一直没有。哪怕先做 50 条真实问题，手工标好正确文档和标准答案，也足够让优化从“凭感觉”变成“有依据”。所以以后再说“RAG 效果提升了”，最好先问自己一句：我的评测集在哪？指标拆开了吗？badcase 归因了吗？答得上来，才是真的做过 RAG。

## 0x02. 蚂蚁面试官：你的 RAG 系统首字响应要 6 秒，用户早跑了，延迟花在哪你知道吗？
> 发布日期：2026-05-26  
> 原文链接：[蚂蚁面试官：你的 RAG 系统首字响应要 6 秒，用户早跑了，延迟花在哪你知道吗？](https://mp.weixin.qq.com/s/6LhDMu6-sakHRYNdk0OpzA)

大家好，我是墨圆。

上周有个读者找我复盘他在蚂蚁的面试。前面聊检索策略聊得挺顺利，面试官一直在点头，然后话锋一转。

👨‍💻面试官问了一句："你的系统上线之后，用户从提问到看到第一个字要等多久？"

🙋‍♂️他说大概 5-6 秒。

👨‍💻面试官的表情立刻变了："6 秒？用户问个社保报销流程等 6 秒，百度搜索都比你快。你知道这 6 秒花在哪了吗？"

🙋‍♂️他说应该是大模型生成比较慢。

👨‍💻面试官追问："是吗？你量过吗？Embedding 调用花了多少毫秒？向量检索花了多少毫秒？Prompt 拼装花了多少毫秒？LLM 首 token 延迟是多少？你都没拆过，凭什么说是大模型慢？"

🙋‍♂️他答不上来了。

👨‍💻面试官最后说了一句："性能优化的第一步不是加缓存、上异步，而是搞清楚时间花在哪。你连瓶颈都没定位就开始优化，这叫盲人摸象。"

这个场景我太熟悉了。做 RAG 项目的人，十个里有九个把全部精力花在"怎么检索更准"上——调 Embedding、换分块策略、加 Rerank。但一到上线，业务方第一句反馈不是"答得不准"，而是"太慢了"。

你的系统效果再好，用户等 6 秒才看到第一个字，体验直接崩盘。很多企业内部知识问答系统上线后日活从几百掉到个位数，不是因为答案不好，就是因为太慢——用户宁可去问同事也不愿意等你的系统。

今天把 RAG 系统的性能优化从定位瓶颈到逐层压缩完整讲一遍。

### 简要回答
RAG 系统的响应延迟不是一个笼统的数字，它由多个环节串联组成。用一个公式来表达就是：

总延迟 = Query Embedding 时间 + 向量检索时间 + Prompt 构建时间 + LLM 首 token 延迟

优化的第一步永远是定位瓶颈——在全链路上做打点计时，看哪个环节耗时最长。然后针对性优化：Embedding 阶段做缓存和批量调用；向量检索阶段选对索引类型、调好参数、做分区检索；架构层面把串行流程改成异步流水线；最后用流式输出（streaming）让用户在 LLM 生成第一个 token 时就看到内容，而不是等整个回答生成完。

在这些优化之上，三级缓存架构（Embedding 缓存、检索结果缓存、答案缓存）是效果最立竿见影的手段——热门问题可以做到近乎零延迟响应。

面试里只说"加了缓存"远远不够。你要说清楚缓存加在哪一层、命中率多少、未命中的请求怎么处理。能拿出全链路延迟的拆分数据，才能让面试官相信你真的做过性能优化。

### 详细解析
#### 先搞清楚：6 秒到底花在哪了
这是性能优化的前提，也是很多人跳过的一步。

我第一次认真做 RAG 性能分析的时候，以为瓶颈一定在 LLM 生成——毕竟大模型推理是最"重"的计算。结果打点一看，完全不是那么回事。

一个典型的 RAG 请求，各环节耗时大概是这样的分布（未做任何优化的情况下）：

Query Embedding（调 API 算向量）：200-400ms。向量检索（Milvus 查询）：50-200ms。Rerank 精排（Cross-Encoder 打分）：300-800ms。Prompt 拼装（拼上下文、截断）：10-30ms。LLM 首 token 延迟：500-2000ms。

加起来大概 1-3 秒。但为什么用户体验到的是 5-8 秒？因为还有一些隐藏开销：网络传输延迟、服务间调用的排队等待、JSON 序列化反序列化、甚至连接建立的握手时间。这些零散的开销加起来，可能又吃掉了 1-2 秒。

所以优化的第一步不是急着动手，而是在每个环节的入口和出口打上时间戳，把完整的耗时分布画出来。你可能会发现，瓶颈根本不在你以为的地方——也许你的 Embedding API 因为网络抖动经常要 600ms，也许你的 Rerank 模型在 CPU 上跑、每次精排 50 条候选集要一秒多。

#### Embedding 阶段：别小看这几百毫秒
如果你用的是远程 Embedding API（比如 OpenAI 的 text-embedding-3-small），每次调用至少 200-300ms 的网络往返。高峰期可能更慢，偶尔还会触发限流返回 429。

这个阶段有三板斧。

第一板斧：缓存。效果最立竿见影。用 Redis 做一层缓存，key 是 query 字符串的哈希值，value 是对应的向量。用户问过"年假怎么算"，下次再问同样的问题直接从缓存取向量，完全跳过 API 调用，延迟从 300ms 降到 1ms 级别。实现上有个细节——先对 query 做规范化处理（统一大小写、去多余空格）再取哈希，避免"年假 怎么算"和"年假怎么算"生成不同的缓存 key。

第二板斧：本地模型替代远程 API。如果你的部署环境有 GPU，可以用 BGE-M3 或者 E5-small 在本地跑 Embedding，彻底省掉网络延迟。BGE-M3 在一张 T4 上推理一条 query 大概 10-20ms，比远程 API 快一个数量级。

第三板斧：异步并发。如果你的系统同时需要给 query 和它的几个重写变体算 Embedding，不要逐条串行调用。用 asyncio 做异步并发，几条 query 的 Embedding 一次性并行发出去，总耗时约等于单条的耗时。

#### 向量检索阶段：索引选对，快十倍
Embedding 算完了，接下来是去向量数据库做检索。这个阶段的优化核心就一个字：索引。

HNSW 索引是当前的最优选择。在百万级向量规模上，HNSW 能把查询延迟压到个位数毫秒，同时保持很高的召回率。Milvus、Qdrant 等主流向量数据库都推荐 HNSW 作为默认索引。

关键参数有两个。efConstruction 是建索引时的参数，设 128 左右就够了，这是一次性开销。efSearch 是查询时的参数，越大召回越全但越慢，64 是个不错的起点。

除了索引本身，还有两个实用优化。

分区检索。如果你的知识库有明确的分类（按部门、按文档类型），把文档按属性切分成 Partition。用户问报销流程，直接在"财务制度"分区里检索，避免全库扫描。搜索范围缩小了，速度和准确率都能提升。

确保数据在内存。这是一个容易忽略的点——查询前必须调用 collection.load()，把数据加载到内存。如果数据还在磁盘上，每次查询都走 IO，延迟高一个数量级。

#### Rerank 阶段：这才是隐藏的延迟大户
很多人在做性能分析的时候会忽略 Rerank 这一层。但 Cross-Encoder 的计算量比 Bi-Encoder 大得多——它要把每一对 (query, doc) 拼在一起做一次完整的 Transformer 推理。如果候选集有 100 条，就是 100 次推理。

在 CPU 上跑 BGE-Reranker-base 对 100 条候选集做精排，耗时可能超过 1 秒。放到 GPU 上能快很多，但不是所有部署环境都有 GPU。

优化策略有两个。

减少候选集数量。初步召回不要取太多——Top 20 或 30 就够了，不需要 Top 100。候选集减半，Rerank 耗时就减半。如果你对召回率有顾虑，可以通过混合检索（向量 + BM25）来提高 Top 20 的覆盖率，而不是靠扩大候选集来兜底。

前几页精排，后面跳过。如果你的系统有分页展示，只对前 1-3 页的结果做精排。用户 90% 的情况只看第一页，对第 4 页之后的结果做精排纯粹浪费算力。

#### 三级缓存架构：效果最炸裂的优化
前面每个阶段的优化是"压缩单次延迟"，三级缓存则是"跳过整个链路"。

第一级：Embedding 缓存。前面讲过了。key 是 query 哈希，value 是向量。TTL 可以设得很长，因为同一段文本的 Embedding 结果是固定的。

第二级：检索结果缓存。key 是 query 哈希 + top_k 参数，value 是检索到的文档 ID 列表。命中这层缓存后，直接跳过 Embedding 计算和向量检索两个环节。如果知识库内容相对稳定（比如公司制度文档不会天天改），TTL 设 24 小时没问题。知识库更新时主动清缓存。

第三级：答案缓存。key 是 query 哈希，value 是最终的回答文本。这层只适合 FAQ 型的高频问题——"年假几天""报销流程是什么"这种答案几乎不变的问题。命中后直接返回缓存答案，延迟趋近于零。

三级缓存的查询逻辑是逐级降级：先查答案缓存（命中直接返回）→ 查检索缓存（命中跳过 Embedding 和检索）→ 查 Embedding 缓存（命中跳过 Embedding 计算）→ 全未命中走完整链路。

在我们的项目中，热门问题占总查询量约 30%，这些基本都能命中第二级或第三级缓存，首字响应时间直接降到 50ms 以内。即便是冷门查询，Embedding 缓存也能省下 200-300ms。

#### 流式输出：让用户"感觉"更快
上面所有优化都是在缩短真实延迟。但还有一种优化是缩短"感知延迟"——用户感觉到的等待时间。

流式输出（streaming）的原理是：LLM 生成的不是完整的回答一次性返回，而是一个 token 一个 token 地实时推送。用户在 LLM 生成第一个 token 的时候就能看到内容开始出现，而不是等整段话全部生成完才看到。

实际体验差异有多大？假设 LLM 生成一段 200 token 的回答需要 3 秒。不开流式，用户等 3 秒才看到任何内容。开了流式，用户在 0.5 秒左右就看到第一个字开始出现，然后文字逐渐涌出，3 秒后完整呈现。虽然"全部完成"的时间是一样的，但用户感知到的等待时间从 3 秒缩短到了 0.5 秒——这个差异对用户留存率的影响非常大。

### 面试里怎么谈性能优化
这道题面试官爱问，是因为它能直接区分"做过 demo"和"上过线"的人。demo 阶段根本不会关心性能，只有真正上线面对用户的人才会被"太慢了"这三个字追着跑。

第一个容易翻车的点：不知道延迟花在哪。面试官问"你的系统延迟多少"，你说"大概 5 秒"，面试官追问"花在哪"，你说"主要是大模型生成慢"——如果你没有实际做过打点分析，这个判断大概率是错的。能拿出各环节的具体数字（Embedding 300ms、检索 80ms、Rerank 500ms、LLM TTFT 800ms），面试官马上就知道你是真测过的。

第二个容易翻车的点：只说"加了缓存"不说细节。面试官一定会追问"缓存加在哪一层""命中率多少""缓存失效怎么处理"。能讲出三级缓存的分层设计、各层的 TTL 策略、知识库更新时的缓存清除机制，才是一个完整的回答。

第三个容易翻车的点：不知道流式输出。面试官问"你的系统用了 streaming 吗"，你说没有——这说明你对用户体验的感知还停留在功能层面，没有到工程层面。流式输出几乎是零成本的优化，但对用户感知延迟的改善极其显著。

一个比较完整的面试回答框架：先讲性能公式，展示系统性思维（总延迟 = Embedding + 检索 + Rerank + LLM TTFT）→ 讲全链路打点定位瓶颈 → 按环节讲优化策略（Embedding 缓存/本地模型、HNSW 索引调优/分区检索、Rerank 候选集控制）→ 讲三级缓存架构 → 讲流式输出 → 给数据（热门查询 50ms、冷门查询 1.5s、优化前 5-6s）。

### 写在最后
性能优化这件事，做 RAG 项目的人很容易忽略——大家的注意力全在"准不准"上，觉得"快不快"是锦上添花。但到了生产环境你会发现，用户对速度的容忍度远低于对准确度的容忍度。他可以接受偶尔答得不够完美，但绝对不接受每次提问都干等 5 秒。

而且性能优化不是一个独立的事情，它跟你的检索策略、分块策略都有连锁反应。Chunk 切太大，Embedding 计算量增加；召回候选集太多，Rerank 计算时间膨胀；Prompt 里塞了太多上下文，LLM 首 token 延迟变长。这些都是跨环节的连锁效应，局部优化解决不了。

所以性能优化的第一步永远是：先看全局，再动手。

三个问题，全部哑火。

这件事让我意识到一个很多 RAG 开发者都有的认知偏差：大家把 90% 的精力花在了"怎么搜得更准"上，却忽略了"搜到之后怎么说得对"这个同样关键的环节。检索只是把正确信息送到了模型面前，模型愿不愿意用、会不会用、有没有乱加东西——这是另一个战场。

今天把 RAG 生成阶段的幻觉压制、引用标注、上下文管理这些问题讲透。

### 简要回答
RAG 的幻觉问题跟普通大模型的幻觉不太一样。普通幻觉是"模型不知道还在编"，RAG 的幻觉是"正确答案就摆在它面前了，它还是往里面掺假"。后者更隐蔽，也更危险——因为回答的大部分内容是对的，只有一两句是编的，用户很难发现。

压制这种幻觉要三层机制配合:

第一层是Prompt 层面的硬约束—— 不是简单写一句"请根据资料回答"，而是要有具体的格式要求、Few-shot 示例引导、以及对"不知道就说不知道"的明确授权。

第二层是引用标注机制—— 要求模型在回答时标注信息来源编号，这不仅是给用户看的，更是对模型的一种隐性约束：当它想编造信息时，找不到对应的来源编号，就会被迫收敛。

第三层是后处理检测—— 在模型输出之后，用规则或者小模型检查回答中是否出现了检索文档中不存在的关键断言。

这三层缺一层，幻觉就很难压到可接受的水平。面试里只答"Prompt 里加了约束"，在面试官看来等于什么都没做。

### 详细解析
#### 为什么检索对了，模型还是会编
要理解这个现象，先要回到大模型生成文本的底层机制。

大模型生成回答的时候，本质上是在做概率预测——给定前面的所有 token，预测下一个最可能的 token。这个预测的依据有两个来源：一个是你通过 Prompt 传进去的上下文（包括检索到的文档），另一个是模型在预训练阶段已经"记住"的知识。

问题就出在第二个来源。当上下文里的信息跟模型的"记忆"有冲突时，模型不一定会以上下文为准。你告诉它"根据以下资料回答"，但模型在生成过程中，某些 token 的概率分布可能更倾向于它预训练时学到的版本，而不是你塞给它的文档版本。

举个具体的场景。知识库里有一份 2024 年更新的员工手册，写着"年假天数为 15 天"。但模型在预训练数据里见过大量的"年假天数为 10 天"或"年假天数为 5 天"。你把 2024 年的手册片段检索出来塞进 Prompt，模型在生成"年假天数为"之后的那个 token 时，"15"这个数字的概率可能并不比"10"高多少——因为预训练的"先验知识"在拉扯它。

另一种更常见的情况是模型自行"补充"信息。文档里写了"该产品支持意外身故和全残保障"，模型输出"该产品支持意外身故、全残保障和住院津贴"——"住院津贴"这四个字是模型自己加的，因为它"觉得"保险产品一般都会有住院津贴。这种基于预训练统计规律的"合理补充"，就是 RAG 幻觉最典型的形态。

在金融、医疗、法律这些领域，这种"看起来合理但没有依据"的补充，可能导致严重的合规问题。

#### 第一层防线：Prompt 硬约束——别只写一句"请根据资料回答"
这是最基础的一层，但大部分人做得太粗糙。

"请根据提供的资料回答问题"——这句话你写了，模型也"看到"了，但它的约束力远没有你想象的那么强。原因很简单：这是一条抽象指令，模型对它的理解是模糊的。什么叫"根据资料"？是只能用资料里的原文，还是可以在资料的基础上做推理？如果资料里没提到的信息，是直接说"不知道"，还是可以用自己的知识补充？这些边界都没有定义清楚。

一个在实际项目中效果明显更好的 Prompt 模板长这样：

跟简单的"请根据资料回答"相比，这个模板做了三件关键的事。

第一，明确禁止添加额外信息。"不要添加资料中没有的内容"这句话看起来跟"根据资料回答"差不多，但它的约束力更强——因为它明确指出了模型不应该做的事情，而不只是指出应该做的事情。在 Prompt Engineering 里，禁止性指令通常比引导性指令更有效。

第二，授权模型说"不知道"。很多幻觉的根源是模型被逼着给答案。如果你不告诉它"可以说不知道"，它在找不到答案的时候就会编一个。明确授权"回答不了就说回答不了"，能显著降低模型的"创作欲望"。

第三，要求标注来源编号。这不仅是给用户看的，更是给模型自己的一个约束信号——后面会详细展开。

除了模板本身，Few-shot 示例的效果也非常好。在 Prompt 里放 1-2 个标准的问答示例——包括一个"能回答"的例子和一个"资料不足、坦诚说明"的例子——模型会模仿示例的回答模式，比单纯靠指令约束更可靠。

#### 第二层防线：引用标注——给模型戴上紧箍咒
引用标注这个功能，很多人觉得它是"锦上添花"的用户体验优化——让用户知道回答的出处，增强信任感。但在我看来，它最大的价值不在用户侧，而在模型侧。

当你要求模型"在引用信息时标注来源编号 [1] [2]"，你实际上给模型设了一个隐性门槛：每一句涉及事实判断的话，都需要有一个来源编号来"背书"。当模型想编造一个信息的时候，它面临一个困境——编的这句话没有对应的来源编号可以标注。虽然模型不一定每次都会因此停下来，但这个机制确实在统计意义上降低了它"自由发挥"的概率。

我自己在项目里管这个叫"引用即约束"——它看上去是做给用户的功能，实际上是套在模型头上的紧箍咒。

工程实现分三步。

第一步：检索结果带编号入 Prompt。每个检索到的文档片段用 [1]、[2]、[3] 编号标记，同时把文档的元信息（标题、页码等）一起记录下来，但元信息不放进 Prompt——只给模型看编号和内容，元信息留着后处理时用。

第二步：模型输出中解析引用编号。模型的回答里会出现 [1]、[2] 这样的标记。用正则表达式提取这些编号。

第三步：编号映射回原始来源。把提取出来的编号跟检索结果的元信息做匹配，最终展示给用户的是类似"[来源：员工手册 2024版 第15页]"这样的可点击链接。

#### 第三层防线：后处理检测——模型说完了，你再查一遍
Prompt 约束和引用标注能降低幻觉概率，但不能杜绝。所以还需要一道"出厂质检"。

最简单的后处理检测方式是关键实体核验。从模型回答中提取所有关键实体（数字、日期、产品名、条款编号等），然后去检索到的原始文档中做精确匹配。如果回答里出现了一个原始文档里完全没有的数字或术语，就标记为"疑似幻觉"。

比如模型说"该产品的等待期为 90 天"，你去原始文档里搜 "90" 这个数字，搜不到——文档里写的是"等待期为 30 天"。这就是一个可以被自动检测到的幻觉。

更精细的做法是训练一个专门的幻觉检测分类器。输入是（模型回答, 检索文档），输出是"忠实 / 不忠实"的二分类。用 NLI（自然语言推理）模型的思路来做——检索文档是前提（premise），模型回答是假设（hypothesis），判断假设是否能从前提中推导出来。如果不能，就标记为幻觉。

检测到幻觉之后怎么处理？有三种策略。保守策略是直接拦截，返回一个更安全的回答（比如"根据现有资料，该产品的等待期为 30 天"，只输出文档中明确有的信息）。中间策略是在疑似幻觉的句子旁边加一个提示标记，告诉用户"此部分信息可能不准确，请以原始文档为准"。激进策略是重新生成——把检测结果作为反馈注入 Prompt，让模型重新回答，这次会更加保守。

在金融合规场景下，我建议用保守策略。宁可少说，不能说错。

#### 还有一个容易被忽视的坑：Lost in the Middle
2023 年有一篇很经典的论文，标题就叫"Lost in the Middle"，揭示了大模型在处理长上下文时的一个显著弱点：模型对上下文开头和结尾的信息关注度高，对中间部分的信息关注度低。

这对 RAG 的影响非常直接。如果你检索到了 10 段文档，按默认顺序塞进 Prompt，最相关的那段恰好排在中间位置（比如第 5 段），模型很可能"没看到"它，反而基于排在开头或结尾的次相关文档来回答。你明明检索对了，但因为排列位置的问题，模型等于没看到正确答案。

解决方案有两个。

第一个是控制数量。不要无脑塞 10 段 15 段文档，通过 Rerank 精排后只取 Top 3-5 段最相关的。文档越少，Lost in the Middle 的影响越小——3 段文档基本不存在这个问题。

第二个是优化排列顺序。把相关度最高的片段放在最前面（Rerank 分数最高的排第一），次相关的放在最后面，最不相关的放在中间。这样利用了模型对开头和结尾的高注意力，确保最重要的信息一定会被模型"看到"。

实际项目中，这两个策略通常配合使用。我们在内测中做过一个对比实验：同样的检索结果，只是调整了文档片段在 Prompt 里的排列顺序——把最相关的从中间移到开头——回答准确率直接提升了近 10 个百分点。代码上就是排个序的事情，几乎没有成本，但效果非常显著。

#### 面试里怎么谈
这道题是 RAG 面试的高频题，而且它考的不是检索能力，而是你对生成阶段的工程把控。

第一个容易翻车的点：只说"Prompt 里加了约束"。面试官一定会追问"加了什么约束"，你如果只说"请根据资料回答"这种笼统的话，说明你没有深入思考过 Prompt 设计。要能讲出具体的策略——禁止性指令比引导性指令更有效、授权模型说"不知道"能降低编造概率、Few-shot 示例能引导回答模式。

第二个容易翻车的点：不知道 Lost in the Middle。这是 RAG 生成阶段最经典的问题之一，2023 年论文发出来之后已经成了面试常考点。如果你做过 RAG 项目却不知道这个问题，面试官会质疑你有没有认真做过效果分析。

第三个容易翻车的点：没有量化过幻觉率。面试官问"你的系统幻觉率是多少"，你说"不太清楚"——这就暴露了你没有做过系统性的评测。幻觉率可以用 Faithfulness 指标来量化：准备一批标注过的问答对，让系统逐条回答，然后判断回答中是否有检索文档不支持的断言。哪怕你只做了 100 条的小规模测试，能说出"我们的幻觉率从 18% 降到了 7%"，就比说"加了 Prompt 约束效果好了很多"有说服力得多。

一个比较完整的回答框架：先定义问题（RAG 幻觉 ≠ 普通幻觉，是"有正确资料还在编"）→ 讲三层防线（Prompt 硬约束 + 引用标注 + 后处理检测）→ 讲 Lost in the Middle 问题和排序策略 → 给一个量化数据（Faithfulness 从 xx 提升到 xx）。如果面试官继续追问，你还可以展开聊"模型自检机制"——生成完答案后让模型再检查一遍自己的回答是否都有文档支撑，发现没有支撑的就自动修正。这个机制代价是多一轮 LLM 调用，通常只在高风险场景（金融合规、医疗问答）中开启。

### 写在最后
RAG 系统有两个战场：一个是检索，一个是生成。大部分人在检索战场上打得很卖力——混合检索、Rerank、Embedding 微调、分块策略优化，每一个环节都有人研究得很深。但到了生成战场，很多人就松懈了，觉得"检索搞好了，大模型自然能答对"。

事实不是这样的。大模型是一个概率模型，不是一个复读机。你把正确答案送到它面前，它在生成回答的时候依然会受到预训练知识的"拉扯"，依然可能添油加醋、自由发挥。这种幻觉比检索不到正确文档更可怕——因为回答的 90% 是对的，只有 10% 是编的，用户很难察觉。

所以 RAG 的生成阶段不是"交给大模型就行了"，它需要同样严肃的工程化处理：Prompt 设计要精细到每一条规则，引用标注要做到每一句话有出处，后处理检测要能抓住每一个可疑的实体，文档排列顺序要考虑模型的注意力分布特征。

这些东西不像检索优化那么有"技术感"，但恰恰是决定 RAG 系统能不能真正上线、能不能让业务方放心用的关键一环。

🙋‍♂️他说："检索完之后再过滤——先从向量库里召回 Top-20，然后把用户没有权限看的文档剔掉。"

👨‍💻面试官顿了一下，接着问了一句让他脊背发凉的话："也就是说，你的向量数据库已经把董事长的薪酬文件检索出来了，只是前端没显示——但这个检索动作本身已经发生了。你觉得这过得了安全审计吗？"

🙋‍♂️他没答上来。

👨‍💻面试官最后说了一句总结："权限控制不是展示层的事，是数据访问层的事。你在错误的位置做了正确的事。"

这件事让我想起了一个很多人做 RAG 项目时不会注意到的盲区：教程和 demo 里从来不讲权限。从 LangChain 的官方教程到网上的各种 RAG 实战帖，流程都是一条直线：文档入库 → 向量检索 → 拼 Prompt → 大模型生成。全程没有"这个用户有没有权限看这份文档"这个判断。

但企业场景里，权限是红线。HR 的薪酬数据、法务的诉讼材料、高管的战略备忘录——这些东西不能因为你搭了个知识问答系统，就让全公司任何人都能搜到。

今天把 RAG 系统的权限控制从架构到实现讲清楚。

### 简要回答
RAG 系统的权限控制，核心问题在于"过滤逻辑放在哪一层"。

大部分人的第一反应是后置过滤——先检索再剔除没有权限的结果。这种方式实现简单，但存在三个根本问题：数据访问已经发生（即使没展示，检索动作本身就是一次数据读取，过不了安全审计）；Top-K 名额被无效占用（召回的 20 条结果里可能 15 条是无权访问的，实际可用的只剩几条，检索质量严重下降）；多租户场景下信息泄漏风险高。

正确的做法是前置过滤（Pre-filtering）：在向量数据库做相似度计算之前，先用权限条件把无权访问的文档排除掉，只在用户有权看到的文档子集里做检索。实现方式是给每个 Chunk 存上权限元数据（部门 ID、角色等级、文档密级等），检索时把用户的权限信息解析成 filter 条件传入。

面试里如果被问到这个话题，只说"做了过滤"不够——你要说清楚过滤放在哪一层、为什么要前置不要后置、权限元数据怎么设计、动态权限变更怎么处理。这才是一个完整的回答。

### 详细解析
#### 后置过滤为什么行不通
先把问题的严重性讲清楚。很多团队选后置过滤，不是因为不懂安全，而是因为这种方式太直观了——检索结果拿到手，遍历一遍，看看每条记录的dept_id跟当前用户的部门对不对得上，对不上就剔掉。几行代码就搞定，不改任何检索链路。

但这种"省事"的做法有三个致命问题。

第一个问题：数据已经被"摸过"了。向量检索是一次数据库读操作。不管你最后有没有展示给用户，那份被检索出来的文档已经被系统处理过了——向量计算已经发生，相似度分数已经算出来，文档内容可能已经被加载进内存。在有安全审计要求的企业里（尤其是金融、医疗、政务），审计系统看的是"这次请求访问了哪些数据"，不是"这次请求展示了哪些数据"。一个普通员工的每次搜索都在触碰他无权访问的薪酬档案、法律文件——这在合规上是不可接受的。

第二个问题：Top-K 被浪费了。向量检索返回的是 Top-K 条结果。假设你设了 K=20，但当前用户只有权限访问知识库中 30% 的文档，那 20 条结果里大概有 14 条是他看不了的。后置过滤把这 14 条剔掉，只剩 6 条——你以为给了 LLM 20 条高质量上下文，实际只有 6 条。更糟糕的是，真正最相关的那条文档可能排在第 25 位，因为前面被 14 条无权限的文档挤掉了，根本没进入 Top-20。

第三个问题：多租户场景泄漏风险。如果你的 RAG 系统服务多个客户（比如一个 SaaS 知识问答产品），不同客户的文档共存在同一个向量空间里。后置过滤意味着——客户 A 的检索请求在底层已经"看到"了客户 B 的文档向量。虽然没显示出来，但在向量索引层面，跨租户的数据是互相可见的。这不只是技术风险，还是商业信任问题。

#### 正确做法：前置过滤
理解了后置过滤的问题，前置过滤（Pre-filtering）的思路就很自然了——在向量相似度计算之前，先用权限条件把不该看的文档排除掉，只在剩余的文档子集里做检索。

具体怎么实现？分三步。

第一步：入库时给每个 Chunk 打上权限元数据。

这是前提。每个文档 Chunk 在存入向量数据库的时候，除了文本内容和 Embedding 向量，还要额外存一组权限相关的字段。最常见的权限模型有这几种：

部门级权限——这个 Chunk 属于哪个部门，只有对应部门的人能看。角色级权限——需要什么角色等级才能访问，比如普通员工只能看 level 1 的文档，经理能看 level 1-3，总监能看所有。文档密级——公开、内部、机密、绝密四个等级，用户的密级必须大于等于文档密级。

这些字段作为 metadata 跟 Embedding 向量一起存进向量数据库。

第二步：检索时解析用户权限，构造 filter 条件。

当用户发起搜索时，后端从用户的认证 Token（比如 JWT）里解析出他的身份信息——属于哪个部门、角色等级多少、密级是什么。然后把这些信息转换成向量数据库的过滤条件，跟 query 向量一起传入检索接口。

这段逻辑的关键在于：过滤发生在向量相似度计算之前。向量数据库先按 filter 缩小候选集，再在候选集内做最近邻搜索。无权限的文档从头到尾不会参与任何计算。

第三步：服务端验证，不信任客户端。

这一点很多人会忽略。有些实现是前端在请求里带上{"dept": "finance", "role": "manager"}，后端直接拿来构造 filter。问题是前端的参数用户可以随便改——把role从"staff"改成"admin"就能越权访问所有文档。

正确的做法是从 JWT Token 的签名验证中获取用户身份。Token 是服务端签发的，篡改了就验签失败。用户权限应该由后端从权限服务查询，而不是依赖前端传递的任何声明。

#### 多租户场景的隔离粒度
前置过滤解决了单企业内部的权限问题。但如果你的 RAG 系统服务多个客户（多租户 SaaS），隔离要求更高。这时候有三种方案可选，安全性依次递增。

Metadata 过滤隔离。所有租户的文档放在同一个 Collection 里，用tenant_id字段做过滤。实现最轻量，但有隐患——如果某次检索的 filter 因为代码 bug 没带上tenant_id，就会暴露所有租户的数据。适合安全要求不高的场景。

Partition 隔离。每个租户一个 Partition，数据物理上分开存储，但共享索引结构。检索时强制指定 Partition，比 Metadata 过滤多了一层物理隔离。中等安全要求的 SaaS 产品通常用这种方案。

Collection 隔离。每个租户独立一个 Collection，完全物理隔离。安全性最高，但资源成本也最高——每个 Collection 都有独立的索引和存储开销。适合金融、医疗等强合规场景。

实际项目中一个常见的折中方案是Partition + Metadata 双层防护：先用 Partition 做物理隔离，再在 Metadata 里记录tenant_id做二次校验。即使 Partition 路由出了问题，Metadata 过滤还能兜底。两层机制同时失效的概率极低。

#### 动态权限变更：最容易被忽视的坑
静态权限好处理，真正麻烦的是权限动态变化——员工升职了能看更多文档、调岗了访问范围变了、离职了要立刻禁止访问。

如果你把权限信息直接写死在每个 Chunk 的 Metadata 里（比如allowed_depts: ["finance", "legal"]），那当 finance 部门拆分重组的时候，你需要批量更新知识库里几万条 Chunk 的元数据。在向量数据库里做这种大规模更新，既慢又容易漏。

更好的设计是权限数据和文档数据分离存储。Chunk 的 Metadata 里只存文档 ID，不存具体的权限规则。权限关系放在关系型数据库（比如 PostgreSQL）里统一管理。检索的时候，先从权限数据库查出当前用户有权访问的文档 ID 列表，然后把这个列表作为 filter 传给向量检索。

这种方式的好处是：权限变更只需要修改关系型数据库里的一条记录，不需要动向量数据库里的任何数据。员工调岗、离职，改的是权限表，不是知识库。

代价是每次检索都要查一次权限数据库。优化方式是加一层短时缓存（比如 Redis，TTL 设 60 秒）——大部分时候直接命中缓存，延迟几乎为零。权限发生变更时，主动清除对应用户的缓存条目，下次检索会重新查询最新权限。

有一个特殊场景需要额外注意：员工离职。这是权限控制里最敏感的操作，要求权限必须立即失效，不能有 60 秒的缓存延迟。处理方式是在离职流程触发时，同步删除该用户的所有缓存条目，确保下一次检索立刻拿到"无权限"的状态。

### 面试里怎么谈权限控制
这道题面试官爱问，是因为它能直接区分"做过 demo"和"做过企业项目"的候选人。所有 RAG 教程都不讲权限，但所有企业 RAG 都必须有权限——你有没有做过，一问就知道。

第一个容易翻车的点：只说"做了过滤"，不说在哪一层。面试官追问"检索前还是检索后"，你答"检索后"，面试官就会追问数据访问的合规性问题。要主动说清楚你做的是 Pre-filtering，并且能解释为什么后置过滤在安全模型上是错误的。

第二个容易翻车的点：不知道 Top-K 名额被占用的问题。这是后置过滤最直接的工程影响——检索质量因为权限过滤而严重下降。面试官问"后置过滤除了安全问题还有什么问题"，你能答出这一点就说明你确实在真实数据上跑过。

第三个容易翻车的点：信任前端传参。面试官问"用户权限信息怎么获取的"，你说"前端传过来的"——这在安全上是不合格的。正确答案是从服务端验证的 JWT Token 中解析，不信任客户端的任何声明。

加分项：能谈到动态权限变更和缓存策略。大部分候选人聊到前置过滤就停了。如果你能主动提到"权限数据不写死在 Chunk 里，放在关系型数据库实时查询，配合缓存做性能优化，离职场景主动失效缓存"——这种细节说出来的人不多，说明你确实思考过生产环境的完整生命周期。

一个完整的面试回答大概是这样的结构：先说后置过滤的三个问题（30 秒）→ 说 Pre-filtering 的实现方案（1 分钟）→ 说多租户隔离的三种粒度和选型依据（30 秒）→ 说动态权限变更的处理（30 秒）。整个回答控制在 3 分钟以内，层次清晰，从问题到方案到细节一气呵成。

### 写在最后
权限控制是企业级 RAG 和 demo 级 RAG 之间最容易被忽视、但也最容易暴露水平的分界线。

做一个没有权限控制的 RAG demo，可能只需要一个下午。但让这个系统过得了企业的安全审计、扛得住多租户的数据隔离要求、应对得了人员调动带来的权限动态变更——这些工作量可能比搭建检索链路本身还要大。

很多人做 RAG 项目的精力全花在"怎么让检索更准"上——调 Embedding、调分块策略、加 Rerank、做混合检索。这些当然都重要，但如果你的系统连"谁有权看什么"都没想清楚，那它就永远只是一个 demo，到不了生产环境。

准确性决定了系统好不好用，安全性决定了系统能不能上线。这是两个维度的事情，缺一个都不行。

结果就是：该查知识的去算数了，该算数的去查知识了，该查数据库的去搜文档了。什么都做，什么都做不好。

今天把 RAG 系统的 Query 理解模块——意图识别、实体提取、路由分发——彻底讲清楚。

### 简要回答
RAG 系统在接收到用户问题后，不应该无差别地全部扔给向量检索。正确的做法是在检索之前加一个 Query 理解层，先搞清楚用户的意图是什么，再决定走哪条处理链路。

最基本的意图分类至少要区分四种：知识问答类（走向量检索）、计算求解类（走计算模块）、数据查询类（走 NL2SQL 或结构化查询）、闲聊类（直接让 LLM 回答或礼貌引导）。意图判断之后还要做实体提取，把用户问题里的关键约束条件（时间、数值、来源）抽出来，辅助后续的检索过滤或计算参数注入。

这件事的工程实现不复杂，但它决定了 RAG 系统的天花板。没有意图识别的 RAG 就像一个不看路标的导航——偶尔能走对，但大部分时候在绕路。

面试里如果被问到"你的 RAG 系统怎么处理不同类型的 query"，只说"全部走向量检索"基本等于告诉面试官你没做过真正的生产系统。

### 详细解析
#### 为什么 RAG 不能"一条路走到黑"
先想一个最简单的问题：用户在一个保险问答系统里可能问什么？

第一种，知识查询型："重疾险的等待期是多少天？"这种问题在知识库里有现成答案，向量检索 + LLM 生成是最佳路径。

第二种，计算求解型："我买了 50 万保额的重疾险，免赔额 1 万，住院花了 8 万，能赔多少？"这里面有三个明确的数值参数，需要套公式计算，不是从文档里能"搜出来"的。

第三种，数据查询型："我上个月提交的理赔申请现在到哪一步了？"这需要去业务数据库里按用户 ID 查记录，跟知识库半毛钱关系都没有。

第四种，闲聊型："你好""谢谢""今天天气不错"。这些跟业务无关的对话，根本不应该触发任何检索，直接让 LLM 回复就行。

如果你的系统不分青红皂白，所有问题都往向量检索这条路上送，会发生什么？

计算型的 query 去搜知识库，可能搜到一堆讲理赔规则的文档，LLM 拿着这些文档试图"推理"出赔偿金额——但大模型的数学能力你也知道，稍微复杂一点的算术它就可能算错。明明一个确定性的计算用 Python 一行代码就能搞定，非要让概率模型来碰运气。

数据查询型的 query 去搜知识库，搜出来的可能是"理赔流程一般需要 5-15 个工作日"这种通用信息，根本不是用户想知道的"我的那笔具体进展"。

这就是没有意图识别的代价：该精确的变成了模糊的，该确定的变成了概率的。

#### 意图识别怎么做：三级方案
知道了为什么要做，接下来看怎么做。意图识别的实现方案有三种，按复杂度递增排列。

第一种：规则匹配。最朴素但最快。维护一张关键词映射表：query 里出现了"多少钱""怎么算""赔付金额"这些词，就分到计算类；出现了"进度""状态""我的申请"就分到数据查询类。实现零延迟，可控性强，但覆盖面有限——用户换个说法你就匹配不上了。

第二种：ML 分类模型。用 BERT 或者轻量级文本分类模型，在标注数据上训练一个四分类器。它能识别同义表达，用户说"帮我看看能拿到多少补偿"也能分到计算类，比关键词鲁棒得多。但需要几百条标注数据来训练。

第三种：LLM 零样本分类。直接在 Prompt 里告诉大模型"请判断以下问题属于哪个类别"，不需要任何训练数据，上手最快。但每次调用要消耗一次 LLM 推理，有延迟也有成本。而且 LLM 偶尔会判断失误，稳定性不如训练好的专用模型。

实际工程中最靠谱的方案是三级组合：规则先行 → ML 兜底 → LLM 处理疑难。

具体的流程是这样的：query 进来先走规则匹配，能命中就直接分类，零延迟；规则匹配不上的，走 ML 分类器，几十毫秒出结果；ML 分类器信心度低的（比如 softmax 最大概率只有 0.4，说明模型自己也不太确定），才调 LLM 做最终判断。

这套组合的好处是：绝大部分 query 在规则层就被处理了，速度极快；少量边界 case 由 ML 模型处理，准确率有保障；极少数的疑难 query 才动用 LLM，成本可控。

#### 实体提取：从 query 里把关键参数挖出来
意图识别解决了"走哪条路"的问题，但光知道意图还不够。很多 query 里藏着关键的约束条件，不把它们提取出来，后续处理就会丢失精度。

举一个我实际遇到的例子。用户问："上周《经济日报》报道的半导体行业景气度排名第几？"

这句话里至少有四个关键实体：时间（"上周"）、来源（"经济日报"）、行业（"半导体"）、查询目标（"景气度排名"）。

如果你不做实体提取，直接把整句话丢给向量检索，它大概率会基于"半导体行业景气度"这几个关键语义去匹配——可能搜出一堆近期关于半导体行业的分析文章，但不一定是"上周经济日报"那篇。用户想要的是一个特定来源、特定时间的特定内容，向量检索的语义匹配做不到这种精确度。

但如果你提取出了时间实体"上周"，检索时就能在元数据层加时间过滤，只搜最近一周的内容；提取出来源"经济日报"，就能限定文档来源。这样检索的精准度会大幅提升。

实体提取的实现有几种方式。对于日期、数字、金额这类有固定格式的实体，用正则表达式又快又准。对于人名、公司名、行业名这类开放性实体，可以用 NER 模型或者直接让 LLM 做结构化抽取。工业界的常见做法是正则 + NER 混合，各取所长。

#### 路由分发：把 query 送到对的地方
意图识别和实体提取都做完了，接下来就是路由——根据分析结果，把 query 送到最合适的处理链路。

意图是"知识问答"→ 走标准的向量检索 + BM25 混合检索链路。如果实体提取拿到了时间约束，就在检索时加上时间过滤；如果拿到了文档来源约束，就限定检索范围。

意图是"计算求解"→ 跳过向量检索，直接走计算模块。从 query 中抽取出数值参数（保额、免赔额、花费金额等），注入到预定义的计算函数或者让 LLM 编写并执行计算代码。结果是确定的，不存在"幻觉"的风险。

意图是"数据查询"→ 走 NL2SQL 模块。让 LLM 把自然语言转成 SQL，从业务数据库里查出用户的具体数据。比如把"我的理赔进度"翻译成 `SELECT status FROM claims WHERE user_id = xxx ORDER BY created_at DESC LIMIT 1`。

意图是"闲聊"→ 不走任何检索，直接让 LLM 以通用对话模式回复，或者礼貌地引导用户回到业务话题上来。

如果你的知识库按主题分成了多个索引（比如"理赔制度""产品信息""投保指南"各一个索引），还可以在路由阶段做多索引选择——根据 query 的主题选择最相关的索引去检索，而不是在全库里大海捞针。这样既提高了检索精度，又减少了不必要的计算量。

#### 一个容易被忽视的问题：路由错了怎么办
这一层防线很多人不会想到，但在生产环境里非常重要。

意图分类器不是 100% 准确的。如果它把一个本该走检索的 query 错误地分到了计算链路，用户就彻底拿不到想要的答案。这比不做意图识别、直接走检索还要糟——至少走检索还有概率碰上对的文档。

所以有一个关键原则：信心不够时，宁可保守走默认检索，也不要冒险走错链路。

具体的实现策略有两个。第一个是设信心度阈值——ML 分类器对某个类别的预测概率低于 0.6，就不走那个分支，回退到默认的检索链路。第二个是多路径并行——同时走两条路（比如既做向量检索，也做计算），最后对比两边的结果取更好的一个。代价是计算量翻倍，但对于准确性要求极高的场景，这个代价值得付。

另外，还需要有回退机制。如果计算模块返回了一个明显异常的结果（比如赔付金额算出了负数），或者 NL2SQL 生成的 SQL 执行报错了，系统应该自动回退到检索链路重新处理，而不是把错误结果直接返回给用户。

### 面试怎么答
这道题面试官爱问，因为它能快速看出你对 RAG 系统有没有全局设计的思维。大部分候选人谈 RAG 就是谈检索——向量检索怎么做、BM25 怎么配、Rerank 怎么排。但检索只是 RAG 系统的一个环节，在检索之前的 Query 理解才决定了"这个问题该不该检索、该怎么检索"。

第一个容易翻车的点：不知道 RAG 有路由层。面试官问"你的系统怎么处理不同类型的 query"，你说"都走向量检索"——这就暴露了你的系统没有分层设计。面试官会觉得你做的是一个 demo，不是生产系统。

第二个容易翻车的点：做了意图识别但没有兜底策略。面试官追问"如果你的分类器判断错了怎么办"，你说不上来——这说明你在工程上没有考虑过 failure case。任何分类器都有出错的时候，有兜底才是成熟的工程设计。

第三个容易翻车的点：只讲了意图识别，没讲实体提取。意图识别决定走哪条路，实体提取决定路上带什么参数。两个缺一个，系统都不完整。

一个比较完整的面试回答框架是四层递进：先讲为什么需要 Query 理解模块（不同 query 需要不同处理链路，不区分就是在碰运气）→ 再讲三级意图识别方案（规则先行、ML 兜底、LLM 处理疑难）→ 然后讲实体提取和路由策略（时间过滤、来源限定、多索引选择）→ 最后讲安全机制（信心度回退、异常兜底、多路径并行）。

如果你还能补一句具体数字，比如"我们上线意图识别之后，用户的首次回答满意率从 61% 提升到了 78%，主要是计算类和数据查询类的 query 不再被错误地走检索链路了"，面试官会更加相信你是真做过这件事的。

### 写在最后
Query 理解模块在很多 RAG 教程和开源项目里是缺失的。你看大部分教程的流程图，都是画一条直线：用户提问 → Embedding → 向量检索 → Top-K 拼 Prompt → LLM 生成。干净利落，但在真实业务场景里根本扛不住。

因为真实用户不会只问"知识类"的问题。他们会问计算题，会问自己的业务数据，会说"帮我看看上次那个事儿怎样了"，甚至会纯粹闲聊。一个成熟的 RAG 系统不能只有一条路——它需要一个"指挥中心"，先搞清楚用户想干什么，再分配最合适的处理方式。

这就是 Query 理解模块的价值。它不做检索，也不做生成，但它决定了检索和生成的质量上限。

很多时候，RAG 系统效果不好，大家第一反应是去调 Embedding 模型、调分块策略、加 Rerank。这些当然都有用，但如果你的 query 从一开始就被送到了错误的链路上，后面做再多优化也是白费功夫——方向错了，越努力越跑偏。

先把路认对，再考虑怎么跑得更快。
