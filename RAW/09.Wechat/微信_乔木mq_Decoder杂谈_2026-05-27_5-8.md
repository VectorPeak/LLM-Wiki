## 0x01. 大模型：Multi-Query Attention
> 发布日期：2026-05-27  
> 原文链接：[大模型：Multi-Query Attention](https://mp.weixin.qq.com/s/-cJKhcpN74vJh_pWgelw2g)

一、Multi-Query Attention 的技术背景与动机

 在基于 Transformer 架构的大语言模型自回归推理过程中，生成每个新 Token 都需要依赖此前所有 Token 的历史信息。为了避免重复计算，系统通常会将历史 Token 的键(Key)和值(Value)矩阵缓存在 GPU 显存中，这一机制被称为 KV 缓存。随着上下文序列长度的增加以及模型参数规模的扩大，KV 缓存占用的显存空间会呈线性增长，进而导致巨大的内存带宽压力，成为限制大模型推理速度与并发量的核心瓶颈。

 传统的多头注意力机制虽然通过多个独立的注意力头增强了模型的表达能力，但在推理阶段，每个头都需要独立存储和读取各自的 KV 缓存，造成了极大的资源冗余。为了解决这一问题，Google Research 于 2019 年提出了 Multi-Query Attention 技术。该技术的核心思想是在保留多头 Query 以维持模型特征提取多样性的同时，让所有注意力头共享同一份 Key 和 Value 投影，从而大幅降低显存占用并提升推理效率。

 二、Multi-Query Attention 的底层原理与数学推导

 传统多头注意力机制的计算逻辑
 在标准的多头注意力机制中，假设隐藏层维度为 d，注意力头数为 h，则每个头的维度为 d_h = d / h。对于输入序列，模型会通过三个独立的权重矩阵 W_Q、W_K、W_V 分别进行线性投影，得到 Query、Key 和 Value 矩阵。其第 i 个头的注意力计算公式如下：
$$
Attention_i(Q_i, K_i, V_i) = softmax((Q_i *K_i^T) / sqrt(d_h))* V_i
$$
 其中，Q_i、K_i、V_i 均为第 i 个头独有的投影结果。在推理阶段的 KV 缓存中，需要存储的形状为 [batch_size, seq_len, h, d_h] 的张量，显存占用与头数 h 成正比。

 Multi-Query Attention 的结构优化
 Multi-Query Attention 对上述结构进行了极简改造。它保留了 Query 的多头结构(即 Q 依然被切分为 h 个头)，但将 Key 和 Value 的投影矩阵降维，使得所有头共享同一组 K 和 V。其数学表达变为：
$$
Attention_i(Q_i, K_shared, V_shared) = softmax((Q_i *K_shared^T) / sqrt(d_h))* V_shared
$$
 在这种设计下，K_shared 和 V_shared 不再具有“头”这一维度(或者说头数为 1)。因此，KV 缓存的形状从 [batch_size, seq_len, h, d_h] 缩减为 [batch_size, seq_len, 1, d_h](在实际工程实现中通常会利用广播机制与 Q 进行计算)。

 三、工程实现细节与性能收益分析

 显存与带宽的优化
 从工程落地的角度来看，Multi-Query Attention 带来的最大收益在于 KV 缓存体积的显著缩减。假设模型拥有 32 个注意力头，采用该技术后，KV 缓存的参数量理论上可减少至原来的 1/32。在长文本生成场景下，这意味着单条请求占用的显存大幅降低，使得在消费级显卡上部署更大参数的模型或处理更长的上下文窗口成为可能。此外，由于解码阶段属于内存密集型操作，减少 KV 缓存的数据量直接降低了从显存读取数据的带宽需求，从而显著提升了 Token 的生成速度。

 计算兼容性与广播机制
 在具体的代码实现(如 PyTorch)中，Multi-Query Attention 并不需要修改底层的矩阵乘法算子。当形状为 [batch, heads, seq_len, head_dim] 的 Query 与形状为 [batch, 1, seq_len, head_dim] 的 Shared-Key 进行点积运算时，深度学习框架会自动触发广播机制，将 K 和 V 在头维度上进行扩展以匹配 Q 的形状。这种设计保证了在不牺牲太多并行计算效率的前提下，实现了存储结构的轻量化。

 四、技术局限与分组查询注意力的演进

 尽管 Multi-Query Attention 在推理加速方面表现优异，但极端的参数共享也带来了一定的副作用。由于所有头强制共享同一组键值对，模型的表达能力和拟合复杂语义关系的能力会受到限制，可能导致训练过程中的不稳定性或在特定任务上的精度下降。

 为了在“推理效率”与“模型精度”之间寻求更优的平衡，学术界进一步提出了分组查询注意力技术。该技术作为 Multi-Query Attention 的泛化形式，不再让所有头共享一份 KV，而是将 Query 头划分为若干个组，每个组内部共享一份 KV。例如 Llama 3 等主流开源模型便采用了此方案。分组查询注意力既保留了 Multi-Query Attention 的大部分显存优化优势，又通过引入多组 KV 恢复了模型的部分多样性表征能力，是目前大模型架构设计中兼顾性能与效率的主流选择。

## 0x02. 算法专栏：Encoder-Decoder
> 发布日期：2026-05-26  
> 原文链接：[算法专栏：Encoder-Decoder](https://mp.weixin.qq.com/s/IweSZxdtJQFGPeEmMj6tzg)

大语言模型的Encoder-Decoder架构是Transformer原始论文中提出的经典序列到序列(Seq2Seq)范式。该架构将输入理解与输出生成解耦为两个独立的模块，在机器翻译、文本摘要等转换类任务中具有重要的应用价值。其核心逻辑与技术细节主要包含以下三个层面：

 一、 模型架构的组成与工作机制

 Encoder-Decoder架构由编码器(Encoder)和解码器(Decoder)两部分堆叠而成，两者通过特定的注意力机制进行交互。
 1. 编码器(Encoder)的特征提取
 编码器的核心任务是“理解”输入序列。它通常由多个相同的层堆叠而成，每层包含多头自注意力机制和前馈神经网络。在处理输入时，编码器采用双向注意力机制，能够同时关注输入序列中的所有词元(Token)，从而捕捉长距离的上下文依赖关系，并将变长的输入序列转化为致密的特征向量表示。
 2. 解码器(Decoder)的自回归生成
 解码器的核心任务是“生成”输出序列。除了包含与编码器类似的结构外，解码器还引入了掩码多头自注意力机制和编码器-解码器注意力机制(Cross-Attention)。掩码机制确保了解码器在预测当前词元时，只能看到历史生成的词元，保证了生成的自回归特性。而交叉注意力机制则允许解码器在生成过程中动态地关注编码器输出的输入特征，实现输入与输出之间的语义对齐。

 二、 架构变体与适用场景

 随着深度学习的发展，基于原始Transformer衍生出了多种架构变体，以适应不同的任务需求。
 1. 经典的Encoder-Decoder架构
 代表模型包括T5、BART等。该架构保留了完整的编码器和解码器结构，非常适合输入与输出界限清晰的序列转换任务，如机器翻译、文本摘要及语法纠错。
 2. 仅编码器(Encoder-only)架构
 代表模型为BERT、RoBERTa。此类架构去除了解码器，仅保留编码器部分，利用双向注意力机制深度提取输入序列的语义特征，主要适用于文本分类、命名实体识别等理解类任务。
 3. 仅解码器(Decoder-only)架构
 代表模型为GPT系列、LLaMA、Qwen等。此类架构去除了编码器，将输入序列作为上下文直接输入解码器，通过单向自回归的方式预测下一个词元。尽管结构简化，但其在海量数据预训练下涌现出了强大的通用能力，成为当前大语言模型的主流选择。

 三、 架构特性的局限与工程考量

 尽管Encoder-Decoder架构在特定领域表现优异，但在大规模语言模型的演进中也暴露出一定的局限性。
 1. 参数利用率与计算效率
 在同等参数量级下，Encoder-Decoder架构的参数利用率相对较低。例如在推理阶段，当编码器处理完输入提示词(Prompt)后便不再参与后续计算，生成过程完全依赖解码器。这意味着模型占用了一定的显存资源，却未能充分发挥全部参数的计算效能。
 2. 扩展定律与涌现能力
 实验表明，随着模型规模的扩大，纯解码器(Decoder-only)架构在 Scaling Law(扩展定律)下更容易涌现出复杂的逻辑推理和上下文学习能力。相比之下，传统的Encoder-Decoder架构在超大规模参数下的性能增长曲线相对平缓，这也是当前千亿级以上生成式大模型普遍转向Decoder-only架构的主要原因。

## 0x03. 算法专栏：MLLM
> 发布日期：2026-05-26  
> 原文链接：[算法专栏：MLLM](https://mp.weixin.qq.com/s/otAgLvo80JPB802d250dZg)

多模态大语言模型(Multimodal Large Language Models, MLLMs)是人工智能从单一文本处理迈向多维度感知与认知的重要技术演进。它打破了传统大型语言模型(LLM)仅能处理单模态文本数据的局限，通过融合视觉、音频等多种异构信息，构建起跨模态的统一语义空间。其核心算法架构与工作机制主要包含以下几个层面：

 一、MLLM的核心架构组成

 多模态大语言模型的典型架构通常由模态编码器(Modality Encoder)、连接器(Input Projector/Connector)以及作为核心基座的大型语言模型(LLM Backbone)三部分构成。
 1. 模态编码器
 这是多模态融合的“特征提取关口”，负责将不同模态的原始输入数据(如图像、视频、音频等)编码为模型可处理的特征表示。主流方案包括双编码器架构(如CLIP模型，分别对图像和文本进行独立编码并映射到同一特征空间)和融合编码器架构(在编码阶段即进行模态特征的深度融合，以提取更深层次的跨模态关联)。
 2. 连接器
 连接器的作用是将模态编码器输出的特征向量，映射到大型语言模型易于理解的语义空间，转化为视觉Token或其他形式的嵌入表示。研究表明，相较于连接器的具体类型，视觉Token的数量以及输入图像的分辨率对模型最终的性能影响更为显著。
 3. 大型语言模型基座
 作为整个系统的核心推理引擎，大型语言模型接收经过连接器处理后的多模态Token序列，利用其强大的自然语言理解与生成能力，完成跨模态的逻辑推理、指令遵循及内容生成任务。

 二、跨模态语义对齐与交互机制

 MLLM的技术关键在于如何实现不同模态信息在统一语义空间内的有效对齐与交互。
 1. 统一表征空间的构建
 通过对比学习等方法，模型在海量“图像-文本”或“音视频-文本”配对数据上进行预训练，拉近正样本对的特征距离，推远负样本对的特征距离。这一过程使得不同模态的数据能够被映射到同一个高维向量空间中，从而实现语义层面的互通。
 2. 深层跨模态推理
 相比于简单的相似度匹配，先进的MLLM能够在编码和解码过程中充分综合多模态信息。例如，在处理视觉问答任务时，模型不仅能识别图像中的物体，还能结合文本指令进行复杂的逻辑推演，挖掘出隐藏在跨模态数据背后的深层语义关联。

 三、技术演进与应用前景

 随着技术的不断发展，多模态大语言模型正在推动人工智能系统突破“感官单一”的壁垒，向更接近人类感知的方向发展。
 1. 从被动认知到主动预测
 当前的MLLM已经具备了强大的跨模态理解能力。未来的演进方向将进一步结合世界模型(World Models)与具身智能(Embodied AI)，使模型不仅具备多维感知能力，还能构建环境动态的内部表征，实现“感知-决策-执行”的闭环，从而在物理世界中完成更复杂的自主行动任务。
 2. 广泛的应用场景
 凭借对文本、图像、声音等多种信息的整合处理能力，MLLM在自动驾驶、智能助理、内容推荐系统、教育培训以及医疗影像分析等领域展现出广阔的应用前景，能够提供更加丰富、自然且高效的人机交互体验。

## 0x04. 算法专栏：Causal Decoder
> 发布日期：2026-05-27  
> 原文链接：[算法专栏：Causal Decoder](https://mp.weixin.qq.com/s/sCbq0f5hVwo9f9n2aNyUyQ)

### 1. Causal Decoder 的基本定位
在生成式大模型(Generative Large Language Models)的体系下，Causal Decoder 是指一类采用因果注意力机制(Causal Attention Mechanism)的解码器架构。它与传统的自回归解码器(Autoregressive Decoder)在结构上高度相似，但在注意力掩码方式、训练目标以及上下文建模范围方面存在关键区别。

标准的因果解码器(如 GPT 系列)对输入序列中的每个位置施加严格的因果掩码，即每个词元(Token)只能关注其左侧的历史词元，而不能看到当前及未来的词元。Causal Decoder 在保持这一因果约束的同时，通常会通过特定的预训练任务(如前缀语言建模或填充语言建模)引入一定程度的双向上下文感知能力。

### 2. Causal Decoder 与标准因果解码器的结构差异
#### 1. 注意力掩码的层级设计
标准的自回归解码器采用固定的下三角掩码矩阵，使得第 i 个位置的注意力仅能覆盖位置 1 至 i-1。Causal Decoder 在此基础上引入了分段掩码或多级掩码的策略。具体而言，在预训练阶段，Causal Decoder 会将输入序列划分为若干片段。对于每个片段内部，模型可以采用双向注意力(即类似编码器的自注意力机制)；而对于跨片段的依赖关系，则仍保持因果约束。

这种设计使得模型在生成任务中保留了自回归的推理特性，同时在理解任务中能够获得更丰富的上下文表示。

### 2. 训练目标的双重性
Causal Decoder 通常采用混合训练目标。一方面，它使用标准的自回归语言建模损失，即预测下一个词元的交叉熵损失。另一方面，它还会引入掩码语言建模损失，随机遮蔽序列中的部分词元并要求模型基于上下文(包括右侧信息)进行预测。这两种损失的加权组合使得 Causal Decoder 在生成质量和上下文理解之间取得平衡。

### 3. Causal Decoder 的数学表达
给定长度为 T 的输入序列 X = (x1, x2, …, xT)，其因果注意力输出可表示为：
$$
Attention(Q, K, V) = softmax( (QK^T) / sqrt(d_k) + M ) V
$$
其中 M 为因果掩码矩阵，满足：当 i ≤ j 时，M_ij = 0；当 i > j 时，M_ij = -∞。对于 Causal Decoder，掩码矩阵 M 被扩展为分块形式：
$$
M = [ M_local, M_cross ][ M_causal, M_causal ]
$$
其中 M_local 对应片段内双向注意力，M_causal 对应片段间因果约束。这种分块掩码使得模型在计算复杂度上仍保持 O(T^2)，但实际内存访问模式与标准因果解码器存在差异。

### 4. 工程实现中的优化要点
#### 1. KV Cache 的差异化处理
在推理阶段，标准因果解码器的 KV Cache 策略较为直接：每生成一个新词元，将其对应的 Key 与 Value 追加到缓存中。Causal Decoder 由于存在片段内双向注意力，其 KV Cache 管理变得复杂。一个常见优化是在预填充(Prefill)阶段预先计算所有词元的 Key 与 Value，但在解码阶段仅更新当前片段的缓存。这要求实现时区分全局缓存与局部缓存两层结构。

#### 2. 掩码矩阵的稀疏化
由于 Causal Decoder 的掩码并非严格的三角形，直接存储稠密掩码会带来较大的显存开销。工程上通常采用块稀疏(Block Sparse)存储格式，将掩码按 32×32 或 64×64 的块进行压缩，利用 GPU 的共享内存(Shared Memory)加速掩码与注意力分数的加法运算。

### 5. Causal Decoder 与 Encoder-Decoder 的对比
在 Transformer 的原始架构中，编码器(Encoder)采用双向注意力，适用于理解任务；解码器(Decoder)采用因果注意力，适用于生成任务。Causal Decoder 介于两者之间：它不需要独立的编码器模块，所有参数共享在同一网络中，通过注意力掩码控制不同位置的可见范围。

与编码器-解码器架构相比，Causal Decoder 在参数量上更为紧凑，避免了编码器与解码器之间的交叉注意力计算，从而在推理延迟上具有一定优势。同时，由于能够通过掩码切换实现双向上下文建模，它在机器翻译、文本摘要等序列到序列任务上的表现优于标准的因果解码器。

### 6. 典型模型实例
在实际的大模型应用中，GLM(General Language Model，由清华大学与智谱 AI 提出)系列模型采用了类似 Causal Decoder 的设计。GLM 在预训练阶段随机遮蔽连续片段，并要求模型按自回归方式生成被遮蔽的内容。这一过程中，模型对未被遮蔽的部分使用双向注意力，对生成部分使用因果注意力，本质上就是一种 Causal Decoder 的实现。

此外，UniLM(Microsoft)也在共享 Transformer 层的基础上，通过动态切换注意力掩码实现了自然语言理解、生成和序列到序列任务的统一，其核心结构亦可归入 Causal Decoder 的范畴。
