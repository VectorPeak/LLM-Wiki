## 一. vLLM 浅析

> 发布日期：2026-03-31  
> 原文链接：[vLLM 浅析](https://mp.weixin.qq.com/s/yOgPwG4g5bAjj5reNw5sNQ)

随着大语言模型能力不断衍进，以LLM为基座的应用或服务爆发性增长，这对模型的推理能力提出了新的挑战：需要同时满足 **高吞吐、低延迟**，并实现 **内存/显存的高效利用** 等关键目标。

 这些需求推动了云原生在AI基础设施中的落地，一系列围绕着 Kubernetes 原生构建的高性能分布式推理框架应运而生。这些框架通过更先进的 **显存管理** 与 **请求调度** 策略，在 Kubernetes 环境中实现接近硬件上限的推理效率，并进一步提供 **GPU 资源调度优化** 、 **多副本负载均衡** 、 **模型热加载** 与 **弹性伸缩** 等能力，逐步成为现代模型推理服务的基础设施底座。

 vLLM 是一个源于 UC Berkeley 的开源项目，它本身是一个专为高吞吐，低延迟LLM推理设计的系统。它的核心要点在于 **分页的KV缓存管理** 和 **连续批处理机制(Continuous Batching)**，本篇文章主要围绕vLLM处理推理请求的核心展开，会逐一解析其中的关键部分。

#### PagedAttention 与 KV Cache 管理
 **PagedAttention** 是 vLLM 最核心的设计之一。它借鉴了操作系统中“分页内存管理”的思想，将原本需要连续分配的 KV Cache 拆解为固定大小的块(block)进行管理，从而解决了传统实现中显存浪费与碎片化严重的问题。

 传统 Transformer 推理往往为每个序列预分配最大长度的连续显存空间存储 K/V 矩阵，导致大量未用空间浪费和内存碎片。而 **PagedAttention** 将每个请求的 KV 缓存拆解成固定大小的 **块(block)**，每个块可以容纳预定数量的Token的 KV，这些块在显存中可以分散存储不必强制连续，它由块表(block table)统一管理，类似于操作系统将内存分页存储并通过页表映射给进程一样，直接实现按需加载和复用。

 在vLLM中一个请求相当于一个"进程"，它的KV缓存由一系列 **块** 组成，这些块相当于进程的虚拟地址空间。新的Token产生时，调度器会调用 KVCacheManager 分配一个空闲块/或找到已存在的共享块给这个Token KV存储，并且记录块ID信息。

 这些块ID在请求的整个生命周期中保持不变，模型在运行计算时通过 **块ID** 将KV写入正确的位置。因为所有请求共享同一块池，未使用的块可以被任何新请求拿来复用，提高内存利用率。

 PagedAttention 还引入**前缀共享(Copy-on-Write)**优化：如果多个请求有相同的提示前缀，vLLM 可以让它们共享前缀对应的KV缓存块，而不重复存储。这在聊天对话的场景中非常实用，用户反复询问类似问题时使用这种方式可以显著减少总内存占用。当新的请求进来，且它的前缀是新的才能将其后续部分写入新的块，实现写时复制。这个机制很显然也是借鉴了操作系统内存页面写时复制的策略，相同上下文无需重复计算和存储，极大地提升了多请求场景下的效率。

 从执行路径上看，一个请求进入 vLLM 后，通常会先经历 prefill 阶段，完成初始 prompt 的前向计算并建立对应的 KV Cache；之后进入 decode 阶段，在调度器的控制下按轮参与动态组批。随着部分请求结束、部分请求补入，整个系统的 batch 组成会持续变化，而底层的 KV blocks 也会被不断分配、复用与回收。这种“块化内存管理 + 动态调度”的组合，构成了 vLLM 高性能推理能力的基础。

#### **Continuous Batching 与调度器** 
传统的静态批处理处理任务的方式是，积累一批请求然后再同时执行，等待这批任务全部结束后再执行下一批。但在LLM推理中，不同的请求 Prompt长度，生成的结果长度，问题的复杂度都不相同。如果使用传统的批处理方式会出现大量的“等待”和“批等待”。简短的请求已经计算完毕，但要等待复杂的长请求。新的请求已达到又要等待下一轮批处理的开始。这也是传统的框架中降低GPU利用率的原因。
 vLLM 将 batch 看作一个持续流动的执行集合。每经历过一轮调度后，系统都会重新审视所有请求，以此来判断各个请求在哪个阶段。调度器会基于当前系统状态组织下一轮执行批次，通过持续接纳新请求，移除已完成的请求，让GPU尽量保持在工作状态。

 通常来说，一个请求会经历两个阶段： **prefill** 和 **decode**。prefill 阶段需要把整段 prompt 喂给模型，在这个过程中要对这些 Token 做完整前向计算，做大量密集的矩阵计算。decode 阶段则是基于已有 KV cache 逐 token 生成，单步计算量会比较小，但是会做频繁的KV读写。

 这两个工作阶段截然不同，偏向也不太一样，默认调度策略和启用 chunked prefill 之后的策略并不一样；尤其在启用 chunked prefill 后，调度器会优先处理 decode 请求，再利用剩余 token budget 安排 prefill，必要时把长 prefill 切块执行。

 调度器的重点在于，它更像是一个不断做局部最优选择的执行器，它分析哪些 decode 必须先推进，哪些 prefill 可以插入，剩余 token budget 还够不够……

 这个机制配合 PagedAttention，因为 KV cache 已经被块化管理，所以请求即使被分多轮、穿插着执行，它的状态也不会因为 batch 变化而被整体搬动；调度器只需要知道这个请求当前对应哪些 blocks、下一步该推进多少 token 即可。

 vLLM 的性能提升不是单点优化，而是系统级设计，这两个部分是相辅相成的。只有高效的 KV Cache 管理，batch 动态的情况下，状态才不会变得混乱。这也是官方文档中把它们并列为核心能力的原因。

#### GPU 资源调度与并行
 **Non-Uniform Memory Access** (NUMA) 在中文中一般称为 非一致性内存访问架构，它描述的是一台多路服务器中，CPU访问不同内存区域的速度并不一样。现代服务器通常情况下是多路CPU，多插槽。当多个CPU抢同一块内存，那么总线就会成为瓶颈。为了避免这种情况，硬件会把内存“分片”挂在不同CPU附近。

 于是形成一台服务器中先分成几个 NUMA Node的状态，每个 Node 都有离自己“更近”的CPU,内存，甚至PCle设备。Non-Uniform 本意就是访问本地更快，访问远端内存更慢。

 在vLLM 引擎在设计上支持 **模型并行和数据并行** 等多种并行模式，以扩展模型规模或提升吞吐。在官方的 Quick Start 示例中，可以通过设置 tensor_parallel_size=2 来将模型切分到两块GPU上并行执行。vLLM内部通过 PyTorch Distributed/NCCL 等通信库在设备间同步张量计算结果。如果是多节点部署，vLLM提供了结合 Ray或 kubeRay参考方案。

 如果启动 **Pipeline Parallelism with KubeRay** 那么不同节点各负责模型的不同层分段，假设一个60层(神经网络层)的大模型分布在3个节点中，那么每个节点分到20层。每当收到一个新的请求，会按照层数的顺序从高到低：1～20层先计算，当计算完毕后得到一个 **中间激活值**，通过 Ray 的远程调用来同步这个消息结果。

 整个过程和批处理方式一样，由各个不同阶段的管道组成，Token像水流一般经过其中，最终计算出不同的结果。类同于工厂的流水线，当第一批Token在第二个节点进行计算时，第一个节点也可能正在处理下一批的请求。形成一种重叠执行，以提高整体吞吐量。

 vLLM对GPU资源也使用了 NUMA 拓扑。通常情况下每个Worker进程都会绑定一个GPU以及其所属 CPU NUMA节点，以此来减少跨NUMA 内存访问延迟。在 Kubernetes中使用 Topology Manager + CPU Affinity来保证这个行为。

 Topology Manager 是 kubelet 中的一个拓扑对其组件，其作用主要是协调CPU，设备(例如GPU)，内存等资源分配，尽量让它们在拓扑上对齐，目标就是降低延迟，提升吞吐。

 CPU Affinity 本意是指：将一个任务的可运行CPU集合约束到指定逻辑CPU集合上，也就是可以让一个进程/容器优先或仅允许在某些CPU核上运行。在 Kubernetes 中，这种约束主要通过 **cpuset cgroup controller** 来实现，当kubelet 启用 static policy 时， **属于 Guaranteed QoS 且 CPU requests 为整数核的容器**，可以被分配到节点上的 **exclusive CPUs**。所以这些CPU不再分配给其他的Pod共享使用，这个隔离性是强制的(不意味着是 物理机的绝对独占，这里的语义更倾向是针对其他的Pod)

 如果在双路服务器上部署 vLLM，多GPU进程应被调度到各自本地的CPU核，以充分利用本地内存带宽。这种 **NUMA-aware** 调度在 Kubernetes 中可通过设置 single-numa-node 策略或 CPU Manager 的 static 模式实现，从而提升 GPU-CPU 通信效率。

 除了Ray管理pipeline并行，vLLM还支持KV Cache 跨节点共享，它可以通过一个独立的KV缓存服务(LMCache)实现不同实例之间KV数据的交换或统一存储。

#### 小结
 vLLM 围绕KV Cache管理，请求调度，并行执行这三个方面重新设计了LLM推理的执行方式。

 PagedAttention 解决了传统连续显存分配带来的浪费与碎片问题，在实现的方案中我们依然能看到操作系统管理页表的影子。Continuous Batching 则让请求不再受制于静态批处理的边界，能够在 prefill 与 decode 两种不同负载之间持续调度，从而尽可能提高 GPU 的利用率;而在更大规模场景下，NUMA 感知调度、张量并行、流水线并行以及跨节点 KV 共享，又进一步把这种高效推理能力扩展到多 GPU、甚至多节点环境中。

 vLLM看起来已经不太像一个单纯的“模型推理框架”，更像是一个专门面向LLM推理场景下的运行时。
