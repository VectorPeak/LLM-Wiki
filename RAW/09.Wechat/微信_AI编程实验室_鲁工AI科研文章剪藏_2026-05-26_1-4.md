## 一、再次提醒所有科研人：请把Claude Code当作全方位的研究工具

> 发布日期：2026-02-05
> 原文链接：[再次提醒所有科研人：请把Claude Code当作全方位的研究工具](http://mp.weixin.qq.com/s?__biz=MzE5ODY5MDU4Mw==&mid=2247484657&idx=1&sn=b45dbfd7a949dca6ca459aa9d1b61c40&chksm=976762d46f129e7605c33cdac9a349f5cfe521dbb5edd09e000743c64c1adf78266b716fd298&scene=126&sessionid=1679649075#rd)

大家好，我是鲁工。

   昨天，我们号达成了一个小小的里程碑，在冷启动仅三个多月后，就达成了创号以来的第一篇10w+阅读的成就。
![image](https://mmbiz.qpic.cn/sz_mmbiz_png/29gExhZKtdoYG9Y4WDXm6ocSBiaAZrJWuUCMicvRT6EeQmbia5YGkAK4ibG7Xq2LOummg13icS2zdZx6mjv0upmsbwg/640?wx_fmt=png&from=appmsg)
   在此感谢各位读者对我们号内容的支持，也侧面说明了我去年下半年选的赛道非常正确，AI编程和AI科研都是上限非常高、生产力和创造力想象空间极大的方向。

   也正是因为这篇文章的影响力，让我更加深入思考，Claude Code等Vibe Coding工具在科研领域的作用。换句话说，以Claude Code的能力，每个科研人，都应该把它当成是全方位的研究工具。

   现在是2026年2月，很多人还在把Claude Code当成是一个听起来很厉害、但价格昂贵且国内无法正常使用的观望工具。

   实际上，我每周80%以上的工作都是Claude Code来帮我完成的。用Claude Code来写代码做项目开发、做科研数据分析和模型实验、做PPT汇报Presentation、做AI视频、日常写作等等。

   更何况，理工科的科研，大量工作都是通过代码来完成的。所以，AI编程 + AI科研、Vibe Coding + Vibe Researching，本身就是密不可分的。

   说个上周我个人的例子。

   一位来自医院的主任团队，想要对标注好的CTA脑部血栓病灶提取影像组学特征并训练机器学习模型来预测术后预后，问题主要有两个：

   一是影像数据是直接在放射科拷的，没做序列区分和整理，一个病例下有十几个影像序列，有机器扫描的原始序列，也有后处理软件处理后的序列，他们没有能力自动化提取其中的CTA序列。

   其次就是不懂影像组学，也不知道如何提取，下游的机器学习二分类建模自然也不会做。
![image](https://mmbiz.qpic.cn/sz_mmbiz_png/29gExhZKtdoYG9Y4WDXm6ocSBiaAZrJWuvc0wwBN7n2jAU5KGHP1rdE4ZMDClia5Bqeric57NGeDdyX8xxGSWxutQ/640?wx_fmt=png&from=appmsg)
   这个需求在我们看来，毫无技术难度，但放在前几年，可能确实有一些工作量。但现在借助于Claude Code，我只花了几分钟看了下影像目录结构，稍微抽样看了几例CTA影像和血栓标注，然后用几分钟给Claude Code描述需求，大概半个小时后，所有工作就完成了。
![image](https://mmbiz.qpic.cn/sz_mmbiz_png/29gExhZKtdoYG9Y4WDXm6ocSBiaAZrJWurCFjZGt6kiclB7vnHiaibAySpKeNmouzNU5qsP9QDCrPPibaGMBjicg0Ovg/640?wx_fmt=png&from=appmsg)
   我只需要验收最终的结果，以及适当review一下流程代码即可。虽说影像组学是无用且过时的技术，但在医学影像领域，用它发论文仍然屡试不爽。
![image](https://mmbiz.qpic.cn/sz_mmbiz_png/MHVsz9lwuhW5WWFrp8PIJMlQLthPWtYDxxSQY6ROLSZP58cOU6Ch3XOjnHBhtLficjAoAMlSia5SAa9wC16HWQ7BVuXdvWZSqqgoiacVuT2zgc/640?wx_fmt=png&from=appmsg)
   上面这个活，如果用传统方式，怎么也得搞个一两天，但现在用Claude Code的Vibe Researching方式，时间可以缩短到半小时以内。人工只需要做好项目规划、需求描述、结果验收和适当的代码review即可。

   如果你说你是一个国内院校的非cs相关专业的硕博在读，导师既没有AI sense和课题经费，个人也无能力和财力使用Claude Code，那么你也可以用国内工具和模型做平替。

   比如Claude Code + 各种国产模型API（DeepSeek V3.2、GLM-4.7、MiniMax M2.1）、Kimi Code + Kimi 2.5、Qwen Code + Qwen3 Coder、Qoder CLI等等。

   Vibe Coding和Vibe Researching是大势所趋、是已经在深刻改变当前软件工程和学术科研范式的革命性事务。

## 二、我设计了一支可以自主协作的AI科研团队

> 发布日期：2026-04-28
> 原文链接：[我设计了一支可以自主协作的AI科研团队](http://mp.weixin.qq.com/s?__biz=MzE5ODY5MDU4Mw==&mid=2247485578&idx=1&sn=bc88a168a566c86642924abd45de11cf&chksm=973fddd0e278dabd65216a822c52d1886c703e6a4de2abede4f3da8e4cd381694ea1dada2ad4&scene=126&sessionid=1679649075#rd)

大家好，我是鲁工。

   熟悉我们号的读者都知道，除了AI编程和Vibe Coding，我们号过去在AI科研和Vibe Researching上也有比较多的实践积累。

   我之前常说，      理工科的科研，大量工作都是通过代码来完成的。所以，AI编程 + AI科研、       Vibe Coding      + Vibe Researching，本身就是密不可分的。我们号过去也写了不少受欢迎的AI科研实践与方法论文章：

  [  我用Claude Code写了一篇45页的文献综述，质量可以发一区SCI  ](https://mp.weixin.qq.com/s?__biz=MzE5ODY5MDU4Mw==&mid=2247484397&idx=1&sn=1e950964a0c8b6e42eeaf172e4ed25b9&scene=21#wechat_redirect)

  [  再次提醒所有科研人：请把Claude Code当作全方位的研究工具  ](https://mp.weixin.qq.com/s?__biz=MzE5ODY5MDU4Mw==&mid=2247484657&idx=1&sn=b45dbfd7a949dca6ca459aa9d1b61c40&scene=21#wechat_redirect)

  [  我用OpenClaw + Discord搭建了一个AI科研团队，7x24在线干活  ](https://mp.weixin.qq.com/s?__biz=MzE5ODY5MDU4Mw==&mid=2247484989&idx=1&sn=d2baa4ebf6d93859736ef5e47ce3dd43&scene=21#wechat_redirect)

   前段时间写的那篇OpenClaw + Discord的AI科研团队协作的实践文章，其实我有个体感一直没展开深入来讲。

   那套多Agent方案能完整运行起来，交付也OK，但协作面始终是错位的。几个AI成员在Discord的各自频道里各自发消息，我作为真人是在旁边旁观的状态。并且每个AI成员之间我还不太敢轻易设置相互通信，不然可能随便两个AI之间都会陷入相互触发的死循环。

   这一次的Moxt尝试，让我对AI团队协作，有了新的体验。

   Moxt团队，他们在做的是一款Agent-native的Workspace，简单来说就是「真人和AI同事坐在同一个工作区里一起干活」。
![image](https://mmbiz.qpic.cn/sz_mmbiz_png/MHVsz9lwuhXczGFkVPsxs5oiahvLs2Qs1J4y6HOibxv5CtpBHSSCefuyAdXsIhbLH1v5t7srDnriagUXJDBJL8aJoOM6QlG6asRko4Szib9q6rs/640?wx_fmt=png&from=appmsg)
   我拿到账号后的第一个念头就是，把之前写那篇AI写SCI综述的方法论搬过来，换个方向试试。医学影像综述写作那套依赖太多闭源期刊文献，这次我索性全走开源，以arxiv和NeurIPS/ICML/ICLR的OpenReview为主，做一篇Harness Engineering方向的研究综述看看。

   [Moxt](https://moxt.ai/)

   聊几个让我上手就觉得和过往工具不一样的点。

   第一个是AI同事不再是一个简单的「聊天对象」。

   在Discord那套流程里，AI Bot是个会说话的头像。Discord只是个Channel，文件修改是建立在部署OpenClaw的主机上的Agent workspace中，任务执行也需要依赖外部的Claude Code等生产力工具。

   Moxt里AI同事是直接坐在workspace里的。它有名字、有头像、有职责，更关键的是它就在我写东西的那份文档里，并且所有干活能力都是原生于Moxt系统中。

   根据综述论文写作的特点，我给这套AI科研团队成员分别定义为：研究主管、文献侦察员、论文分析师、论文撰写员、质量检查与终稿交付员。
![image](https://mmbiz.qpic.cn/mmbiz_jpg/MHVsz9lwuhWtQDibqxYTGK6xiaKlIJEfjDokKsPWqfYic2MwzSibIzK2AbsISPS40AGWQG8aHA1uvVyJHRTichM6yPQDuIeW8K9oUibrjvh2lrx3Q/640?wx_fmt=jpeg)
   在Moxt中新建AI同事也非常简单，直接在AI Teammates下创建AI同事即可：
![image](https://mmbiz.qpic.cn/mmbiz_png/MHVsz9lwuhUuLfBjoucA1S2JhxUWbpAY51DGw7NArrwRic32x5e3063YTf6cxlibdI8KWBzx780tSWvxJuic0r5eS4TSBnIDvib3R5MRzej42UY/640?wx_fmt=png&from=appmsg)
   第二个是@文档这个动作。

   当然这个也不是新做法了，之前在Cursor等AI IDE中，也是通过@的方式为当前会话指定上下文。之前我在OpenClaw或者Discord中，是直接把文件路径告诉AI，让它指定读取。

   Moxt里也可以直接在对话里@一份文档，AI接着就能读懂上下文干活。说到这我插一句。

   @文档才是科研协作最原生的动作。做综述本质上就是围绕一堆文献在一份文档上反复推敲：@人、@文献、@上一版结构。这件事以前在Notion里是在人和人之间发生，现在是人和AI、AI和AI都能发生。

   回到实测。

   我这次跑的方向是Agent的Harness Engineering综述，因为今年这个方向非常火，我们顺便用Moxt做个全面、前沿的paper review。

   为了可以快速搭建写作工作流和协作框架，我直接把之前我开源在GitHub上的medical-research-review skill丢给Moxt，然后先让它拆解和迁移适配到AI前沿和顶会领域的综述写作。然后再按上面创建的AI同事角色进行分配，最后梳理出角色分工和协作流水线：
![image](https://mmbiz.qpic.cn/mmbiz_png/MHVsz9lwuhUVdLDE21bRwcZwMricwicQdDcHIDcGZhjZ1rBbMDWiaKKqbYKJBIibbusSVWfCHChzpDBZpibN7gzEjsQxSLZQhSrGSPfPFn67t1ic0/640?wx_fmt=png&from=appmsg)
   大概流程就是，由我直接跟研究主管对话，分配综述写作任务，然后由研究主管根据协作流水线，分别指派文献侦察员、论文分析师、论文撰写员和质检员协作完成写作任务。

   然后Moxt会在Team Space里新建一个harness-engineering-ai-agents目录，五个AI同事各自分工和协作产生的文档就会放在这个目录下：
![image](https://mmbiz.qpic.cn/sz_mmbiz_png/MHVsz9lwuhUOuEoA9dmzQUgicevkdeIVZ0GXPhnqsPBEgGicvI28J1S6Murs85YT1agOJBQvrVbLBSrG3hDvy9D8JLfjbyOK7iad5MM3VKyAAY/640?wx_fmt=png&from=appmsg)
   回到我们刚刚的任务，我直接跟研究主管指派Harness Engineering的前沿研究进展写作综述：

   然后研究主管会输出一份项目启动报告，把项目定位与目标、研究方向分类体系、项目初始输出文件、五阶段工作流和项目启动状态交待清楚。

   比如，文献侦察员对全部开源文献和顶会采集后输出了一份文献矩阵：
![image](https://mmbiz.qpic.cn/mmbiz_png/MHVsz9lwuhWPlk5mI6ZnISDzSUTyZNTTTbMTCN3e23nChhZCq05cSlBRiaNMP1B7ficRgWrESicibzPzqQMQ8pxp38SYFH8mVW9LSoopq8YQvIM/640?wx_fmt=png&from=appmsg)
   文献侦察员总共侦查到200多篇文献，其中arxiv论文约150篇、各顶会论文40余篇已经工业界博客文章10余篇，包括OpenAI和Anthropic等顶级团队的工业实践。

   然后是论文分析师基于文献矩阵出具论文深度分析报告。这个过程是最耗时的阶段。大概一个多小时后，我收到了论文分析师输出的一份基于200+文献的深度分析报告，直接给我干了50多页的内容：

   实测过程中，我观察到两个细节的点值得记录一下。

   一个是Session可观测性。每个AI同事干活时的完整轨迹都能回看。它调了哪个工具、中间想了什么、最后为什么给出了这个输出，一清二楚。

   这个能力对做研究的人太重要了。综述里随手一个数据、一句论断，我都得能查到它是从哪来的。学术可追溯性的底线，就建立在这种能回查上。

   另一个是Cron定时任务。我让论文侦察员每天早上8点自动扫一遍arxiv新论文，把候选清单推送到Teamspace里@我和科研主管一下。这个现在算是Agent的标配功能了，但在Moxt中用起来体验会更加丝滑。

   其实这套AI团队协作协作的配置，可以放大适用于任何写作类的任务，比如自媒体的微信公众号协作，我只要把对应角色改一下，协作流程基本上大差不差。比如我加一个选题流程和一个去AI味的流程：

   聊一下这次体验和OpenClaw + Discord那套的差异。

   虽然这两套方案我都还在用，场景真的不一样。

   OpenClaw + Discord更偏向「让多个AI Bot跑在一个公共聊天室里，我作为用户时不时丢任务进去」。适合轻量的定时监控、日报汇总、运营类公共任务。

   Moxt更偏向「我和AI同事一起坐在工作区里做一件有交付物的事」。综述写作、一人公司的核心业务流、产品文档协作这种场景，Moxt的工作面更顺滑。

   OpenClaw + Discord通常有较高部署和编排门槛，我之前自己摸索都花了不少时间，但Moxt打开网页就能用，几乎没有上手门槛。

   也想顺便说一下目前感受到的人和AI协作的边界。

   Moxt现阶段的AI同事之间的自主协作与编排还在继续优化打磨中，预计很快能上线。我现在的实际用法，是我作为中间调度人，按照综述写作的流水线顺序，把任务在几个AI同事之间起承转合。

   但这恰好和Moxt团队反复强调的一个说法对上了：AI是放大人的能力的工具，替代从来不是目标。作为真人，我始终在那条协作链条里。我的判断、我的审美、我的科研直觉，决定这篇综述最后写成什么样子。AI同事在每一段都放大了我的手脚，但方向盘一直在我手里。

   这一点恰好是我最舒服的地方。

   做研究的人最怕什么？其实不是没有AI能用。真正怕的是AI把活全干完之后，你不知道自己在整件事里还剩下什么。Moxt这个workspace给我的感觉是，我还在方向盘上，AI同事坐在副驾和后排，随时能帮我切换车道。

   等同此理，大模型的输入信息质量非常重要。只有提供了高质量的上下文（Context），AI将其编排和加工成我们期望的输出物。所以，对应到用Claude Code写文献综述这项任务上，最重要的工作就是给AI提供高质量的文献。

### 为什么是Discord
   搭多Agent系统，第一个要解决的问题是：这些Agent跑在哪里？

   最后选了Discord，原因很直接：Discord天然支持多频道分区，每个Bot有独立身份和头像，@mention机制原生就有，群聊里谁说了什么一目了然。说白了，Discord就是一个现成的多Agent协作操场，你不用自己造轮子。

### 5个科研Agent怎么分工
   通用的多Agent系统，角色设计一般是总指挥、调研员、内容创作、工程师、审核员这种架构。但科研场景有本质区别。

   科研是Pipeline驱动的：文献调研 → 数据准备 → 算法开发 → 论文写作，每个环节的输出就是下一个环节的输入。这和内容运营那种并行发散的模式完全不同。

   所以我设计了5个科研专属角色：

 **科研总指挥** （director），相当于课题组的PI。它负责接收研究需求、拆解任务、追踪进度、管Deadline。在Discord群聊里，它是唯一一个全局监听所有消息的角色，其他4个Agent都要被@才会响应。

 **文献调研员** （literature），每天早上8点自动扫arXiv新论文，按关键词检索，把结果写入共享知识库。比如你可以设定关键词是large language model和     medical image segmentation等。

 **数据工程师** （data），负责评估公开数据集、设计预处理方案、检查数据质量。科研中数据问题往往是最容易被忽视但影响最大的环节，单独拎出来一个角色盯着很有必要。

 **算法研发员** （algorithm），核心技术执行者。模型设计、实验执行、消融分析都归它管。我在它的SOUL.md里特别强调了：实验必须记录随机种子、不cherry-pick结果、至少3次运行取平均值。

 **论文撰写员** （writer），读取文献调研员和算法研发员的产出文件，起草论文各章节。它熟悉MICCAI、CVPR、TMI、Nature等会议和期刊的写作风格。
![image](https://mmbiz.qpic.cn/sz_mmbiz_png/MHVsz9lwuhX0vSJuFTv9Z4QjftlYsd0jibLPQibxgjicBmawnfqz1SDFGq56QzLPXjYMNRLmfugHnmibcLPAJoghHaVhI5vv1HbZNkCteKpXFf4/640?wx_fmt=png&from=appmsg)
   这5个角色之间的数据流向很清晰：文献调研员的输出喂给算法研发员（提供研究Gap和对比基线），同时也喂给论文撰写员（提供Related Work素材）。数据工程师准备好的数据集交给算法研发员跑实验。算法研发员的实验结果交给论文撰写员写Experiments部分。科研总指挥全程协调收口。

### 架构的三个关键设计
   角色定好了，怎么让它们在Discord里真正协作起来？这里有三个关键的架构设计。

 **Bindings路由**。   每个Discord Bot对应一个Agent，通过openclaw.json里的bindings配置做硬绑定。消息进来，Gateway先看是哪个Bot收到的，直接路由到对应Agent。这是确定性路由，不依赖模型判断。

   X上的实测经验很有说服力：如果不在入口层做路由，让所有Agent自己判断该不该处理，后面的协作全是乱的，有时候两个Agent抢答，有时候都不答。

   配置很简单，核心就是channel + accountId → agentId的映射：

```text
{  "bindings": [    { "agentId": "director",   "match": { "channel": "discord", "accountId": "Bot的Discord ID" } },    { "agentId": "literature", "match": { "channel": "discord", "accountId": "Bot的Discord ID" } },    // data, algorithm, writer 同理  ]}
```
 **会话隔离**。   dmScope设成per-account-channel-peer，按账号+渠道+对端用户三维隔离。同一个人从Discord找同一个角色，上下文不串。不同用户找同一个角色，完全隔离。如果课题组有多人使用，这个配置能确保不同同学的实验数据和论文讨论绝不会串在一起。

 **双轨治理**。   这是我觉得整套系统里最精妙的设计。配置轨做硬约束（bindings路由、会话隔离、Agent互ping限制），规则轨做软引导（SOUL.md人格定义、TEAM-RULEBOOK协作规则、AGENTS.md运行手册、TOOLS.md定义每个Agent可以使用的外部工具）。两条轨道叠加，不把一切压在模型的自觉上。

   举个具体例子：我把Agent之间的ping-pong turns设成了0，物理上禁止Agent互相闲聊。同时在TEAM-RULEBOOK里写明：群聊中只有科研总指挥可以主动发言，其他角色等待被@，不要和其他Agent客套或做无意义确认。配置层管住了能不能聊，规则层则是管住了该不该聊。

   科研场景为什么特别需要双轨治理？因为学术严谨性要求极高。如果不在SOUL.md里反复强调不编造论文不杜撰数据，模型在长对话后是真的可能开始创造不存在的引用。配置层控制不了内容质量，只能靠规则层加上渐进式记忆反馈来纠正。

### 记忆系统和自动化
   多Agent系统跑起来之后，记忆管理是最容易被忽视但最影响长期效果的部分。

   参考X上的相关设计，我采用的是五层记忆架构。最底层是每日流水日志（memory/2026-03-10.md），记录当天的实验细节和文献笔记。

   往上是长期记忆（MEMORY.md），只存经过验证的稳定信息，比如Synapse数据集上lr=1e-4效果最好、用户偏好PyTorch不用Keras。

   再往上是群聊记忆（GROUP_MEMORY.md），存团队达成的研究共识，比如已确认的模型架构方案。这里有个隐私红线：绝不混入私聊内容。

   然后是冷归档（超过30天的daily memory自动归档），最顶层是语义检索（先memory_search召回，再memory_get精确读取）。

   Saboo有句话说得好：Mental notes don't survive session restarts. Files do. Agent随时间变好，不是因为模型在进步，是因为上下文和记忆越来越丰富。他的经验是第一版Agent表现平庸是正常的，到第30次迭代才会真正好用。

   自动化调度也是科研场景的刚需。我用OpenClaw的Cron功能配了几个定时任务：文献调研员每天早上8点扫arXiv，算法研发员每天上午10点检查实验状态，科研总指挥每周五下午5点出项目周报。调度顺序是故意设计的，文献扫描最先跑，因为其他角色可能需要最新文献信息。

   论文Deadline前2到4周，我会临时增加调度频率，进入冲刺模式：论文撰写员每天下午2点更新写作进度，算法研发员每天下午4点检查补充实验。投稿完成后再把这些临时任务暂停掉。

   配合HEARTBEAT.md的自愈机制，如果某个Cron任务滞后超过26小时，系统会自动force re-run。科研节奏不像内容运营那样每天要产出，但一旦进入Deadline冲刺，任何一个环节卡住都会影响全局。

### 真正的生产力在于外部能力
   系统搭好了，5个Agent跑起来了，但说实话，光靠OpenClaw和所使用的模型本身对话能力，其实几乎没啥科研生产力。

   这套系统要想真正转化为科研生产力，关键在于每个Agent能调用的外部能力。

   比如文献调研员，如果只靠模型自身的知识，它能给你的信息截止到训练数据的时间点，最新的arXiv论文它根本不知道。但是接上Semantic Scholar的MCP、配上联网搜索Skills、使用Zotero-MCP访问本地Zotero文献库，它就真的能每天扫描最新论文了。

   再比如算法研发员，如果它只能在对话里建议你调学习率、改patch size，那和ChatGPT没太大区别。但如果它能通过Claude Code或者Codex这样的Coding工具直接读你的训练脚本、分析实验日志、甚至帮你改代码提交PR，那生产力就完全不是一个量级了。

   数据工程师也是同理。接上数据库查询的MCP、文件系统读写的Skills，它才能真正检查数据管道的状态，而不只是在chatbox里面一顿瞎聊。

   所以我的建议是：搭完基础的多Agent框架之后，第一时间把精力放在给每个Agent配备合适的MCP和Skills上。框架是骨架，外部能力才是肌肉。没有肌肉的骨架，站得起来但干不了活。

   这个其实我也还在配置，因为我的OpenClaw是部署在远程vps上的，像Claude Code和Codex这类对使用环境敏感的强生产力工具，我还没敢在上面安装，所以，我的这套系统其实我自己也暂时还没发挥其科研生产力。

### 搭建不难，调教才是功夫
   借助AI编程工具，搭建这套系统的初始框架其实很快。创建5个Discord Bot、写好openclaw.json配置、给每个Agent写一版SOUL.md，一个下午基本能跑通。OpenClaw的文档和社区教程都很齐全，照着配就行。

   但能跑和好用是两回事。

   我搭完第一版之后，科研总指挥的任务拆解太粗，文献调研员的检索范围不够精准，论文撰写员的风格和我常投的会议不匹配。这些问题只有在实际使用中才会暴露出来。

   Saboo管这个过程叫corrective prompt-engineering：观察Agent表现，给反馈，改SOUL.md，再观察，再调。他的经验是第一版Agent表现平庸很正常，到第30次迭代才会真正好用。

   所以建议是：搭建的时候大胆一点，5个Agent一起上没问题。但上线之后要有耐心，每天用、每天观察、持续调SOUL.md和MEMORY.md。框架搭得再漂亮，不经过实际使用的打磨也只是个空架子。

   这套系统我反复折腾配置了几个晚上，到昨晚才搭建成功，各类外部工具配置都还没配齐，上下文都还有限，所以本身也需要逐渐运行和积累上下文来打磨其性能。

### 安全和成本
   安全方面必须提一句。OpenClaw目前已经有多个高危漏洞被公开披露，根据公开报道，ClawHub上被发现的恶意Skills数量已经超过800个。

   建议在隔离环境中运行Gateway（专用VM或独立设备），不要暴露到公网，第三方Skills安装前务必审查源码（安装  skill-vetter进行技能安全审查）。科研场景还有额外的考量，未发表的实验数据和论文草稿属于敏感信息，要确保Agent不会通过外部API泄漏。

   成本方面，Discord和OpenClaw本身都免费。主要开销是AI模型的API费用。我个人的策略是不同科研Agent角色用不同模型：科研总指挥和论文撰写员用高端模型（GPT-5.4），文献调研员和数据工程师用国内性价比模型（MiniMax M2.5），算法研发员用Claude Opus 4.6，日常对话可以切     gemini-3.1-flash-lite。

   全部配齐的话，大概每个月成本300-400美元左右（其实Claude Code的Max订阅是大头）。

   用OpenClaw + Discord搭科研多Agent系统，核心就是这些。框架不复杂，5个角色、Bindings路由、会话隔离、双轨治理、五层记忆、Cron调度，每一层都有明确的设计意图。

   但从能跑到跑得好，中间的时间全在外部工具配置和日常Agent的交互中积累上下文。Agent帮你省的是重复劳动的时间，思考和判断这件事，还得是人来。
