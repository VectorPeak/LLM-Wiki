## 0x01. 大模型：最小化首字延迟与最小化token间延迟
> 发布日期：2026-05-27  
> 原文链接：[大模型：最小化首字延迟与最小化token间延迟](https://mp.weixin.qq.com/s/FLLrJCxVuyuM1A3ju4BZMQ)

在大语言模型(LLM)的在线服务场景中，首字延迟(Time-to-First-Token, TTFT)和token 间延迟(Time-per-Output-Token, TPOT)是衡量用户体验的两个核心指标。二者反映推理流程中不同阶段的性能瓶颈，优化目标常存在冲突，需根据应用场景权衡取舍。

### 1. 概念定义与技术来源
首字延迟(TTFT)指从用户提交请求到收到第一个生成 token 所经历的时间。该延迟主要由Prefill 阶段决定：系统需对完整 prompt 进行一次性前向计算，初始化 KV Cache，并完成首个输出 token 的预测。TTFT 受 prompt 长度、模型规模、硬件算力及显存带宽共同影响，尤其在长上下文场景下显著增加。

token 间延迟(TPOT)指连续两个输出 token 之间的平均时间间隔。该延迟由Decode 阶段主导：每步仅处理一个新 token，依赖已缓存的 KV Cache 计算注意力，属于典型的串行、访存密集型过程。TPOT 主要受 KV Cache 大小、采样策略、Unembedding 层计算及 GPU 调度效率影响。

在问答、搜索等短响应场景中，用户更关注“是否立刻有回应”，因此最小化 TTFT 是首要目标。此时可接受稍高的 TPOT，甚至牺牲部分吞吐量。典型策略包括：优先调度短 prompt 请求、限制最大输入长度、使用低精度 Prefill 加速。

在内容创作、代码生成等长输出场景中，用户更在意“整体生成速度流畅”，因此最小化 TPOT 和提升 tokens/sec 更关键。此时可容忍稍高的首字等待时间。优化重点在于提升 decode 并发能力，如动态批处理(Continuous Batching)、KV Cache 压缩、专家混合(MoE)模型的高效路由。

## 0x02. 大模型：Embedding Lookup
> 发布日期：2026-05-27  
> 原文链接：[大模型：Embedding Lookup](https://mp.weixin.qq.com/s/p_YgFEdooSGHQh6XG3JsBA)

在大语言模型(LLM)的输入处理流程中，Embedding Lookup(嵌入查找)是将离散的 token ID 转换为连续稠密向量表示的第一步操作。该过程虽计算简单，却是连接符号世界与神经网络数值空间的关键桥梁，直接影响模型的语义表达能力和训练效率。

### 1. 基本原理与实现方式
Embedding Lookup 的核心是一个可学习的参数矩阵，称为Embedding Table或Token Embedding Layer，其形状为，其中是词表大小(如 128,000)，是模型隐藏维度(如 4096)。给定一个 token ID 序列，Embedding Lookup 通过索引操作从该矩阵中提取对应的行向量，形成形状为的嵌入序列。

这一操作在底层通常由 GPU 的 gather 指令或 cuBLAS 的 indexing kernel 实现，本质上是一次并行化的内存读取，不涉及浮点运算。由于每个 token ID 独立映射，该过程天然支持批处理和序列并行。

#### 1.2 在模型架构中的位置与作用
在标准 Transformer 中，Embedding Lookup 位于网络最前端。输入文本经 tokenizer 分词后得到整数 ID，立即通过 lookup 转为向量。随后，这些向量通常与位置编码(如 RoPE 或绝对位置嵌入)相加，作为第一层注意力模块的输入。

值得注意的是，在多数现代 LLM(如 LLaMA、Qwen)中，输出层的 Unembedding 权重与输入 Embedding 权重共享(weight tying)，即。这种设计不仅减少约的参数量，还在理论上促使输入与输出语义空间对齐，提升训练稳定性。

#### 1.3 性能特性与工程挑战
尽管 Embedding Lookup 本身无计算开销，但在大模型场景下面临显著工程挑战：

首先，显存占用高。Embedding Table 的大小为。以 LLaMA-7B 为例(V=32K, d=4096, FP16)，仅此一层就占用约 512MB；而 LLaMA2-70B(V=32K, d=8192)则达 1GB。对于超大词表模型(如 128K tokens)，该层可占总参数的 20% 以上。

其次，访存模式敏感。Lookup 是典型的随机内存访问：token ID 序列决定访问哪些行，若 ID 分布不均(如高频词集中)，可能导致缓存局部性差，降低带宽利用率。在 GPU 上，非对齐或分散的 gather 操作可能成为瓶颈，尤其在小 batch 场景下。

此外，在长序列推理中，虽然 lookup 仅执行一次，但其输出需与位置编码融合并送入 Prefill 阶段，因此仍影响首字延迟(TTFT)。而在微调或增量预训练中，若扩展词表，需对新 token 的 embedding 向量进行合理初始化(如均值采样或基于语义相似词复制)，否则易导致训练不稳定。

#### 1.4 优化与变体技术
为应对上述挑战，工业界采用多种优化策略：

1. 量化：将 Embedding Table 从 FP16 量化至 INT8 或 INT4，显著降低显存与带宽需求，推理时再反量化。由于 lookup 无计算误差累积，量化对精度影响较小。
2. 分片存储：在模型并行(如 tensor parallelism)中，Embedding Table 可按行或列切分到不同设备，lookup 后通过 all-gather 合并结果。
3. Hash Embedding / Adaptive Embedding：对超大词表，将低频词映射到共享嵌入槽，减少参数总量，常用于推荐系统，但在通用 LLM 中较少使用。
4. 缓存高频词：在 CPU offload 场景中，将 top-K 高频 token 的 embedding 缓存在 GPU 显存，其余从主机内存加载，平衡速度与容量。
Embedding Lookup 是大模型中最基础却不可忽视的操作。它看似只是一个“查表”，实则承载了词汇语义的初始编码，并深刻影响模型规模、内存布局与推理效率。

## 0x03. 大模型：Residual Stream
> 发布日期：2026-05-27  
> 原文链接：[大模型：Residual Stream](https://mp.weixin.qq.com/s/4voKxSLUA6Hsqyuq_BRi4A)

在 Transformer 架构中，Residual Stream(残差流)是指贯穿整个网络、承载输入 token 表示并逐步被各子层修改的隐藏状态序列。

### 1. Residual Stream 的结构与作用
Transformer 每一层由两个主要子模块组成：多头自注意力机制(Multi-Head Self-Attention, MHSA)和前馈神经网络(Feed-Forward Network, FFN)。这两个模块均以残差连接(Residual Connection)方式接入主路径。具体而言，对于第 l 层的输入，计算流程为：

初始时，Residual Stream 由 token embedding 与位置编码相加构成。随着层数加深，每一层的注意力机制和 FFN 都向该流“写入”新的信息，而原始信息始终通过残差连接保留下来。这种设计使得模型既能学习复杂的上下文变换，又避免梯度消失，支持极深网络训练。

#### 1.2 为何 Residual Stream 至关重要
首先，它提供了信息的恒等通路。即使某一层的子模块学习效果不佳，原始表示仍可通过残差连接传递下去，保障模型下限性能。这在深层模型中尤为重要，是 Transformer 能扩展至数十甚至上百层的基础。

其次，Residual Stream 是语义逐步精炼的载体。研究表明，浅层主要捕获词法和局部句法特征，中层处理语义角色和依存关系，深层则聚焦任务相关或抽象概念。所有这些变化都体现在 Residual Stream 的演化过程中。

第三，它支持模块化干预与可解释性分析。例如，在模型编辑(Model Editing)或归因(Attribution)研究中，可通过向 Residual Stream 注入特定向量来操控输出(如强制模型认为“巴黎是意大利首都”)，从而验证内部知识的存储位置。类似地，使用探针(probing)方法可在不同层的 Residual Stream 上训练分类器，分析语义信息的出现时机。

#### 1.3 与注意力和 FFN 的交互关系
注意力机制的作用是从 Residual Stream 中“读取”其他 token 的信息，并将聚合结果“写回”当前 token 的流中。例如，当处理“它”时，注意力可能从“猫”处读取信息，并将“猫”的语义注入“它”的表示。

FFN 则对每个 token 的 Residual Stream 独立进行非线性变换，常被视为参数化的知识库。有研究指出，FFN 中的某些神经元专门响应特定事实(如“Eiffel Tower → Paris”)，其输出直接叠加到 Residual Stream 上，从而影响最终预测。

值得注意的是，由于残差连接的存在，注意力和 FFN 的输出并非完全替代原表示，而是作为“增量更新”加入。这使得模型行为更具鲁棒性，也解释了为何删除单个头或神经元通常不会导致灾难性失败。

#### 1.4 工程与研究意义
在模型压缩中，理解 Residual Stream 的演化有助于识别冗余层。例如，若某两层的流几乎不变，可考虑合并或剪枝。

在推理优化中，部分系统尝试缓存中间层的 Residual Stream 用于 speculative decoding 或 early exiting，以加速生成。

在安全对齐领域，通过监控或约束 Residual Stream 的变化，可检测有害内容的生成路径，为干预提供依据。

总结来看，Residual Stream 不仅是 Transformer 的技术实现细节，更是理解大模型“如何思考”的核心视角。它将复杂的深度网络简化为一条信息流，每一层都在其上做微小但关键的修改，最终汇聚成连贯、智能的输出。

## 0x05. 大模型：旋转位置编码RoPE

在大语言模型(LLM)中，旋转位置编码(Rotary Position Embedding, RoPE)是一种广泛采用的相对位置编码方法，用于向 Transformer 的注意力机制注入 token 之间的位置信息。相比早期的绝对位置编码(如原始 Transformer 中的正弦/余弦编码或可学习嵌入)，RoPE 通过将位置信息以旋转变换的方式融入 Query 和 Key 向量，在保持高效计算的同时，显式建模了 token 间的相对距离，显著提升了长序列建模能力。

## 0x06. 大模型：绝对位置编码
> 发布日期：2026-05-27  
> 原文链接：[大模型：绝对位置编码](https://mp.weixin.qq.com/s/izJxxeNV3PL5rZVFNEh1ww)

在 Transformer 架构中，由于自注意力机制本身对输入序列的顺序不敏感，必须显式引入位置信息，使模型能够区分“词在句中的位置”。绝对位置编码(Absolute Positional Encoding)是最基础且广泛使用的位置表示方法，其核心思想是为每个 token 分配一个与位置索引直接相关的固定或可学习向量。

### 1. 基本原理与形式
绝对位置编码通过将位置信息注入到 token embedding 中，使模型在后续计算中能感知序列顺序。典型实现方式有两种：

1. 可学习的位置嵌入(Learned Position Embeddings)
2. 固定函数式编码(如正弦位置编码)
#### 1.2 在大模型中的应用现状
尽管绝对位置编码概念简单，但在现代大语言模型中面临显著挑战：

1. 上下文长度扩展问题
2. 外推能力差
3. 与 RoPE 等相对位置编码的对比
#### 1.3 改进与变体
为缓解绝对位置编码的局限，研究者提出多种增强策略：

1. 插值法(Position Interpolation)
2. NTK-aware Scaled RoPE
3. ALiBi(Attention with Linear Biases)
绝对位置编码是 Transformer 早期解决序列顺序问题的直接方案，实现简单且在短文本任务中表现良好。然而，在大模型追求超长上下文(如 32K、128K tokens)的背景下，其缺乏外推能力和固定位置绑定的缺陷日益凸显。当前工业界已普遍转向 RoPE 等相对位置编码方案。

在大语言模型(LLM)的自回归推理过程中，Key(K) 是实现高效生成的核心机制之一。由于 Transformer 的自注意力机制要求每个新 token 与所有历史 token 计算相关性，若每次 decode 步骤都重新计算全部历史 K 和 V，将导致严重的计算冗余。

因此，系统需在推理过程中对 K 和 V 进行持久化缓存(通常称为 KV Cache)，以避免重复计算。

### 1. KV 缓存的基本原理
在 Prefill 阶段，模型对用户输入的完整 prompt 执行一次前向传播，计算出每一层中每个 token 对应的 Key 和 Value 向量。这些向量被显式存储在内存中，形成初始的 KV Cache。进入 Decode 阶段后，每生成一个新 token，仅需：
1. 计算该 token 自身的 K 和 V；
2. 将其追加到对应层的 KV Cache 末尾；
3. 在后续步骤中，直接读取完整的 KV Cache(包含 prompt 和已生成 token)用于注意力计算。

通过这种方式，注意力机制中的 QK^T 点积操作无需重新处理整个历史序列，只需与缓存中的 K 矩阵相乘，大幅降低计算复杂度。

#### 1.2 KV Cache 的数据结构与内存布局
KV Cache 的逻辑结构为四维张量：

- batch_size(B)：并发请求数；
- num_heads(H)：注意力头数；
- sequence_length(S)：当前上下文长度(动态增长)；
- head_dim(D)：每个头的维度。
实际存储时，为提升访存效率，通常采用连续内存布局，如 (B, H, S, D) 或 (B, S, H, D)，并确保对齐以适配 GPU 的内存访问模式。

为此，现代推理引擎(如 vLLM)引入PagedAttention机制，将 KV Cache 划分为固定大小的“页”(pages)，类似操作系统虚拟内存。物理页可非连续，但逻辑上拼接为完整序列。这显著提升了显存利用率，并支持高效的内存共享(如树状推测解码中的分支复用)。

#### 1.5 挑战与前沿方向
尽管 KV Cache 极大提升了推理效率，但仍面临若干挑战：

- 长上下文外推：当 S 远超训练长度，位置编码(如 RoPE)可能失真，影响注意力准确性；
- 内存墙瓶颈：decode 阶段大量时间花在读取 KV Cache，成为典型的 memory-bound 操作；
- 动态长度调度：如何在多请求混合场景下高效管理不同长度的 cache，是推理引擎调度的核心难题。
未来方向包括：

- 压缩缓存：利用低秩近似或稀疏表示压缩 K/V；
- 近存计算架构：将 KV Cache 存储于高带宽近存单元，减少数据搬运；
- 选择性缓存：基于重要性评分仅缓存关键 token 的 K/V，牺牲部分精度换取效率。

## 0x08. 大模型：张量拼接Concat
> 发布日期：2026-05-27  
> 原文链接：[大模型：张量拼接Concat](https://mp.weixin.qq.com/s/3j-xhyx1r46oS30cFT6PWQ)

在大语言模型(LLM)的架构实现和推理流程中，张量拼接(Concatenation，简称 Concat)是一种基础但关键的操作，用于将多个张量沿指定维度合并为一个更大的张量。

### 1. 基本定义与语义
张量拼接是指在保持其他维度不变的前提下，将两个或多个形状兼容的张量沿某一轴(dimension)连接。例如，给定两个张量和，沿序列长度维度(dim=1)拼接后得到。

该操作不引入可学习参数，也不改变元素值，仅调整内存布局。因此，Concat 本身无计算开销，但涉及大量数据移动，在大模型中可能成为性能瓶颈。

#### 1.2 典型应用场景
1.多头注意力输出融合在 Transformer 的多头自注意力机制中，每个注意力头独立计算输出向量，形状为，其中为单个头。所有头的输出首先沿头维度拼接为，再通过线性投影()映射回原始隐藏维度。此处的 Concat 是实现“多视角信息聚合”的必要步骤。
2.KV Cache 动态扩展在自回归推理的 decode 阶段，每生成一个新 token，其对应的 Key 和 Value 向量需追加到历史 KV Cache 末尾。这一过程本质上是对 cache 张量沿序列长度维度进行 Concat。例如，原有 cache 为，新增 token 的 K 为，拼接后变为。
3.模型并行中的通信结果整合在张量并行(Tensor Parallelism)中，如 Megatron-LM 的列切分策略，各 GPU 分别计算部分 FFN 或注意力输出，最终需通过 All-Gather 通信收集完整结果，再沿隐藏维度拼接，恢复原始张量形状。
4.上下文拼接(Prompt + Generated Tokens)在某些推理框架中，为简化 attention mask 逻辑，会将 prompt 和已生成 token 的 embedding 拼接成完整上下文序列，再送入模型。虽然现代系统多用 KV Cache 避免重复计算，但概念上仍依赖 Concat 思想。

#### 1.3 性能挑战与优化
尽管 Concat 无浮点运算，但在大模型中面临显著工程挑战：

- 内存分配开销：每次拼接通常需分配新内存并拷贝所有源数据。在 decode 阶段，若每步都执行 naive Concat，KV Cache 扩展会导致 O(n²) 内存拷贝总量，严重拖慢生成速度。
- 内存碎片：频繁的小块拼接易造成显存碎片，降低后续大张量分配的成功率，甚至触发 OOM。
- 带宽压力：拼接操作需读取全部输入张量并写入新张量，对显存带宽敏感，尤其在长序列下。
为应对上述问题，工业界采用多种优化策略：
1.预分配缓存池(Pre-allocated Cache)推理引擎(如 vLLM)预先分配足够大的连续显存块作为 KV Cache 存储池，新 token 的 K/V 直接写入预留位置，避免运行时拼接。这本质上是用空间换时间，消除动态 Concat 开销。
2.PagedAttention 机制受操作系统虚拟内存启发，vLLM 将 KV Cache 划分为固定大小的物理页，逻辑上连续的序列可由非连续物理页组成。追加新 token 仅需分配新页并更新页表，无需数据移动，彻底规避传统 Concat。

3. Kernel 融合在多头注意力中，部分实现将 head 拼接与后续投影融合为单一 GEMM 操作，避免显式拼接中间结果，减少内存写回。
4. 就地拼接(In-place Concat)当源张量生命周期结束且目标内存可复用时，部分框架尝试就地扩展，但受限于内存布局对齐要求，通用性较低。
张量拼接虽是基础操作，但在大模型的高效实现中扮演着“隐形瓶颈”角色。尤其在自回归生成的 KV Cache 管理中，朴素 Concat 会导致严重性能退化。现代高性能推理系统的核心创新之一，正是通过预分配、分页或融合等技术，将显式的 Concat 转化为隐式的指针操作或内存写入，从而在保证功能正确性的同时，极大提升吞吐与延迟表现。理解 Concat 的原理与优化路径，是掌握 LLM 推理系统设计的关键一环。
