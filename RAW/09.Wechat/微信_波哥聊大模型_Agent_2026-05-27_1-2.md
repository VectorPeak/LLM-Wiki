### 写在前面
 2026 年的 AI 工程师面试已经发生了质变。面试官不再满足于"请解释 ReAct 是什么"这类概念复述，而是直接追问"ReAct 循环中幻觉导致工具被反复错误调用，你怎么从架构层面阻断？"

 本文从一份涵盖 150+ 道真实面试题的题库中，按主题筛选出最高频、区分度最大的问题，并给出每道题的考察意图与回答框架。无论你是准备面试的候选人，还是出题的面试官，都可以作为参考。

### 1. Agent 架构与核心组件
 这是几乎所有 Agent 岗位面试的开场区域，从基础认知到系统设计逐步升级。

#### 1.1 Agent 与 LLM / ChatBot 的本质区别是什么？
 **考察意图：** 判断候选人是否理解 Agent 的"感知-规划-行动"闭环，而非仅停留在"能调工具的 ChatBot"。

 **回答框架：** 核心区别不在于能否调用工具，而在于三个维度。第一是目标驱动性：LLM 是单轮请求-响应，Agent 围绕一个目标持续运行直到完成或显式终止。第二是自主决策：Agent 自行决定下一步做什么——是调用工具、检索信息、还是直接回答——而非等待用户逐步指令。第三是状态持续性：Agent 维护跨步骤的工作记忆，能基于之前的观察结果调整后续行动。

 从架构上看，LLM API 调用是一次函数求值(input → output)，而 Agent 是一个带有循环控制流的运行时系统。

#### 1.2 Agent 的核心组件如何协同工作？
 **考察意图：** 验证候选人能否画出完整的组件交互图，而非简单列举 Planning / Memory / Tool Use。

 **回答框架：** 一个典型 Agent 系统包含四个核心模块：

 **规划模块(Planning)：** 接收用户目标，分解为可执行的子任务序列。可由 LLM 直接生成(如 ReAct 的思维链)，也可由外部规划器(如 HTN、MCTS)驱动。

 **记忆模块(Memory)：** 短期记忆保持当前任务的上下文窗口内容；长期记忆通过向量数据库或结构化存储持久化历史经验。两者通过检索机制协同——当前上下文不足时，从长期记忆中召回相关信息注入。

 **工具模块(Tool Use)：** 维护可用工具的注册表(含函数签名、语义描述、参数 schema)，模型通过结构化输出表达调用意图，由运行时安全执行并将结果回注上下文。

 **执行引擎(Orchestrator)：** 控制整个循环的流转——接收规划输出、调度工具执行、将观察结果反馈给 LLM、判断是否终止。它还负责异常处理、超时熔断和最大步数限制。

 协同关系的关键在于：规划模块产出"做什么"，工具模块负责"怎么做"，记忆模块解决"基于什么信息做决定"，执行引擎确保整个循环"安全可控地运转"。

#### 1.3 ReAct 的推理-行动-观察循环如何具体实现？
 **考察意图：** 2026 年考 ReAct 主要是考工程实现中的状态管理与容错设计。

 **回答框架：** ReAct 的核心循环可以拆成五个步骤：

 **Thought：** LLM 生成一段推理文本，说明当前应该做什么以及为什么。

 **Action：** 基于推理结果，LLM 输出一个结构化的工具调用请求(含工具名 + 参数)。

 **Execution：** 运行时解析该请求，校验参数合法性后执行工具。

 **Observation：** 将工具返回结果格式化后追加到上下文。

 **Loop or Terminate：** LLM 基于观察结果决定是继续下一轮循环还是输出最终答案。

 关键设计要点有三个。第一是停止条件：必须有显式的终止信号(如特殊 token 或结构化字段 "finish": true)加上兜底的最大步数限制，否则模型可能无限循环。第二是观察结果的格式化：工具返回的原始数据需要压缩和结构化，否则会迅速耗尽上下文窗口。第三是幻觉阻断：当模型连续多轮产生相同或无效的 Action 时，需要检测并强制打断，可通过重复检测 + 回退机制实现。

### 2. 推理与规划范式对比
 这是区分度最高的题目区域。能把 CoT、ReAct、ToT、Reflexion 的结构性差异讲清楚的候选人凤毛麟角。

#### 2.1 对比 CoT、ReAct、ToT 的核心差异
 **考察意图：** 考察候选人对推理范式搜索空间结构的理解深度。

 **回答框架：** 三者的根本差异在于推理结构和搜索空间的形状：

 **CoT(Chain-of-Thought)** 是线性链式推理，搜索空间是一条单一路径。模型一步步推导，没有外部工具交互，也没有回溯能力。优势是简单可控，局限是一旦中间步骤出错，错误会沿链传播且无法修正。

 **ReAct** 是带外部交互的线性推理。它在 CoT 的基础上增加了"行动-观察"环节，模型可以通过工具获取真实信息来校正推理方向。搜索空间仍然是线性的，但每一步都有外部反馈信号。

 **ToT(Tree-of-Thought)** 将搜索空间从线性扩展为树状。模型在每个决策节点生成多个候选推理分支，通过评估函数(自评或外部评估)选择最优分支，支持回溯和剪枝。代价是计算开销随分支因子指数增长。

 选择依据可以简化为：简单推理任务用 CoT，需要外部信息的任务用 ReAct，需要探索多种方案的复杂决策用 ToT。

#### 2.2 端到端 LLM 规划 vs 模块化 Agent 架构在长程任务中的瓶颈
 **考察意图：** 高阶题，判断候选人对 10+ 步决策链场景的工程判断力。

 **回答框架：** 端到端 LLM 规划(如单次 prompt 让模型输出完整计划)的核心瓶颈在于：可靠性随步骤数指数下降。10 步决策链中每步 95% 的正确率，端到端正确率仅约 60%。且出错后缺乏定位手段，调试基本靠"换 prompt 碰运气"。泛化性看似好(毕竟是通用 LLM)，但实际上复杂领域的规划质量极不稳定。

 模块化 Agent 架构(如 Plan-and-Execute)将规划与执行解耦，每个子任务独立执行和验证。可靠性通过子任务级重试和回退提升，可调试性显著改善(能定位到具体哪个子任务失败)。但额外开销在于：模块间的接口定义和状态传递增加了系统复杂度，且规划模块本身的质量仍然依赖 LLM 能力。

 两者的根本权衡是：端到端简单但脆弱，模块化复杂但可控。生产环境中，主流选择是模块化架构加兜底策略。

#### 2.3 除 ReAct 外的推理-行动范式
 **考察意图：** 检验知识广度和深度，面试官通常会追问"你实际用过哪种？"。
**高频考点包括：**
 **Plan-and-Execute：** 先由高层 planner 生成完整计划，再由 executor 逐步执行。设计动机是将"想清楚"和"动手做"分离，避免 ReAct 中规划与执行耦合导致的短视行为。核心组件包括 planner(生成子任务列表)、executor(执行单个子任务)、replanner(根据执行结果修正后续计划)。

 **Reflexion：** 在任务完成后增加一个反思环节，模型评估自己的表现并生成经验总结，存入长期记忆供未来任务使用。设计动机是让 Agent 从失败中学习，而非每次从零开始。

 **ReWOO：** 将推理与工具调用完全解耦——先一次性生成所有推理和工具调用计划，再批量执行所有工具调用，最后将结果统一注入进行最终推理。动机是减少 LLM 调用次数，降低延迟和成本。

### 3. Memory 系统设计
 这是 2026 年热度上升最快的考点。题库中此类问题出现了至少 10 次变体，反映了工业界在生产部署中遇到的真实痛点。

#### 3.1 短期记忆与长期记忆的本质区别与协同设计
 **考察意图：** 这道题出现频次极高，是检验候选人是否有实际系统设计经验的试金石。

 **回答框架：** 两者在三个维度上有本质差异：

| 维度 | 短期记忆 | 长期记忆 |
| :--- | :--- | :--- |
| 信息生命周期 | 当前会话/任务内有效 | 跨会话持久化 |
| 存储机制 | 上下文窗口(token 序列) | 向量数据库 / 知识图谱 / 结构化 DB |
| 访问方式 | 自动包含在每次 LLM 调用中 | 按需检索(语义搜索、关键词匹配) |

 协同设计的关键挑战有三个。第一是注入时机：长期记忆的检索结果应在每轮对话前动态注入短期记忆，但检索延迟和相关性噪声需要权衡。第二是一致性：当短期记忆中的信息与长期记忆冲突时(如用户刚更新了偏好)，需要明确的优先级策略——通常短期优先。第三是淘汰策略：短期记忆超出上下文窗口时，需要决定丢弃什么(简单截断 vs 摘要压缩 vs 重要性评分)。

#### 3.2 长上下文场景下的外部记忆增强机制
 **考察意图：** 不要泛泛而谈，面试官期望听到具体方案的权衡分析。

 **回答框架：** 三种主流方案各有适用场景：

 **滑动窗口(Sliding Window)** 是最简单的策略，保留最近 N 轮对话，丢弃更早的内容。优势是实现零成本、延迟可控；劣势是早期关键信息(如用户声明的约束条件)一旦滑出窗口就永久丢失。适合短对话或信息密度均匀的场景。

 **摘要压缩(Summary Compression)** 在上下文接近上限时，用 LLM 对历史对话生成摘要替换原文。优势是保留关键信息的同时大幅压缩 token 数；劣势是摘要过程本身引入信息损失和延迟，且摘要质量不可控。适合需要保留全局语境的长对话。

 **向量检索增强(Retrieval-Augmented Memory)** 将所有历史对话存入向量数据库，每轮对话前基于当前查询检索相关片段注入上下文。优势是理论上可以覆盖无限长的历史；劣势是检索相关性不保证、引入额外延迟、且需要维护索引更新。适合历史信息分散但偶尔需要精确召回的场景。

 生产环境中通常组合使用：最近几轮保留原文 + 更早的内容摘要化 + 关键信息存入向量库按需检索。

#### 3.3 记忆摘要的触发机制应考虑哪些指标？
 **考察意图：** 非常细节但极实用的设计题，考察工程经验深度。

 **回答框架：** 常见的触发维度有四个，各有优劣：

 **轮数阈值：** 每 N 轮触发一次摘要。简单可预测，但忽略了对话密度差异——有时一轮对话就包含大量关键信息，有时十轮都是闲聊。

 **Token 消耗阈值：** 当上下文 token 数达到窗口容量的某个比例(如 70%)时触发。更贴近实际资源约束，但无法区分哪些 token 是高价值信息。

 **语义边界检测：** 当对话主题发生明显转换时触发摘要。能保持语义完整性，但主题检测本身不可靠，可能误判或漏判。

 **任务阶段转换：** 在 Agent 完成一个子任务并进入下一个子任务时触发。最符合任务逻辑，但需要 Agent 具备明确的阶段感知能力。

 实践中推荐的策略是：以 token 阈值为主触发条件，辅以语义边界检测进行微调，确保摘要不会在一个语义单元的中间发生。

### 4. Tool Calling 与函数调用
 Tool Calling 是 Agent 落地的工程核心。2026 年的面试已经从"知道 Tool Calling 是什么"转向了"20+ 工具时怎么选"和"调用失败怎么恢复"。

#### 4.1 Tool Calling 的核心机制
 **考察意图：** 基础必考题，但好的回答需要覆盖完整的生命周期。

 **回答框架：** Tool Calling 的完整生命周期包含五个环节：

 **工具注册：** 将可用工具的函数签名(名称、描述、参数 JSON Schema)注入系统 prompt 或专用工具描述区域。

 **意图识别：** LLM 在推理过程中判断是否需要调用工具，以及调用哪个工具。模型输出一个结构化对象(如 {"tool": "search", "params": {"query": "..."}})，而非直接执行。

 **参数解析与校验：** 运行时从模型输出中提取工具调用意图，按 JSON Schema 校验参数类型、必填字段和值域约束。

 **安全执行：** 在沙箱环境中执行工具调用，设置超时、权限和资源限制。

 **结果注入：** 将工具返回结果格式化后追加到对话上下文，供模型在下一步推理中使用。

 核心设计原则是 **权责分离**：模型只负责表达"想调什么工具、传什么参数"的意图，实际执行由运行时系统控制。这样模型的错误输出不会直接产生副作用，运行时可以插入校验、审批和回滚机制。

#### 4.2 20+ 工具场景下的工具选择决策维度
 **考察意图：** 高阶题，考察候选人对复杂工具路由问题的系统性思考。

 **回答框架：** 工具选择不是简单的"语义最匹配"，而是一个多维度的联合决策：

 **语义匹配度：** 用户意图与工具功能描述的语义相似性。这是基础维度，但仅靠语义匹配在工具数量多时区分度不够。

 **前置约束满足：** 某些工具有调用前提(如"查询订单详情"需要先获取订单 ID)。未满足前置约束的工具应被过滤。

 **执行代价：** 不同工具的延迟、费用、资源消耗差异显著。在满足功能需求的前提下应优先选择轻量工具。

 **失败回退路径：** 优先选择有替代方案的工具调用链，避免单点故障导致整个任务失败。

 **权限与安全等级：** 涉及敏感操作(如删除、支付)的工具需要额外的确认机制。

 工程实现上，常见策略是分层路由：先用意图分类器将查询路由到工具子集(如"数据查询类"、"操作执行类")，再在子集内用 LLM 做精细选择。这样既降低了 LLM 的选择空间复杂度，也便于针对不同类别设置差异化的安全策略。

#### 4.3 如何防止工具调用的无限循环？
 **考察意图：** 直击生产环境痛点，面试官想听具体的工程手段。

 **回答框架：** 防止无限循环需要多层防线：

 **第一层是硬性限制。** 设置全局最大步数(如 15 步)和单工具最大重试次数(如 3 次)。超出立即终止并返回 fallback 响应。

 **第二层是重复检测。** 监控最近 N 步的工具调用序列，如果出现相同工具 + 相同参数的连续重复调用，判定为循环并强制打断。

 **第三层是收敛性检测。** 追踪每步的"进展指标"(如已完成的子任务数、已获取的新信息量)。如果连续多步没有实质进展，触发重新规划或降级处理。

 **第四层是建模"是否需要外部信息"的决策边界。** 可以在 prompt 中显式要求模型先评估"当前上下文是否已经足够回答问题"，只有在评估为"不足"时才触发工具调用。或者训练一个轻量级分类器做前置判断。

### 5. 多 Agent 协作
 多 Agent 系统是 2026 年最前沿的面试方向。题目从"是什么"全面转向了"怎么设计"和"出了问题怎么办"。

#### 5.1 多 Agent 系统的角色分工、通信与冲突协调
 **考察意图：** 综合性最强的设计题，好的回答需要有清晰的架构思维。

 **回答框架：** **角色分工** 通常基于三种策略：能力感知分配(根据每个 Agent 的专长分配子任务)、语义路由(根据查询内容路由到对应领域 Agent)、负载均衡(在同质 Agent 间分配以提升吞吐)。能力感知分配最常用，但前提是每个 Agent 的能力边界被准确定义——如果定义模糊，会出现任务无人认领或多人争抢。

 **通信设计** 需要在两个维度上做选择。结构化消息 vs 自然语言：结构化消息(JSON schema)确保可解析性和类型安全，但表达能力有限；自然语言灵活但解析不可靠。中心化 vs 去中心化：中心化(通过主 Agent 或消息总线协调)实现简单、一致性好，但主节点是瓶颈和单点故障；去中心化(Agent 间点对点通信)适合高度自治的场景，但一致性维护困难。

 **冲突协调** 的典型场景是两个 Agent 对同一事实给出矛盾结论。常见机制包括：投票表决(多数 Agent 的结论胜出)、置信度加权(按每个 Agent 输出的置信度加权融合)、仲裁 Agent(专门的 Judge Agent 裁决分歧)。

#### 5.2 中心化协调 vs 去中心化协商的适用条件
 **考察意图：** 考察对系统架构模式的理解，以及在具体场景下的选型能力。

 **回答框架：** **中心化协调** 适用于三种情况：任务之间有强依赖关系(子任务 B 的输入依赖子任务 A 的输出)、需要全局一致性保障(如最终输出不能自相矛盾)、Agent 数量较少(通常 < 10)且异构性强。典型代表是 Supervisor 模式，一个主 Agent 持有全局状态和任务计划，子 Agent 只执行被分配的子任务并汇报结果。风险是主 Agent 成为性能瓶颈和认知瓶颈——当子任务数量增多时，主 Agent 的上下文窗口和决策质量同步下降。

 **去中心化协商** 适用于：任务之间松耦合、Agent 高度自治且能力对等、系统需要弹性扩展。典型代表是 Contract Net Protocol——一个 Agent 广播任务需求，其他 Agent 根据自身能力竞标，发起者选择最优竞标者。风险是协商过程本身消耗大量通信开销，且缺乏全局视角容易导致局部最优。

#### 5.3 单 Agent 输出不可靠时的鲁棒性保障机制
 **考察意图：** 面试官通过此题判断你是否有生产环境经验。

 **回答框架：** 常见机制按复杂度递增排列：

 **置信度门控：** 每个 Agent 输出附带置信度分数，低于阈值的输出不参与最终决策或被标记为"需人工审核"。实现简单，但置信度校准本身是难题——LLM 的自评置信度往往过高。

 **交叉验证：** 让多个 Agent 独立处理相同子任务，对比结果一致性。不一致时触发仲裁。代价是计算成本翻倍，且需要定义"一致"的判断标准。

 **回滚协议：** 当下游 Agent 发现上游输出有问题时，触发回滚到上一个已验证的检查点，重新执行。设计难点在于确定回滚边界——哪些步骤的中间结果可以安全重用，哪些需要完全重做。

 **结果验证模块：** 在 Agent 输出和最终使用之间插入一个独立的验证层(可以是规则引擎、另一个 LLM 或领域知识库)。这是最可靠的方案，但验证模块本身的设计和维护是额外成本。

### 6. RAG 系统核心问题
 RAG 虽然不是纯 Agent 话题，但在 Agent 面试中几乎必考——因为绝大多数 Agent 系统都需要 RAG 作为知识检索模块。

#### 6.1 RAG 系统的端到端流程与关键挑战
 **考察意图：** 验证候选人对 RAG 全链路的掌握程度。

 **回答框架：** RAG 的端到端流程分为四个阶段，每个阶段都有核心挑战：

 **查询处理：** 用户原始查询通常不适合直接用于向量检索(太口语化、太模糊、或包含多个子问题)。关键技术包括查询重写(让 LLM 将口语化查询改写为检索友好形式)、查询分解(将复合问题拆分为多个子查询)、HyDE(生成假设性答案文档来增强查询向量)。

 **检索与召回：** 核心挑战是召回率与精确率的平衡。稠密向量检索擅长语义匹配但可能遗漏精确关键词，BM25 擅长精确匹配但缺乏语义理解。混合检索 + RRF 融合是当前主流方案。

 **重排序(Rerank)：** 用 Cross-Encoder 等精排模型对初筛结果重新打分。核心权衡是精度提升 vs 延迟开销——Cross-Encoder 需要对每个 query-document 对做完整的交叉注意力计算。

 **生成与忠实性控制：** 核心挑战是让 LLM 基于检索内容生成而非"自由发挥"。常见策略包括在 prompt 中显式要求"仅基于以下内容回答"、引用标注、以及后处理阶段的事实一致性校验。

#### 6.2 文档分块(Chunking)策略的权衡
 **考察意图：** 高频题，且有多种变体。核心考察对粒度-语义完整性-检索精度三角关系的理解。

 **回答框架：** 分块策略的核心权衡是：块太小，语义不完整，检索召回的片段缺乏足够上下文；块太大，向量表示被稀释，检索精度下降，且可能引入无关噪声。

 主流策略包括：

 **固定长度切分：** 实现最简单，适合结构均匀的文本。风险是在句子或段落中间断开，导致语义断裂。

 **基于结构标记切分(如** **Markdown** **标题、HTML** **标签)：** 利用文档自身的层级结构识别语义边界。适合结构化文档，但对无结构或结构混乱的文档无效。

 **语义切分：** 通过嵌入向量相似度检测相邻段落的语义转换点，在语义变化最大处切分。理论上最优，但计算开销大且切分点的阈值需要调优。

 **滑动窗口 + 重叠：** 在任何切分策略基础上引入片段间重叠(通常 10-20%)，缓解边界处的语义断裂。代价是存储冗余和检索时可能召回高度重复的片段。

 选择依据取决于文档类型：法律条文适合按条款结构切分，代码适合按函数/类切分，长叙事文本适合语义切分 + 重叠。

#### 6.3 检索冗余与多样性控制
 **考察意图：** 考察候选人对 MMR 等技术的理解以及其局限性认知。

 **回答框架：** 语义相似但信息冗余的片段会浪费上下文窗口空间并干扰生成质量。解决方案分两端：

 **检索端** 的核心方法是 MMR(Maximal Marginal Relevance)，在选择下一个文档时同时考虑与查询的相关性和与已选文档的差异性，通过 λ 参数控制相关性-多样性的权衡。MMR 的局限在于：它假设文档间的相似度可以用向量余弦距离准确衡量，且无法捕捉信息层面(而非语义层面)的冗余。

 **后处理端** 包括：聚类去重(对召回文档做聚类，每个聚类只保留最相关的代表文档)、摘要融合(将多个高度重叠的片段合并为一个精简摘要)。

### 7. 系统工程与生产可靠性
 这是最具实战区分度的题目区域。能回答好这些问题的候选人，通常有实际的 Agent 系统上线经验。

#### 7.1 当前 Agent 系统的关键技术瓶颈
 **考察意图：** 面试官想听你从架构层面的根因分析，而非列举表面现象。

 **回答框架：** 两个最根本的瓶颈：

 **规划的不可靠性** 源于 LLM 的自回归生成本质。模型在生成计划时是逐 token 展开的，无法像传统规划器那样做全局搜索和约束求解。结果是：计划的质量高度依赖 prompt 和模型能力，且随任务复杂度增加急剧下降。中间某一步的规划错误会通过后续步骤的条件依赖链放大。

 **状态跟踪的脆弱性** 源于 Transformer 注意力机制的局限。在长上下文中，模型对早期信息的关注度随距离衰减(尽管理论上注意力是全局的，实际训练分布导致了"近端偏好")。当 Agent 执行了 10+ 步后，早期的用户约束或工具返回结果可能被有效"遗忘"，导致后续行为与初始目标偏离。

#### 7.2 Planning Agent 的容错与恢复机制
 **考察意图：** 直接考察系统设计能力。

 **回答框架：** 当高层规划的某个子任务执行失败时，容错机制应按优先级依次尝试：

 **本地重试：** 对同一子任务用不同参数或策略重试(如换一个搜索关键词)。设置最大重试次数。

 **子任务替换：** 如果原子任务有等价替代方案(如 API A 不可用时切换到 API B)，自动路由到替代路径。

 **局部重规划：** 将失败子任务及其后续依赖任务返回给 planner，要求基于当前已有结果重新规划剩余部分。关键是保留已成功执行的结果，避免全量重做。

 **全局降级：** 当多次重规划仍失败时，降级为返回部分结果 + 失败说明，而非无限重试。

 每种机制都需要配合 checkpointing——在每个子任务成功完成后保存状态快照，使回滚和重规划有明确的恢复起点。

### 附：2026 Agent 面试的三个趋势总结
 **从概念复述到工程判断。** 面试不再问"ReAct 是什么"，而是问"ReAct 循环中出现幻觉导致工具被反复错误调用，你的阻断策略是什么？"候选人需要展示的是处理过真实问题的经验和判断力。

 **Memory 设计成为一等公民。** 长短期记忆协同、摘要触发策略、跨 Agent 状态同步——这些问题在题库中反复出现，频次甚至超过了 RAG 检索优化。这反映了工业界的真实痛点：Agent 系统上线后最先暴露的问题往往不是检索不准，而是对话长了之后"忘事"。

 **多 Agent 协作从论文走向面试桌。** 角色分工策略、通信协议设计、容错回滚机制、一致性保障——这些不再是 ICML workshop 的讨论话题，而是面试中的高频问题。原因很直接：2026 年越来越多的生产系统开始采用多 Agent 架构，企业需要能设计和维护这类系统的工程师。

早期的 LLM 应用大多是"一问一答"：给定输入，模型输出一段文本，交互结束。但当任务变得复杂——需要访问外部工具、处理多步骤依赖、或者在执行中途纠正错误——单轮生成就力不从心了。

 Agent 框架的出现正是为了应对这类挑战。一个 Agent 不仅能调用工具，还需要规划、执行、观察反馈，并在必要时调整策略。而支撑这些能力的，是三种经过充分研究的推理机制： **ReAct** (推理与行动交织)、 **Reflection** (自我反思与修正)、 **Memory** (跨步骤或跨会话的信息持久化)。

### ReAct：将推理链与工具调用交织执行
#### 核心思想
 ReAct(Reasoning + Acting)由 Yao et al. (2022) 提出，核心想法是： **让模型在每次行动前先显式推理，在观察到行动结果后再继续推理**，形成 Thought → Action → Observation 的循环。

 与纯 Chain-of-Thought(CoT)相比，ReAct 的推理步骤不再只是内部语言游戏，而是和真实的外部操作(搜索、计算、API 调用)紧密耦合。与纯工具调用(如 function calling)相比，ReAct 保留了可解释的中间推理过程，便于调试和干预。

#### 执行循环
```text
Thought: 分析当前状态，决定下一步行动
Action: tool_name(args)
Observation: <工具返回的结果>
Thought: 根据观察更新理解，规划下一步
Action: ...
...
Final Answer: <最终输出>
```
每一轮 Thought 是模型对当前信息的内化与分析； Action 是对外部环境的操作； Observation 是环境反馈，直接拼接回 prompt 供下一轮使用。

#### 代码示例(基于 LangChain 风格伪代码)
```python
from langchain.agents import create_react_agent
from langchain_core.tools import tool

@tool
def search_web(query: str) -> str:
    """搜索互联网获取最新信息"""
    return web_search_api(query)

@tool
def python_repl(code: str) -> str:
    """执行 Python 代码并返回结果"""
    return exec_sandbox(code)

### 创建 ReAct Agent
agent = create_react_agent(
    llm=llm,
    tools=[search_web, python_repl],
    prompt=react_prompt_template  # 包含 Thought/Action/Observation 格式说明
)

### Agent 会自动循环执行，直到输出 Final Answer
result = agent.invoke({"input": "2024年全球AI芯片市场规模是多少？"})
```
#### 局限性
 ReAct 的主要问题是 **错误累积**：一旦某步 Thought 判断有误，后续行动都会建立在错误前提上，且模型不会主动意识到偏差。此外，对于需要长距离规划的任务，每步只看"当前观察"的局部性也是瓶颈。

### Reflection：让 Agent 审视并修正自身输出
#### 为什么需要自我反思
 ReAct 解决了"如何行动"的问题，但没有解决"如何判断行动是否正确"的问题。Reflection 机制引入了一个额外的评估步骤：模型(或独立的 critic 模型)对已有的输出或轨迹进行批评，再由 actor 模型根据反馈修改。

 这一思路的代表工作包括：

 **Reflexion** (Shinn et al., 2023)：将反思结果以自然语言形式写入记忆，在下一次尝试时作为上下文输入

 **Self-Refine** (Madaan et al., 2023)：迭代地让模型对自身输出提供 feedback 并改写，无需外部信号

 **Critic-Actor 架构**：将生成(Actor)和评估(Critic)分离为两个独立 LLM 调用，降低自我验证的确认偏差

#### Reflexion 的记忆写入机制
 Reflexion 的关键设计在于：反思结果不是临时的，而是 **写入持久化的语言记忆**，在下一轮任务启动时读取。
```text
Episode 1:
  尝试解题 → 失败
  Reflection: "我直接搜索了结果，但忽略了单位换算，导致数值错误。
               下次应该先确认单位再进行计算。"
  → 写入 memory

Episode 2:
  读取 memory 中的反思 → 在 Thought 中主动检查单位 → 成功
```
这与人类从错误中学习的过程高度一致：失败本身不产生学习，对失败的反思和归因才产生学习。

#### 实现时的工程考量
 **单模型 vs. 双模型**：让同一模型既生成又反思存在确认偏差——模型往往会认为自己的输出是合理的。用独立的 critic prompt 甚至独立的模型实例，能有效缓解这一问题。

 **反思粒度**：粒度太细(针对每个 token)开销过大；粒度太粗(只在任务失败时反思)信号稀疏。实践中常以"步骤"或"子任务"为单位触发反思。

 **终止条件**：迭代反思如果不加限制，容易陷入无限循环或过拟合于某个错误的修正方向。需要设置最大迭代次数，并引入外部验证信号(如单元测试通过率、人工评分)作为停止依据。
```python
def reflection_loop(task, actor_llm, critic_llm, max_iter=3):
    output = actor_llm.generate(task)

    for i in range(max_iter):
        # Critic 评估当前输出
        critique = critic_llm.evaluate(
            task=task,
            output=output,
            prompt="指出输出中的具体问题，说明如何改进")

        # 如果 Critic 认为质量达标，提前终止
        if critique.is_satisfactory:
            break

        # Actor 根据 critique 修改输出
        output = actor_llm.refine(
            task=task,
            previous_output=output,
            feedback=critique.content)

    return output
```
### Memory：跨步骤与跨会话的信息持久化
#### 记忆的四种类型
 Agent 的记忆系统通常按 **持久化范围** 和 **检索方式** 分为四类，对应人类记忆的不同层次：

| 类型 | 存储位置 | 生命周期 | 典型用途 |
| :--- | :--- | :--- | :--- |
| **In-context Memory** | Prompt 上下文 | 当前会话 | 对话历史、中间结果 |
| **External Memory** | 向量数据库 / KV 存储 | 持久化 | 长期知识、用户偏好 |
| **Episodic Memory** | 结构化日志 | 持久化 | 过往任务轨迹、Reflexion 反思 |
| **Procedural Memory** | 模型权重(微调) | 永久 | 技能、格式、领域知识 |

 实际 Agent 系统中最常用的是前两种，因为它们无需改变模型本身。

#### In-context Memory 的管理
 In-context Memory 受 token 窗口限制，需要主动管理：

 **滑动窗口**：只保留最近 N 轮对话，丢弃早期历史。简单但会丢失重要信息。

 **摘要压缩**：当历史过长时，用 LLM 将早期历史压缩为摘要，再拼接进 prompt。
```python
def compress_history(messages: list, llm, threshold=4000) -> list:
    """当历史超过阈值时，压缩早期消息"""
    if token_count(messages) < threshold:
        return messages

    # 保留最近 10 轮，压缩更早的历史
    recent = messages[-10:]
    older = messages[:-10]

    summary = llm.invoke(
        f"将以下对话历史压缩为简洁摘要，保留关键信息：\n{format(older)}")

    return [{"role": "system", "content": f"早期对话摘要：{summary}"}] + recent
```
**选择性保留**：用模型或规则判断哪些消息"重要"(如包含用户偏好、关键决策)，优先保留。

#### External Memory 与 RAG 集成
 External Memory 通常以向量数据库实现，与 RAG(Retrieval-Augmented Generation)架构天然契合：
```text
写入路径：新信息 → Embedding 模型 → 向量数据库
读取路径：当前查询 → Embedding 模型 → 相似度检索 → 注入 Prompt
```
对于 Agent 场景，External Memory 的写入时机很关键：

 **任务完成后写入**：将任务结果、Reflection 反思写入，供未来任务检索

 **实时写入**：每个重要的 Observation 都持久化，适用于长期运行的 Agent

 **手动触发写入**：用户或系统显式标记"值得记住"的信息

#### 记忆检索的精度问题
 向量相似度检索的精度并不总能满足需求。常见的改进策略：

 **混合检索(Hybrid Search)**：将语义相似度(向量)与关键词匹配(BM25)结合，提升精确匹配的召回率。

 **时间衰减**：为记忆条目附加时间戳，检索时对近期记忆给予更高权重，避免过时信息干扰当前决策。

 **结构化元数据过滤**：在向量检索前，先用结构化字段(任务类型、用户 ID、时间范围)做硬过滤，缩小检索空间。

### 三者的协同：一个完整 Agent 的推理栈
 孤立地看，ReAct 负责执行、Reflection 负责修正、Memory 负责积累。在完整的 Agent 系统中，三者构成一个多层次的推理栈：
```text
任务输入
  │
  ▼
[Memory 读取] ← 检索相关历史经验和 Reflection 笔记
  │
  ▼
[ReAct 执行循环]
  Thought → Action → Observation → Thought → ...
  │
  ▼
[Reflection 评估]
  是否达到目标？输出质量是否满足要求？
  │
  ├── 满足 → 输出最终结果
  │
  └── 不满足 → 生成反思笔记 → [Memory 写入] → 重试 ReAct
```
这个结构在 HumanEval、AlfWorld、WebShop 等基准上都有实验验证：三者组合的成功率显著高于任意单一机制。

 **典型的工程取舍**：

 对于 **响应时延敏感** 的场景(如实时对话)，Reflection 的迭代开销难以接受，可以省略，只保留 ReAct + 轻量 Memory

 对于 **一次性任务** (如代码生成、报告撰写)，Reflection 的多轮迭代带来的质量提升通常值得等待

 对于 **长期运行的 Agent** (如自动化运营、持续监控)，Memory 的设计是核心挑战，需要仔细考虑写入策略、检索精度和记忆老化

### 总结
 ReAct 将推理与行动解耦但又显式关联，解决了纯 CoT 无法感知外部反馈的问题；Reflection 引入自我评估循环，让 Agent 能从失败中积累改进信号；Memory 提供跨步骤和跨会话的信息持久化，使 Agent 的能力随时间积累而非每次从零开始。

 当前这三种机制仍有若干开放问题值得持续关注：Reflection 的评估信号质量(如何判断"反思是否准确"？)、Memory 的遗忘与更新策略(旧记忆如何老化而不干扰新任务？)、以及多 Agent 场景下记忆共享的一致性保证。这些问题既是工程挑战，也是当前学术研究的活跃方向。
