如果前几篇文章考的是"能不能训出来"，这一篇考的就是"能不能跑得动、跑得快、跑得稳"。2026 年推理与工程方向的面试有两个显著特征： **KV Cache 相关问题占比超过 30%** ，以及 **vLLM/FlashAttention 已成为默认基础设施知识** ——不了解这些的候选人，连面试的"入场券"都拿不到。

本文从 100+ 道真实题目中提炼核心考点，涵盖推理机制、显存优化、量化部署、推理框架和系统工程五大主题。

## 一、KV Cache：推理优化的绝对核心

KV Cache 在这份题库中出现了 **20+ 次** ，是推理方向当之无愧的第一考点。面试官从"是什么"一直追问到"怎么压缩"和"怎么分页管理"。

### 1.1 KV Cache 的核心作用与复杂度分析

**考察意图：** 入门必答，但面试官期望你能准确说出复杂度变化。

**回答框架：**

在自回归解码中，生成第 t 个 token 时需要计算它对所有前 t-1 个 token 的注意力。如果不做缓存，每生成一个新 token 都要重新计算所有历史 token 的 K 和 V 向量，单步注意力的时间复杂度为 O(t · d)，生成长度为 L 的序列总复杂度为 O(L² · d)。

KV Cache 的做法是： **将已计算过的历史 token 的 K 和 V 向量缓存在显存中，每步只计算新 token 的 Q/K/V，然后与缓存中的历史 K/V 做注意力** 。这将单步复杂度降至 O(t · d)（只需新 Q 与所有历史 K 做点积），但由于不再重复计算历史 K/V，总计算量从 O(L²) 量级降至 O(L) 量级。

本质上，KV Cache 是一个经典的 **空间换时间** 设计：用显存存储历史键值对，换取避免重复计算的时间收益。

**显存开销公式：** 对于一个 L 层、h 个注意力头、每头维度 d\_k 的模型，生成长度为 s 的序列，KV Cache 占用 = 2 × L × h × d\_k × s × sizeof(dtype)。以 LLaMA-7B（L=32, h=32, d\_k=128, FP16）生成 4K tokens 为例：2 × 32 × 32 × 128 × 4096 × 2 bytes ≈ **2 GB** 。

当序列长度或并发 batch 增大时，KV Cache 显存会迅速成为瓶颈。

### 1.2 KV Cache 的主流优化方法

**考察意图：** 出现 10+ 次，考察对不同优化策略的系统性理解。

**回答框架：**

KV Cache 优化可按照策略分为四大类：

**分页管理（PagedAttention / vLLM）：** 传统实现为每个序列预分配一块连续显存，长度按最大可能值设定，导致大量显存浪费（碎片化）。PagedAttention 借鉴操作系统虚拟内存的分页思想，将 KV Cache 拆分为固定大小的"页"（如 16 tokens 一页），按需分配和释放，显存利用率从传统方案的 20-40% 提升到 90% 以上。

**量化压缩：** 将 KV Cache 从 FP16 量化到 INT8 或 INT4，直接将显存减半或减至 1/4。核心挑战在于注意力计算对 K/V 的精度敏感——K 的量化误差会通过 softmax 被放大。常见做法是对 K 保留更高精度、对 V 做更激进的量化。

**稀疏化 / 截断：** 基于观察到的注意力模式（如大部分注意力集中在近邻 token 和少数"锚点" token），只保留部分 KV 条目。策略包括滑动窗口（只保留最近 N 个 token 的 KV）、注意力头剪枝（移除注意力分数方差低的头的 KV）、动态截断（基于注意力分数实时筛选需保留的 KV 条目）。

**Chunked Prefill：** 将长输入的预填充阶段分块处理，避免一次性分配整个输入长度的 KV Cache。这在处理长上下文输入时尤其重要。

### 1.3 预填充（Prefill）与解码（Decode）的本质区别

**考察意图：** 出现 5+ 次，考察对推理两阶段计算特征的精确理解。

**回答框架：**

自回归推理分为两个截然不同的阶段：

| 维度 | Prefill（预填充） | Decode（解码） |
| --- | --- | --- |
| 计算模式 | 并行处理所有输入 token | 逐个生成新 token |
| 核心操作 | 大矩阵乘法（compute-bound） | 向量-矩阵乘法（memory-bound） |
| 并行性 | 高（所有 token 同时计算） | 低（严格串行） |
| 瓶颈 | 计算量（FLOPS） | 显存带宽（读取权重和 KV Cache） |
| KV Cache | 一次性生成全部输入 token 的 KV | 每步追加一个 token 的 KV |

Prefill 阶段是 **compute-bound** ——输入序列越长，计算量越大，但可以充分利用 GPU 的并行计算能力。Decode 阶段是 **memory-bound** ——每步只生成一个 token，计算量很小，但需要从显存中读取整个模型权重和不断增长的 KV Cache，受限于显存带宽。

显存瓶颈方面：Prefill 阶段需要一次性为整个输入序列分配 KV Cache 空间，当输入很长时可能触发 OOM；Decode 阶段 KV Cache 逐步增长，且在高并发时多个序列的 KV Cache 累积效应显著。两阶段各有显存压力点，但 Prefill 的峰值显存通常更高（因为还需存储中间激活值）。

## 二、FlashAttention：必修基础设施知识

FlashAttention 在题库中出现 **8+ 次** ，已从"加分项"变为"必答题"。

### 2.1 FlashAttention 的核心原理

**考察意图：** 高频必考，需要理解 IO 感知设计的本质。

**回答框架：**

标准 Attention 的计算过程是：

1. 计算 S = QK^T（n×n 矩阵），写入 HBM
2. 计算 P = softmax(S)，写入 HBM
3. 计算 O = PV，写入 HBM

这个过程需要将中间结果 S 和 P（各 O(n²) 大小）写入高带宽显存（HBM），在长序列下 HBM 读写成为瓶颈。

FlashAttention 的核心思想是 **IO 感知的分块计算（Tiling）** ：

- 将 Q、K、V 矩阵分成小块（tile），使每块能放入 GPU 的片上共享内存（SRAM，容量小但带宽极高）
- 在 SRAM 中完成该块的 QK^T、softmax、PV 计算，只将最终结果写回 HBM
- 通过 **在线 softmax 算法** （维护运行中的 max 和 sum），实现分块计算 softmax 时的数值正确性

结果是： **显存复杂度从 O(n²) 降至 O(n)** （不需要具现化 n×n 的注意力矩阵），HBM 访问次数大幅减少，实际速度提升 2-4 倍。

关键点是：FlashAttention **不降低理论计算量** （FLOPs 不变），它优化的是 **计算访存比** ——减少了对慢速 HBM 的读写，让计算单元（Tensor Core）不再等待数据。

### 2.2 FlashAttention 的版本演进

**考察意图：** 出现 4+ 次，考察对技术迭代的跟踪。

**回答框架：**

**FlashAttention v1：** 提出 IO 感知 Tiling + 在线 softmax 的核心框架。显存从 O(n²) 降至 O(n)，速度提升 2-4x。但 SRAM tile size 固定，未充分利用 GPU 并行度。

**FlashAttention v2：** 优化了三个关键点：减少了非矩阵乘法的 FLOPs（如 softmax rescaling 的冗余计算）、改进了 Q/K/V 的循环顺序使并行度更高（外循环遍历 Q 而非 K/V）、更好地利用了 GPU 的 warp 级并行。实测速度在 v1 基础上再提升约 2x。

**FlashAttention v3（Hopper 架构适配）：** 针对 H100 GPU 的新硬件特性（如 TMA——Tensor Memory Accelerator、wgmma 指令）做了深度适配。利用 TMA 实现异步数据预取（计算和数据搬运重叠），进一步提升吞吐。

核心演进主线是： **从软件层面的算法优化 → 与硬件微架构深度协同** ，每一代都在更细粒度的层面挖掘 GPU 的性能潜力。

## 三、量化部署

量化是大模型从实验室走向生产的关键技术，面试中出现频率稳定在 6+ 次。

### 3.1 GPTQ vs AWQ vs SmoothQuant 的核心差异

**考察意图：** 高频对比题，考察对不同量化范式设计动机的理解。

**回答框架：**

三者的核心设计差异在于"如何处理量化对模型精度的影响"：

**GPTQ（逐层权重量化）：** 基于二阶信息（Hessian 矩阵的近似）对权重逐层进行最优量化。核心思想是：量化某个权重时，考虑该层所有权重的相互关系，用未量化的权重"补偿"已量化权重引入的误差。需要少量校准数据来估计 Hessian，但只量化一次（不需要在线校准）。适合权重量化到 INT4/INT3 的极低精度场景。

**AWQ（激活感知权重量化）：** 核心观察是：权重中的不同通道对模型输出的重要性不同，少数"显著通道"（salient channels）承载了大部分关键信息。AWQ 通过分析激活值分布找到这些显著通道，对它们施加缩放因子使其量化误差更小。不需要逐层校准，更轻量。

**SmoothQuant（激活平滑量化）：** 解决的问题不同——激活值中存在极端离群值（outlier），使得激活的量化非常困难。SmoothQuant 通过一个数学等价变换，将激活中的离群值"转移"到权重中（乘以一个逐通道缩放矩阵），使激活分布更平滑、更易量化。特别适合 W8A8（权重和激活都量化到 INT8）的场景。

选择依据：只做权重量化（推理时仍用 FP16 激活）→ GPTQ 或 AWQ；需要权重+激活同时量化 → SmoothQuant；追求极低精度（INT4/INT3）→ GPTQ；追求部署简便 → AWQ。

### 3.2 INT8 vs INT4 量化的权衡

**考察意图：** 考察对量化精度和硬件约束的实际理解。

**回答框架：**

| 维度 | INT8 | INT4 |
| --- | --- | --- |
| 表示范围 | \[-128, 127\]，256 个值 | \[-8, 7\]，16 个值 |
| 显存节省（vs FP16） | 2x | 4x |
| 精度损失 | 通常可控（<1% 性能下降） | 显著，需额外补偿 |
| 硬件支持 | 广泛（A100/H100 原生支持） | 有限（需特殊 kernel） |

INT4 的核心挑战是：16 个离散值远不足以表示权重的完整分布。缓解策略包括：

**分组量化（Group Quantization）：** 将权重矩阵按固定大小（如 128 个元素一组）分组，每组独立计算缩放因子和零点。这样每组只需用 INT4 表示组内的相对差异，而组间差异由浮点缩放因子捕捉。

**Outlier 混合精度处理：** 对少数分布极端的权重通道保留 FP16 精度，其余用 INT4。代价是实现复杂度增加。

### 3.3 FP16 vs BF16 的选择

**考察意图：** 出现 3+ 次，考察对数值格式的理解。

**回答框架：**

两者都是 16 位浮点，但位分配不同：

**FP16：** 1 位符号 + 5 位指数 + 10 位尾数。精度高（尾数 10 位），但动态范围小（指数 5 位，最大值约 65504）。

**BF16：** 1 位符号 + 8 位指数 + 7 位尾数。精度低（尾数 7 位），但动态范围与 FP32 相同（指数 8 位）。

BF16 更适合大模型训练的原因：大模型训练中梯度和激活值的动态范围很大（从极小到极大），FP16 的窄动态范围容易导致上溢或下溢（需要 loss scaling 来补偿），BF16 的宽动态范围避免了这个问题。精度损失在大模型的统计平均效应下影响有限。

## 四、推理框架与系统设计

vLLM 已成为 2026 年推理框架的"默认知识"，相关问题出现 8+ 次。

### 4.1 vLLM 的核心设计

**考察意图：** 必考题，PagedAttention 是 vLLM 的标志性创新。

**回答框架：**

vLLM 的核心优化围绕两个支柱：

**PagedAttention（分页 KV Cache 管理）：** 传统方案为每个序列分配一块连续显存，按最大生成长度预留，导致显存碎片化严重（实际利用率低于 40%）。PagedAttention 将 KV Cache 拆分为固定大小的"页"，维护一个虚拟-物理页映射表。好处有三：按需分配（不预留）、非连续存储（消除碎片）、页共享（prompt 共享场景下多个序列可共享同一页的 KV Cache）。

**连续批处理（Continuous Batching）：** 传统批处理需要等一个 batch 中所有序列都结束才能开始下一个 batch。连续批处理允许在任何序列结束时立即插入新请求，最大化 GPU 利用率。

两者协同的效果：PagedAttention 解决了"多个序列并发时 KV Cache 的高效管理"问题，连续批处理解决了"请求到达时间不均匀时的调度效率"问题。合在一起使 vLLM 的吞吐量比传统实现高 2-4 倍。

### 4.2 SGLang 与 vLLM 的对比

**考察意图：** 考察对新一代推理框架的跟踪。

**回答框架：**

SGLang 在 vLLM 的基础上进一步优化了 **复杂推理调度** 场景：

**RadixAttention（前缀树共享）：** 对于有共同前缀的多个推理请求（如同一 system prompt + 不同用户输入），SGLang 用 Radix Tree 组织 KV Cache，最大化前缀共享。这在树状生成（如 Tree-of-Thought）和并行采样（如 Best-of-N）场景下尤其有效。

**编程模型：** SGLang 提供了 Python 级别的推理编程接口，支持在一次推理调用中组合多个生成步骤（如先生成分析、再生成代码），调度器能跨步骤复用 KV Cache，避免冗余计算。

与 vLLM 的核心差异：vLLM 优化的是"单次请求"的显存和吞吐，SGLang 优化的是"多步复杂推理"的端到端延迟。

### 4.3 推理性能的量化评估维度

**考察意图：** 出现 4+ 次，考察对性能指标的全面理解。

**回答框架：**

大模型推理性能应从以下维度评估：

**首 Token 延迟（Time to First Token, TTFT）：** 从请求到达到输出第一个 token 的时间。由 Prefill 阶段主导，输入越长 TTFT 越高。对实时对话体验至关重要。

**Token 间延迟（Inter-Token Latency, ITL）：** 相邻 token 之间的生成间隔。由 Decode 阶段单步推理时间决定。影响流式输出的流畅度。

**吞吐量（Throughput）：** 单位时间内生成的 token 总数（tokens/sec）。受限于 batch size、KV Cache 显存和 GPU 计算能力的最小瓶颈。

**显存占用（Memory Footprint）：** 模型权重 + KV Cache + 激活值的总显存。决定了能同时服务多少并发请求。

**P99 延迟：** 第 99 百分位的端到端延迟。在生产环境中比平均延迟更重要，因为少数长尾请求会严重影响用户体验。

各指标间的典型权衡：增大 batch size 提升吞吐但增加延迟；量化降低显存但可能影响生成质量；KV Cache 压缩降低显存但增加计算开销。

## 五、显存估算与分布式训练

显存分析和分布式并行策略是大模型工程岗位的"硬通货"。

### 5.1 推理显存估算

**考察意图：** 笔试高频题，面试官期望你能算出具体数字。

**回答框架：**

以 Qwen3-8B、FP16 为例：

**模型权重：** 8B × 2 bytes = **16 GB**

**KV Cache（单序列 4K tokens）：** 假设 L=32 层，h=32 头，d\_k=128，则 2 × 32 × 32 × 128 × 4096 × 2 bytes ≈ **2 GB / 序列**

**激活值（单步推理）：** 推理时不需要存储梯度，激活值可以逐层丢弃（不像训练需要保留用于反向传播），通常只需存储当前层的中间结果，占用较小（约数百 MB）。

**最小显存：** 模型权重 16 GB + 单序列 KV Cache 约 2 GB ≈ **18 GB** （需要至少 24 GB GPU，如 A10、4090）。

如果需要同时服务 8 个并发序列：16 + 8 × 2 = **32 GB** （需要 A100-40G 或更高）。

推论：在高并发场景下， **KV Cache 而非模型权重才是显存的主要消耗者** 。这就是为什么 KV Cache 优化（PagedAttention、量化、稀疏化）如此重要。

### 5.3 3D 并行策略与通信瓶颈

**考察意图：** 出现 4+ 次，考察对分布式训练架构的理解。

**回答框架：**

3D 并行是三种并行策略的组合：

**数据并行（Data Parallelism）：** 每个 GPU 持有完整模型副本，处理不同的数据 batch。通信发生在 **梯度同步** 阶段——所有 GPU 需要 AllReduce 梯度，通信量 = 模型参数量 × dtype 大小。优势是实现简单，劣势是每 GPU 需要存储完整模型。

**张量并行（Tensor Parallelism）：** 将单层的权重矩阵切分到多个 GPU 上。通信发生在 **每层的前向和反向传播中** ——矩阵乘法的结果需要跨 GPU AllReduce。优势是减少单 GPU 显存，劣势是通信极频繁（每层两次），需要高速互联（如 NVLink）。

**流水线并行（Pipeline Parallelism）：** 将模型的不同层分配到不同 GPU 上。通信发生在 **层间的激活传递** ——上一组层的输出需要发送到下一组层的 GPU。通信量较小，但存在 **pipeline bubble** （某些 micro-batch 计算时其他 GPU 空闲）。

**通信瓶颈排序：** 张量并行 > 数据并行 > 流水线并行。张量并行的通信量最大且最频繁（每层都通信），因此通常限制在单机内的 GPU 间（利用 NVLink 的高带宽）；数据并行的梯度同步每步一次，可以跨机器（利用 InfiniBand）；流水线并行通信量最小，可以跨机器。

### 5.4 DeepSpeed ZeRO 三阶段

**考察意图：** 出现 5+ 次，与前篇的训练显存估算互补。

**回答框架：**

ZeRO 的核心思想是消除数据并行中的冗余存储。标准数据并行中，每个 GPU 都存储完整的模型参数、梯度和优化器状态——这是 N 倍的冗余（N = GPU 数量）。

**ZeRO-1：** 分片优化器状态。将 Adam 的一阶矩、二阶矩和 FP32 主参数副本分到各 GPU，每 GPU 只存 1/N。这一步消除了最大的冗余——优化器状态通常占总显存的 60-70%。

**ZeRO-2：** 在 ZeRO-1 基础上分片梯度。每 GPU 只存储其负责更新的那部分参数的梯度。梯度在 AllReduce 后可按分片丢弃。

**ZeRO-3：** 在 ZeRO-2 基础上分片模型参数本身。每 GPU 只存 1/N 的参数。前向和反向传播时需要 AllGather 收集完整参数，用完即释放。通信开销最大，但显存节省最极致。

## 六、系统工程与生产部署

这部分题目覆盖面广，从 API 设计到缓存策略到并发处理，考察候选人的全栈工程能力。

### 6.1 流式响应（Streaming）的端到端实现

**考察意图：** 出现 5+ 次，是大模型服务的核心工程问题。

**回答框架：**

流式响应的端到端链路涉及四个环节：

**Token 生成层：** 模型每 decode 一步产生一个（或一批）token，通过回调或队列推送到传输层。关键指标是 Inter-Token Latency——决定了流式输出的"流畅度"。

**传输协议层：** 最常用的方案是 **SSE（Server-Sent Events）** 。HTTP 响应的 Content-Type 设为 `text/event-stream` ，服务端通过 `data: {...}\n\n` 格式逐条推送 token。相比 WebSocket，SSE 更轻量且与 HTTP 基础设施兼容。每条消息通常包含：当前 token、累计 token 数、finish\_reason（如 null / stop / length）。

**前端消费层：** 浏览器通过 `EventSource` API 或 `fetch` + ReadableStream 消费流式数据。需要处理：连接中断后的自动重连、部分 token 的拼接、用户主动中断时的 abort 信号。

**工程细节：** 反向代理（如 Nginx）需要关闭 response buffering（ `proxy_buffering off` ），否则会等到整个响应完成才转发。连接保活（keepalive）设置需要考虑生成可能持续数十秒。

### 6.2 Redis 在大模型服务中的应用

**考察意图：** 出现 4+ 次，考察对缓存设计的实际经验。

**回答框架：**

Redis 在大模型服务中的三个典型应用场景：

**语义缓存（问答对缓存）：** 将 prompt 的 embedding 作为 key（或其哈希），模型响应作为 value。核心收益是避免重复推理，将 P99 延迟从秒级降至毫秒级。关键挑战在于 **语义等价性判定** ——"北京天气怎么样" 和 "今天北京天气如何" 语义相同但文本不同，需要基于 embedding 相似度而非精确匹配做缓存命中判定。

**会话状态管理：** 用 Redis Hash 存储多轮对话的上下文。Key 为 session\_id，Field 为轮次编号，Value 为压缩后的对话内容。需要设置合理的过期时间（如 30 分钟无活动后自动清理），防止显存泄漏。

**请求队列与限流：** 用 Redis List 或 Stream 实现请求排队，用 Redis 的原子操作实现 token 级限流（如每分钟最多消费 10K tokens）。

Key 设计原则：使用 `service:model:hash(prompt)` 的层级结构；长文本 value 使用 msgpack 或 protobuf 序列化（比 JSON 更紧凑）；流式响应场景需等生成完成后再缓存完整响应。

### 6.3 高并发 LLM 服务的限流策略

**考察意图：** 考察对生产级服务治理的理解。

**回答框架：**

大模型服务的限流需要多粒度协同：

**请求级限流：** 控制每秒接受的请求数（QPS）。用令牌桶或滑动窗口实现。问题是不同请求的资源消耗差异巨大——生成 10 个 token 和 2000 个 token 的请求消耗的 GPU 时间相差百倍。

**Token 级限流：** 控制每分钟消费的 token 总数（TPM）。更公平地反映实际资源消耗。实现上需要在请求开始前估算 token 数（基于输入长度和 max\_tokens 参数），并在生成完成后修正。

**用户级限流：** 按用户或 API Key 设置独立配额，防止单用户占用过多资源。需要用 Redis 做分布式计数。

**并发级限流：** 控制同时在 GPU 上执行的请求数，直接对应显存和 KV Cache 的可用容量。这是最底层的保护——超过显存容量就会 OOM。

### 6.4 进程、线程与协程在 LLM 服务中的选择

**考察意图：** 考察对并发模型的理解。

**回答框架：**

**进程（Multiprocessing）：** 每个进程有独立的内存空间和 GIL。适合 CPU 密集型任务（如大规模 PDF 解析、embedding 批量计算）。在 LLM 推理中，通常一个进程管理一个或多个 GPU，进程间通过 IPC 或消息队列通信。

**线程（Threading）：** 共享内存空间，受 Python GIL 限制（只有 IO 等待时才能真正并行）。适合 IO 密集型的辅助任务（如日志写入、监控指标上报）。在 LLM 服务中不是主要的并发模型。

**协程（asyncio）：** 单线程内的协作式调度，适合大量 IO 等待的场景（如批量调用外部 API、等待 GPU 推理结果）。在 LLM API Gateway 层非常适合——一个请求在等待 GPU 推理时，协程可以切换去处理其他请求的预处理。

典型架构：API Gateway 用 asyncio（处理海量并发连接）→ 推理 Worker 用多进程（每进程绑定 GPU）→ 辅助 IO 用线程池。

## 七、长上下文建模

长上下文是 2026 年的热门工程方向，面试题既涉及算法层面也涉及系统层面。

### 7.1 长上下文的显存挑战

**考察意图：** 考察对长上下文工程约束的量化理解。

**回答框架：**

将上下文从 4K 扩展到 128K 时，显存增长的主要来源：

**KV Cache：** 线性增长（与序列长度成正比）。128K / 4K = 32 倍增长。这是可通过技术手段缓解的增长——PagedAttention、KV 量化、稀疏化都能有效控制。

**注意力计算的中间激活值：** 标准 Attention 中 QK^T 矩阵为 n×n，显存为 O(n²)。128K 序列的 QK^T 矩阵约 128K × 128K × 2 bytes ≈ **32 GB / 层 / 头** （不可接受）。FlashAttention 将其降至 O(n)，这是长上下文的必备技术。

**梯度和优化器状态（训练时）：** 与模型参数成正比，不随序列长度变化。这是"不变"的部分。

结论：KV Cache 是可控的线性增长，注意力激活是必须通过 FlashAttention 解决的二次增长，后者是长上下文的"本质性瓶颈"。

### 7.2 主流长序列注意力方案

**考察意图：** 考察对不同方案的权衡理解。

**回答框架：**

**Ring Attention：** 将长序列切分到多个 GPU 上，每个 GPU 处理一个子序列的 Q，通过环形通信依次接收其他 GPU 的 KV 进行注意力计算。理论上可以处理任意长度的序列（序列长度 = GPU 数 × 单 GPU 能处理的长度）。通信开销与 GPU 数线性增长。

**StreamingLLM：** 基于观察到"注意力锚点"现象——模型倾向于对序列开头的几个 token 分配高注意力，即使这些 token 语义上不重要。StreamingLLM 只保留初始 token 和最近窗口内 token 的 KV Cache，实现常数显存的"无限长度"推理。代价是丧失了对中间历史的精确访问。

**多尺度稀疏注意力：** 对近处 token 用全注意力（精细建模局部依赖），对远处 token 用稀疏注意力（粗粒度建模长程依赖）。不同层可以使用不同的稀疏模式。在建模能力和计算效率间取得平衡，但稀疏模式的设计需要领域知识。

## 2026 年推理与工程面试的三个趋势

**KV Cache 成为推理面试的"万能入口"。** 从原理到优化到显存估算，KV Cache 一个知识点就能串起整个推理优化的知识体系。面试官通过 KV Cache 能高效检验候选人对推理全链路的理解——不知道 PagedAttention 的候选人在 2026 年基本没有竞争力。

**FlashAttention 和 vLLM 从"加分项"变为"入场券"。** 这两个技术已经成为大模型推理的标准基础设施。面试官不再问"你知不知道 FlashAttention"，而是问"v2 相比 v1 改了什么"和"PagedAttention 的页大小怎么选"。

**系统工程题比例显著上升。** 流式响应实现、Redis 缓存设计、限流策略、并发模型选择——这些"传统后端"问题以大模型场景为背景重新出现在面试中。原因很直接：2026 年大模型的落地场景激增，企业需要的不只是"会调模型"的人，更需要"能把模型服务化"的人。
