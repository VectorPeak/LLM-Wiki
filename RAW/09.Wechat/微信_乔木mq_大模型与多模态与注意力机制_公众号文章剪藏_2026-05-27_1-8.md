## 0x01. 大模型算法：大模型训练的三种并行
> 发布日期：2026-03-08
> 原文链接：[大模型算法：大模型训练的三种并行](https://mp.weixin.qq.com/s/xPDZ8MRssdJoAkXAOM1Evw)
在大语言模型(LLM)的分布式训练中，单台设备无法容纳超大规模参数与批量数据，必须采用 **并行策略**将计算、通信和存储分布到多个设备上。目前主流的三种基础并行方式为：**数据并行**(Data Parallelism, DP)、**模型并行**(Model Parallelism, MP)和**流水线并行** (Pipeline Parallelism, PP)。它们在划分维度、通信模式和开销特性上各有不同，通常组合使用以实现高效扩展。

### 1. 三种并行方式详解
#### 1.1 数据并行 (Data Parallelism, DP)
- **划分方式**：每个设备持有完整的模型副本，但分配不同的训练数据子集(mini-batch 的一部分)；
- **前向/反向**：各设备独立计算梯度；
- **同步机制**：每步训练结束后，通过 **AllReduce** 操作对所有设备的梯度求平均，并同步更新模型参数；
- **典型框架**：PyTorch DDP、DeepSpeed ZeRO(DP 的增强版)。

#### 1.2 模型并行 (Model Parallelism, MP)
- **划分方式**：将模型参数(如权重矩阵)切分到多个设备，每个设备仅持有一部分模型；
- **常见形式**：
- **张量并行** (Tensor Parallelism)：将单个算子(如 Linear 层)的权重按行或列切分，计算时需设备间通信；
- **专家并行** (Expert Parallelism)：在 MoE 架构中，将不同专家分配到不同设备；
- **通信特点**：前向和反向过程中需频繁交换中间激活值和梯度，通信与计算高度交织；
- **典型框架**：Megatron-LM(张量并行)、FairSeq-MoE。

#### 1.3 流水线并行 (Pipeline Parallelism, PP)
- **划分方式**：将模型按层切分为若干“阶段”(stages)，每个设备负责连续几层；
- **执行方式**：将一个 mini-batch 划分为多个 micro-batches，以流水线方式依次通过各阶段；
- **通信内容**：设备间传递的是 **激活值**(前向)和 **梯度** (反向)；
- **关键挑战**：需协调 micro-batch 的调度以避免“气泡”(bubble，即设备空闲等待)；
- **典型框架**：GPipe、PipeDream、DeepSpeed Pipeline。

### 2. 通信开销对比分析
通信开销主要由 **通信量**(传输数据大小)和**通信频率** (每步发生次数)决定。假设：

- 模型总参数量为；
- 词表维度或隐藏层维度导致激活值大小为 (通常与 batch size 和序列长度成正比)；
- 设备数为；
- 使用 FP16(2 字节/元素)。

#### 2.1 数据并行 (DP)
- **通信内容**：全部参数的梯度；
- **通信量**： 字节(梯度为 FP16)；
- **通信频率**：每训练步 1 次 AllReduce；
- **总开销**：，与模型大小线性相关；
- **瓶颈**：当 极大(如千亿参数)，AllReduce 成为性能瓶颈。

#### 2.2 模型并行 (MP，以张量并行为例)
- **通信内容**：前向中的部分激活值、反向中的部分梯度；
- **通信量**：每层约 字节(取决于切分方式)；
- **通信频率**：每层计算后即需通信，共 次(L 为层数)；
- **总开销**：，与序列长度和 batch size 强相关；
- **特点**：通信可与计算重叠(overlap)，但对网络延迟敏感。

#### 2.3 流水线并行 (PP)
- **通信内容**：micro-batch 的激活值(前向)和梯度(反向)；
- **通信量**：每个 micro-batch 约 字节；
- **通信频率**：每个 micro-batch 在 stage 间传递一次，共 次(M 为 micro-batch 数)；
- **总开销**：，但因流水线调度，实际有效通信密度低于 MP；
- **优势**：通信总量通常小于 MP，且可与计算较好重叠。

### 3. 综合比较与工程权衡
| 并行方式 | 存储节省 | 通信类型 | 通信量级 | 适用场景 |
|:--- |:--- |:--- |:--- |:--- |
| 数据并行 | 无 | 梯度 AllReduce | 高(O(P)) | 模型可放入单卡，需大 batch |
| 模型并行 | 高 | 激活/梯度点对点 | 中高(O(L·A)) | 单层过大(如 FFN、Attention) |
| 流水线并行 | 高 | 激活/梯度流水 | 中(O(M·A)) | 模型极深，层数多 |

- **ZeRO 的改进**：DeepSpeed ZeRO 将 DP 中的优化器状态、梯度、参数进一步分片，使 DP 也能节省显存，模糊了 DP 与 MP 的界限；
- **混合并行**：实际系统(如 Megatron-DeepSpeed)通常采用 **3D 并行**：DP × MP × PP，兼顾扩展性与效率；
- **通信开销排序** (典型 LLM 场景)：
数据并行 > 模型并行 > 流水线并行
(注：此排序基于千亿模型、长序列设定；若 batch 很小，MP 可能更重)

### 4. 总结
大模型训练的三种基本并行方式从不同维度拆解计算任务：数据并行复制模型、分发数据；模型并行切分参数、协同计算；流水线并行分层处理、流水执行。它们的通信开销特性决定了各自的适用边界。现代超大规模训练几乎总是采用混合并行策略，在模型规模、硬件拓扑和网络带宽之间寻找最优平衡点。理解这三种并行的本质与通信代价，是设计高效分布式训练系统的基础。

## 0x02. 大模型算法：DeepSpeed ZeRO显存切分
> 发布日期：2026-03-08
> 原文链接：[大模型算法：DeepSpeed ZeRO显存切分](https://mp.weixin.qq.com/s/eu8rUy1tjpGy5TrxQzaRQQ)

在大语言模型(LLM)的分布式训练中， **DeepSpeed ZeRO** (Zero Redundancy Optimizer) 是一种突破显存瓶颈的核心技术，通过将优化器状态、梯度和模型参数在多个 GPU 之间进行切分(partitioning)，消除传统数据并行中的冗余存储，从而显著降低单卡显存占用。ZeRO 分为三个阶段(Stage 1/2/3)，每一阶段逐步切分更多组件，实现从百亿到万亿参数模型的高效训练。

### 1. 背景：传统数据并行的显存瓶颈
在标准数据并行(如 PyTorch DDP)中，每个 GPU 持有完整的模型副本，并独立计算前向与反向。虽然计算可并行，但以下三类数据在所有 GPU 上完全冗余：

- **模型参数** (Parameters)：FP16 权重；
- **梯度** (Gradients)：反向传播产生的梯度；
- **优化器状态** (Optimizer States)：如 Adam 中的 momentum 和 variance，通常为 FP32，占用 2× 参数量。

以 LLaMA-7B(70 亿参数)为例，仅优化器状态就需约 56 GB(7e9 × 4 bytes × 2)，远超单卡显存容量。ZeRO 的核心思想是： **这些数据无需全量驻留每张卡，只需在需要时通信聚合即可**。

### 2. ZeRO 三阶段切分机制
#### 2.1 ZeRO Stage 1：优化器状态切分
- 将优化器状态按 GPU 数量均分，每个 GPU 仅存储 1/N 的 optimizer states；
- 前向/反向仍使用完整模型；
- 更新参数时，各 GPU 负责更新自己分到的那部分参数，再通过 all-gather 同步；
- 显存节省：主要减少 optimizer states 冗余，适用于中小规模模型。

#### 2.2 ZeRO Stage 2：梯度 + 优化器状态切分
- 在 Stage 1 基础上，进一步将梯度也切分；
- 反向完成后，每个 GPU 仅保留对应参数块的梯度；
- 参数更新时，直接在本地梯度和 optimizer states 上执行，无需先聚合全部梯度；
- 显存节省：梯度和 optimizer states 均降为 1/N，是目前最常用的配置。

#### 2.3 ZeRO Stage 3：参数 + 梯度 + 优化器状态全切分
- 进一步将模型参数本身切分到各 GPU；
- 前向计算时，通过 **通信调度** 动态收集当前层所需的参数子集；
- 反向同理，在需要时 gather 参数，计算完立即释放；
- 引入 **参数持久化**(persistent parameter)和**预取** (prefetching)机制，隐藏通信延迟；
- 显存节省：模型参数、梯度、optimizer states 全部降为 1/N，支持超大规模模型。

### 3. 关键技术细节
#### 3.1 通信与计算重叠
ZeRO 通过异步通信(如 NCCL)与计算流水线，将参数 gather / gradient reduce-scatter 操作与前向/反向计算重叠，减少空闲等待。

#### 3.2 内存高效管理
使用显存池(memory pool)统一管理参数、梯度和激活值，避免频繁分配/释放导致的碎片。

#### 3.3 与模型并行协同
ZeRO 可与张量并行(Tensor Parallelism)、流水线并行(Pipeline Parallelism)组合使用。例如，Megatron-LM + ZeRO-3 可在千卡集群上训练万亿参数模型。

#### 3.4 激活值检查点 (Activation Checkpointing)
虽然 ZeRO 不直接处理激活值，但常与 gradient checkpointing 联用，进一步降低显存峰值。

### 4. 显存节省效果示例
以 100B 参数模型(FP16 权重，Adam 优化器)在 128 张 GPU 上训练为例：

| 组件 | 传统 DP (每卡) | ZeRO-2 (每卡) | ZeRO-3 (每卡) |
|:--- |:--- |:--- |:--- |
| 参数 | 200 GB | 200 GB | 1.56 GB |
| 梯度 | 200 GB | 1.56 GB | 1.56 GB |
| Optimizer States | 400 GB | 3.12 GB | 3.12 GB |
| **总计** |**800 GB**| **~4.7 GB** |**~6.2 GB** |

可见，ZeRO-3 将单卡显存需求从不可行降至可部署范围。

### 5. 挑战与权衡
#### 5.1 通信开销增加
切分越细，通信次数越多。ZeRO-3 需频繁 gather 参数，对网络带宽要求高。

#### 5.2 实现复杂度高
需精细调度通信与计算，避免死锁或性能回退。

#### 5.3 小 batch 效率低
当 micro-batch 较小时，通信延迟难以被计算掩盖。

### 6. 总结
DeepSpeed ZeRO 通过系统性地消除数据并行中的存储冗余，将大模型训练从“显存受限”转向“计算与通信受限”。其三阶段设计提供了灵活的 trade-off 空间：Stage 2 平衡效率与复杂度，广泛用于百亿模型；Stage 3 支撑万亿级训练，是当前超大规模 LLM 工程落地的关键基础设施。

## 0x03. 大模型算法：FlashAttention的IO优化
> 发布日期：2026-03-08
> 原文链接：[大模型算法：FlashAttention的IO优化](https://mp.weixin.qq.com/s/-24DhFSGaQszNVsjhcKLQQ)

在大语言模型(LLM)的训练与推理中， **注意力机制**(Attention) 是计算和内存开销的核心瓶颈。标准实现中，自注意力涉及大量中间张量(如 QK^T 点积结果、softmax 输出等)，这些张量需频繁读写高带宽显存(HBM)，导致严重的**IO 瓶颈**。**FlashAttention** 是一种 I/O 感知的注意力算法，通过重新组织计算流程，显著减少对 HBM 的访问次数和总量，从而提升速度并降低显存占用。

### 1. 标准注意力的 IO 瓶颈
标准多头自注意力的计算流程为：
1. 计算 (形状)；
2. 应用因果掩码并缩放；
3. 计算；
4. 计算输出。

其中， 和 是中间矩阵，大小为。在长序列(如 n=8192)下，仅 就需数百 MB 显存。更关键的是，这些中间结果必须写入 HBM 以供反向传播使用，造成：

- **高显存占用**：限制最大 batch size 或序列长度；
- **高带宽压力**：GPU 计算单元常因等待数据而空闲；
- **反向传播冗余读取**：需重新加载 和 以计算梯度。

### 2. FlashAttention 的核心思想
FlashAttention 的关键洞察是： **HBM 带宽远低于片上 SRAM(如 GPU 的 shared memory)带宽**。因此，应尽可能将计算“拉入”片上内存，避免中间结果进出 HBM。

其主要策略包括：

#### 2.1 分块计算 (Tiling)
将 Q、K、V 沿序列长度维度切分为小块(tiles)，每次仅加载一个 Q 块和多个 K/V 块到 SRAM；

#### 2.2 在线 softmax (Online Softmax)
在 SRAM 内累积 softmax 的归一化因子(如 max 和 sum)，无需存储完整 和；

#### 2.3 融合前向与反向
通过数学重写，使反向传播可复用前向的分块信息，避免重计算或重加载。

### 3. IO 优化的具体机制
#### 3.1 减少 HBM 写入
标准实现需将 和 写入 HBM 供反向使用；FlashAttention 在 SRAM 中完成 softmax 并直接计算，仅将最终输出写回 HBM，中间结果不落地。

#### 3.2 减少 HBM 读取
反向传播时，标准方法需重新读取 、 、Q、K、V；FlashAttention 通过保存少量统计量(如 softmax 的归一化系数)，结合重计算 K/V 块，大幅降低读取量。

#### 3.3 显存占用下降
由于无需存储 的中间矩阵，显存复杂度从 降至 (仅缓存输入和输出)。例如，在 n=2048 时，显存节省可达 5–10 倍。

### 4. 性能收益与适用场景
- **速度提升**：在 A100 上，FlashAttention 比标准 cuDNN 实现快 2–3 倍；
- **支持更长上下文**：显存节省使 8K–16K 长度训练成为可能；
- **端到端训练加速**：因注意力是 Transformer 瓶颈，整体训练吞吐显著提升；
- **兼容性好**：输出与标准注意力数学等价，可无缝替换。

### 5. 后续演进
FlashAttention 已发展至 v2 和 v3：

- **FlashAttention-2**：优化线程调度，进一步提升 GPU 利用率；
- **FlashAttention-3**：引入 FP8 支持和更激进的融合策略，适配新一代硬件(如 H100)。

此外，其思想已扩展至稀疏注意力(如 FlashSparse)、滑动窗口(如 Mistral 使用的变体)等场景。

### 6. 总结
FlashAttention 通过将计算与内存访问协同优化，从根本上缓解了注意力机制的 IO 瓶颈。它不仅是工程技巧，更是“算法-硬件协同设计”的典范：在不改变模型结构的前提下，仅通过重排计算顺序，就实现了速度与显存的双重突破。

作为当前主流 LLM 框架(如 Hugging Face、vLLM、Megatron-LM)的标准组件，FlashAttention 已成为高效大模型训练与推理的基础设施。

## 0x04. 大模型算法：vLLM的PagedAttention
> 发布日期：2026-03-08
> 原文链接：[大模型算法：vLLM的PagedAttention](https://mp.weixin.qq.com/s/c9zOAdAvdGHV1rMtzcMS8A)

在大语言模型(LLM)的高效推理系统中， **PagedAttention**是 vLLM 推理引擎提出的一项关键技术，用于优化自回归生成过程中 **KV Cache** (Key-Value Cache) 的内存管理。

核心思想借鉴操作系统中的 **虚拟内存与分页机制**，通过将 KV Cache 划分为固定大小的物理“页”，实现非连续内存分配、消除碎片、提升显存利用率，并支持高效的共享与动态扩展。

### 1. 背景：KV Cache 的内存挑战
在标准 Transformer 自回归推理中，每生成一个新 token，其对应的 Key 和 Value 向量需追加到历史 KV Cache 末尾。该缓存具有以下特性：

- 序列长度动态增长；
- 不同请求的上下文长度差异巨大；
- 总显存占用常超过模型权重本身(尤其在长上下文场景)。

传统实现通常为每个请求预分配最大长度的连续显存块。这导致两大问题：

#### 1.1 内存浪费：短请求占用大量未使用空间；
#### 1.2 内存碎片：频繁分配/释放不同大小的块，使显存无法有效复用，降低并发能力。
### 2. PagedAttention 的核心设计
PagedAttention 将 KV Cache 视为逻辑上连续、物理上分页的结构：

#### 2.1 分页机制
- 将每个请求的 KV Cache 划分为多个固定大小的“页”(如每页容纳 16 个 token 的 K/V 向量)；
- 物理页在显存中可非连续存储，类似操作系统的虚拟页表；
- 逻辑序列通过页表映射到物理页集合。

#### 2.2 动态扩展
- 当生成新 token 时，若当前页未满，直接写入；
- 若已满，则分配一个新物理页并更新页表，无需移动已有数据；
- 避免了传统 Concat 操作带来的 O(n) 内存拷贝开销。

#### 2.3 共享支持
- 若多个请求共享相同 prompt 前缀(如批量问答)，其对应页可被多个请求引用；
- 实现 KV Cache 的 **写时复制** (Copy-on-Write)，显著节省显存。

### 3. 技术优势
#### 3.1 显存利用率提升 2–4 倍
实验表明，在典型工作负载下，PagedAttention 可将 KV Cache 的内存浪费从 50%+ 降至接近 0%，大幅提高单卡并发请求数。

#### 3.2 消除内存碎片
固定页大小使内存分配器能高效复用释放的页，避免因碎片导致的 OOM(Out-of-Memory)。

#### 3.3 支持超长上下文
即使总长度远超单次分配上限，只要物理页池足够，即可动态扩展，突破传统连续分配限制。

#### 3.4 兼容现有注意力计算
PagedAttention 修改了 KV Cache 的存储布局，但通过定制 CUDA kernel，在注意力计算时自动拼接逻辑连续的 K/V 矩阵，对上层模型透明。

### 4. 实现细节
- **页表管理**：每个请求维护一个页表，记录逻辑页到物理页的映射；
- **Kernel 融合**：注意力计算 kernel 直接读取分页存储的 K/V，按需 gather 数据，避免显式拼接；
- **内存池**：预分配大块显存作为页池，页分配/回收通过轻量级指针操作完成；
- **与 FlashAttention 兼容**：PagedAttention 可与 FlashAttention 等高效注意力算法结合，进一步加速计算。

### 5. 应用场景与影响
PagedAttention 已成为 vLLM 高吞吐、低延迟推理的核心支柱，广泛应用于：

- 多租户 LLM 服务(高并发、变长请求)；
- 长上下文任务(如 32K+ 文档问答)；
- 树状推测解码(Speculative Decoding)中的分支共享。

其成功也启发了后续系统(如 SGLang、TensorRT-LLM)采用类似分页或块化缓存策略。

### 6. 总结
PagedAttention 并非新的注意力机制，而是一种 **面向系统的 KV Cache 内存管理创新**。它通过将操作系统成熟的分页思想引入 LLM 推理，巧妙解决了动态序列下的内存效率难题。

这一设计体现了“算法-系统协同优化”的典范：在不改变模型架构的前提下，仅通过底层存储与调度改进，就实现了数倍的性能提升。

## 0x05. 大模型算法：推理侧KV Cache内存管理
> 发布日期：2026-03-08
> 原文链接：[大模型算法：推理侧KV Cache内存管理](https://mp.weixin.qq.com/s/smrjjS4RRKVVVqTXxJ17HQ)

在大语言模型(LLM)的自回归推理过程中， **KV Cache** (Key-Value Cache)是提升生成效率的核心机制。它通过缓存历史 token 的 Key 和 Value 向量，避免在每一步 decode 中重复计算整个上下文的注意力，从而将时间复杂度从 O(n²) 降至 O(n)。

然而，随着上下文长度和并发请求数的增长，KV Cache 的显存占用迅速成为系统瓶颈。因此， **高效的内存管理技术** 成为现代 LLM 推理引擎的关键组成部分。

主要挑战包括：

#### 1.1 动态增长：L 随生成过程不断延长，无法预知最终大小；
#### 1.2 长度差异大：不同请求的 prompt 长度和生成长度差异显著；
#### 1.3 内存碎片：频繁分配/释放不规则大小的张量导致显存碎片；
#### 1.4 带宽压力：decode 阶段需频繁读取 KV Cache，成为 memory-bound 操作。
### 2. 主流内存管理技术
#### 2.1 预分配连续缓存 (Naive Pre-allocation)
- 为每个请求按最大支持长度(如 4K)预分配连续显存；
- 优点：实现简单，访存高效；
- 缺点：严重浪费显存(短请求占用大量未用空间)，并发能力受限。

#### 2.2 PagedAttention (vLLM 提出)
- 受操作系统虚拟内存启发，将 KV Cache 划分为固定大小的“物理页”(如 16 tokens/页)；
- 逻辑上连续的序列由非连续物理页组成，通过页表映射；
- 优势：
- 显存利用率接近 100%，消除内部碎片；
- 支持共享前缀(prefix caching)，多个请求复用相同 prompt 的页；
- 支持高效内存交换(swap to host)；
- 实测显示，在相同硬件下，vLLM 的吞吐可达 Hugging Face Transformers 的 10–20 倍。

#### 2.3 块级分配与池化 (Block-wise Allocation)
- 将显存划分为固定大小的块(block pool)，每个请求按需申请块；
- 类似 PagedAttention，但粒度更粗，适用于不支持细粒度页表的硬件；
- 需维护块分配器(如 buddy allocator)以减少外部碎片。

#### 2.4 滑动窗口缓存 (Sliding Window)
- 仅保留最近 N 个 token 的 KV Cache(如 Mistral-7B 的 4K 窗口)；
- 将内存占用从 O(L) 降至 O(N)，适用于局部依赖强的任务；
- 缺点：无法访问窗口外的历史信息，牺牲长程记忆能力。

#### 2.5 量化压缩
- 将 KV Cache 从 FP16 量化至 INT8 或 FP8；
- 显存减少 50% 以上，且对生成质量影响较小(因注意力对数值精度不敏感)；
- 需在 attention kernel 中集成反量化操作，增加计算开销。

### 3. 高级优化策略
#### 3.1 Prefix Caching (前缀缓存共享)
- 若多个请求具有相同 prompt 前缀(如系统提示词)，其 KV Cache 可共享；
- 结合 PagedAttention，只需复制页表指针，无需数据拷贝；
- 在 chat 场景中可显著提升吞吐。

#### 3.2 推测解码中的缓存复用
- 在 speculative decoding 中，多个候选序列共享公共前缀的 KV Cache；
- 仅对分歧部分分配新缓存，减少冗余存储。

#### 3.3 分层缓存架构
- 将高频访问的近期 token 存于高带宽 HBM，低频历史存于主机内存或 SSD；
- 通过预测访问模式实现智能换入换出，适用于超长上下文(>100K)。

### 4. 系统实现考量
- **Kernel 融合**：将 KV Cache 写入与 attention 计算融合，减少中间张量生命周期；
- **内存对齐**：确保 cache 块按 GPU 要求对齐(如 256-byte)，提升访存效率；
- **异步分配**：在 Prefill 阶段预分配 decode 所需页，避免 decode 时阻塞。

### 5. 总结
KV Cache 的内存管理已从简单的张量拼接演进为复杂的系统工程问题。以 PagedAttention 为代表的现代技术，通过借鉴操作系统思想，实现了显存利用与性能的双重突破。

未来方向包括更智能的缓存淘汰策略、硬件感知的分配算法，以及与稀疏注意力、状态空间模型等新架构的协同设计。

## 0x06. 大模型算法：Ovis模型

**Ovis**是由阿里巴巴通义实验室于 2024 年推出的一系列 **多模态大模型** (Multimodal Large Models)，专注于实现高性能的视觉-语言理解与生成能力。

名称“Ovis”源自拉丁语，意为“羊”，象征温顺、协作与感知力，也呼应了模型在人机交互中追求自然、可靠与共情的设计理念。Ovis 系列并非单一模型，而是一个涵盖不同规模(如 Ovis1.5-13B、Ovis1.6-72B)和能力侧重的技术体系，核心目标是在图像理解、图文生成、多模态推理等任务上达到或超越国际先进水平。

### 1. 核心架构与技术特点
#### 1.1 统一的视觉-语言编码框架
Ovis 采用“视觉编码器 + 大语言模型”的主流多模态架构：

- **视觉编码器**：基于改进的 ViT(Vision Transformer)结构，支持高分辨率输入(如 448×448 或更高)，并引入局部-全局混合注意力机制，以兼顾细节感知与整体语义；
- **语言主干**：使用通义千问(Qwen)系列的大语言模型作为文本解码器，具备强大的语言生成与推理能力；
- **跨模态对齐模块**：通过可学习的投影层(vision-to-language projector)将视觉特征映射到 LLM 的语义空间，实现端到端联合训练。

#### 1.2 高分辨率与动态视觉 token 压缩
为平衡计算效率与细粒度理解，Ovis 引入 **自适应视觉 token 选择机制**：

- 对高分辨率图像先提取密集 patch 特征；
- 再通过轻量级门控网络或聚类方法，动态压缩为固定数量(如 256–512)的代表性 visual tokens；
- 相比固定下采样，该策略保留更多关键区域信息(如文字、人脸、小物体)，显著提升 OCR、细粒度识别等任务表现。

#### 1.3 多阶段训练策略
Ovis 的训练分为三个阶段：

- **预训练**：在大规模图文对数据(如 LAION、内部清洗数据)上进行对比学习与生成式预训练，建立基础对齐；
- **有监督微调** (SFT)：使用高质量指令数据(如人工标注的 VQA、图像描述、定位问答)进行行为对齐；
- **强化学习优化** (RLHF/RLAIF)：引入多维偏好反馈(有用性、事实性、安全性)，进一步提升输出质量与可控性。

### 2. 关键能力与应用场景
#### 2.1 复杂视觉理解
- 支持多图输入、长上下文图文交错(如 PDF 解析)；
- 能准确回答涉及空间关系、属性比较、逻辑推理的问题(如“左边第二个杯子是什么颜色？”)；

#### 2.3 工具调用与智能体集成
- Ovis 可作为多模态智能体(Agent)的感知模块，解析用户上传的截图、图表或手写笔记，并触发后续操作(如调用计算器、搜索、绘图 API)；

### 3. 性能与开源情况

## 0x07. 大模型算法：多模态大模型的视觉特征传递
> 发布日期：2026-03-08
> 原文链接：[大模型算法：多模态大模型的视觉特征传递](https://mp.weixin.qq.com/s/wHM5CkeIKemY5kKC-meQ6A)

在多模态大语言模型(Multimodal LLM, MLLM)中，如何将 **视觉信息有效注入纯文本语言模型** (LLM) 是实现图像理解、图文问答、视觉推理等能力的核心问题。

由于 LLM 本身仅接受 token 序列作为输入，视觉特征必须被转换为与文本嵌入兼容的表示形式，并无缝融入其上下文处理流程。目前主流方法可分为三类： **视觉 token 化**、**特征投影对齐**和 **端到端联合建模**。

### 1. 基本流程概述
典型的多模态大模型架构包含两个主要组件：

#### 1.1 视觉编码器
(Vision Encoder)：如 ViT(Vision Transformer)、CLIP-ViT 或 SAM，用于提取图像的稠密特征；

#### 1.2 语言解码器
(Language Decoder)：如 LLaMA、Qwen 等预训练 LLM，负责生成文本。

视觉特征传递的关键步骤是： **将视觉编码器输出的特征图或 patch embeddings 转换为一组“伪 token”向量，并拼接到文本 prompt 的 embedding 序列中，作为 LLM 的输入**。

- **代表模型**：Flamingo、BLIP-2(Q-Former 前阶段)、LLaVA；
- **优点**：结构简单，保留空间布局信息；
- **缺点**：token 数量固定，长尾细节可能丢失；直接拼接可能导致模态对齐不足。

#### 2.2 可学习投影层 (Learnable Projection / Connector)
- **动机**：视觉特征空间与文本 embedding 空间分布差异大，直接拼接效果有限；
- **实现**：在视觉 token 与 LLM 之间插入一个小型神经网络(如 MLP 或 Transformer 层)，称为 **connector**或 **projector**，将视觉特征映射到 LLM 的语义空间；
- **训练策略**：通常冻结 LLM 和视觉编码器，仅微调 connector(高效且稳定)；
- **代表模型**：LLaVA(使用单层 MLP)、MiniGPT-4；
- **优势**：显著提升跨模态对齐质量，计算开销小；
- **变体**：使用 Q-Former(Querying Transformer)作为高级 connector，通过可学习 query 向量从视觉 token 中提取任务相关摘要(如 BLIP-2)。

#### 2.3 端到端联合训练 (End-to-End Fine-Tuning)
- **原理**：在大规模图文对数据上，同时微调视觉编码器、connector 和 LLM；
- **特点**：允许视觉特征与语言表示深度协同演化；
- **挑战**：计算成本高，需大量高质量图文数据；
- **代表模型**：KOSMOS-1、CogVLM、InternVL；
- **优化技巧**：
- 使用 LoRA 对 LLM 进行参数高效微调；
- 在视觉编码器后引入 pooling 或压缩模块，减少 token 数量；
- 引入位置感知机制(如 2D RoPE)保留空间坐标信息。

### 3. 关键技术考量
#### 3.1 视觉 Token 数量与分辨率权衡
- 更多 token 保留细节但增加 LLM 输入长度(影响速度与显存)；
- 实践中常采用动态压缩(如 CLIP-ViT-L/14 输出 257 tokens，经 Q-Former 压缩至 32)。

#### 3.2 位置信息保留
- 图像 patch 的空间顺序对理解至关重要；
- 解决方案：在视觉 token 上添加 2D 位置编码，或在 connector 中引入空间注意力。

#### 3.3 模态对齐损失
- 除语言建模损失外，可加入对比损失(如 image-text similarity)辅助对齐；
- 但在纯 decoder 架构中较少使用，因 LLM 本身不输出图像表示。

#### 3.4 推理效率优化
- 视觉编码可离线预计算，避免重复推理；
- 在 batch 推理中，相同图像可共享视觉特征，提升吞吐。

### 4. 典型架构对比
| 方法 | 是否可学习 | 视觉 token 数 | 训练成本 | 代表模型 |
|:--- |:--- |:--- |:--- |:--- |
| 直接拼接 | 否 | 高(~256) | 低 | 早期 Flamingo |
| MLP 投影 | 是(轻量) | 高 | 中 | LLaVA |
| Q-Former 摘要 | 是 | 低(~32) | 中 | BLIP-2 |
| 端到端联合微调 | 是 | 可变 | 高 | CogVLM |

### 5. 总结
将视觉特征传递给 LLM 的核心在于 **跨模态表示对齐**。从简单的 token 拼接，到引入可学习投影层，再到端到端联合优化，技术演进始终围绕“如何让语言模型真正‘看懂’图像”这一目标。

当前最佳实践通常采用 **冻结主干 + 轻量 connector 微调** 的范式，在效率与性能之间取得良好平衡。未来方向包括更智能的视觉 token 压缩、动态 token 选择、以及与检索增强等外部记忆机制的结合，以支持更复杂的多模态推理。

## 0x08. 大模型算法：BLIP系列的三篇工作
> 发布日期：2026-03-08
> 原文链接：[大模型算法：BLIP系列的三篇工作](https://mp.weixin.qq.com/s/vY1hnJIo6EYhJeWRsjNzWg)

BLIP(Bootstrapped Language-Image Pretraining)系列是由 Salesforce Research 提出的一组多模态大模型方法，旨在高效融合视觉与语言信息。其核心贡献不仅在于预训练策略，更在于 **如何将视觉编码器(如 ViT)与大型语言模型** (LLM)。

从 BLIP 到 BLIP-2 再到 InstructBLIP，该系列通过不断演进的 Adapter 设计，在保持 LLM 冻结的前提下，实现了更强的视觉理解、更低的训练成本和更好的指令遵循能力。

### 1. BLIP(2022)：双编码器 + 单解码器架构
BLIP 首次提出统一的图像-文本多任务预训练框架，但其架构尚未直接对接 LLM：

- **视觉编码器**：使用 ViT 提取图像特征；
- **文本编码器/解码器**：基于 BERT 改造，兼具编码(用于图文匹配)和解码(用于图像描述生成)能力；
- **连接方式**：图像特征通过线性投影后，作为额外 token 拼接到文本序列开头，输入共享的 Transformer；
- **Adapter 特点**：无显式 Adapter 模块，视觉信息直接注入主干网络；整个模型端到端训练，参数量约 140M–800M。

此时的“连接”是紧耦合的，无法复用外部 LLM，且训练成本高。

### 2. BLIP-2(2023)：引入 Q-Former 作为轻量级视觉 Adapter
BLIP-2 的关键突破是 **解耦视觉模块与语言模块**，首次实现“冻结预训练 LLM + 轻量视觉适配”的范式：

- **视觉编码器**：冻结的 ViT(如 ViT-g)，不参与训练；
- **语言模型**：冻结的 LLM(如 OPT、FlanT5)，保持原始能力；
- **核心创新：Q-Former** (Querying Transformer)
- 一个小型 Transformer，包含两组可学习 query 向量：
- **ITC Queries**：用于图像-文本对比学习；
- **ITM Queries**：用于图像-文本匹配；
- **通用 Queries**：用于向 LLM 传递视觉信息；
- Q-Former 以 ViT 输出的 patch tokens 为 key/value，query 向量与其交互，输出压缩后的视觉表征(如 32 个 token)；
- **连接机制**：Q-Former 输出的视觉 token 拼接到文本 prompt 前，作为 LLM 的输入；
- **训练成本**：仅训练 Q-Former(约 100M 参数)，ViT 和 LLM 完全冻结，训练资源需求降低 10–100 倍。

Q-Former 本质上是一个 **参数高效的视觉 Adapter**，将高维、冗余的图像特征提炼为 LLM 可理解的紧凑语义摘要。

### 3. InstructBLIP(2023)：指令微调驱动的 Adapter 优化
InstructBLIP 在 BLIP-2 基础上，聚焦 **提升指令遵循与开放域问答能力**，对 Adapter 进行了数据与训练策略层面的改进：

- **架构延续**：仍使用 Q-Former 作为视觉 Adapter，冻结 ViT 和 LLM；
- **关键变化**：

#### 3.1 高质量指令数据构建：收集并清洗 26M 图文指令对，覆盖描述、推理、知识问答等任务；
#### 3.2 两阶段训练：
- 第一阶段：在通用图文数据上继续预训练 Q-Former；
- 第二阶段：在指令数据上微调 Q-Former + LLM 的输入嵌入层(部分解冻)；

#### 3.3 Adapter 表示增强：通过指令监督信号，使 Q-Former 输出的视觉 token 更具任务导向性，能更好激活 LLM 中的相关知识；
- **效果**：在零样本 VQA、图像描述等任务上显著超越 BLIP-2，尤其在需要外部知识的场景(如“图中动物生活在哪个大洲？”)。

### 4. 三阶段 Adapter 演进总结
| 工作 | 视觉-语言连接方式 | Adapter 形式 | LLM 是否冻结 | 训练目标 |
|:--- |:--- |:--- |:--- |:--- |
| BLIP | 视觉 token 拼接，共享主干 Transformer | 无(紧耦合) | 否 | 多任务联合训练 |
| BLIP-2 | Q-Former 提取视觉摘要 → 拼接至 LLM 输入 | Q-Former(可训练) | 是 | 视觉-语言对齐 |
| InstructBLIP | 同 BLIP-2，但 Q-Former 经指令微调 | 指令优化的 Q-Former | 部分解冻 | 指令遵循 + 知识问答 |

### 5. 技术意义与影响
#### 5.1 参数效率典范：BLIP-2 证明仅需训练 <1% 的总参数(Q-Former)，即可让任意 LLM 具备视觉能力；
#### 5.2 模块化设计：视觉编码器、Adapter、LLM 三层解耦，支持灵活替换(如换 ViT-H 或 LLaMA-2)；
#### 5.3 推动多模态 LLM 发展：为后续 LLaVA、MiniGPT-4 等工作提供基础范式；
#### 5.4 降低研究门槛：无需千亿级多模态训练，即可构建高性能视觉语言系统。
### 6. 总结
BLIP 系列的演进清晰展示了多模态大模型中“连接视觉与语言”的技术路径：从早期的端到端联合训练，到 BLIP-2 引入轻量 Q-Former 作为通用视觉 Adapter，再到 InstructBLIP 通过指令微调使其具备任务感知能力。
