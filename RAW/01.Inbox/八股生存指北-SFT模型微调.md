# 1.为什么要微调? 微调 VS RAG?
## RAG与微调的区别?

微调是xxx+通过训练的方向将知识载入LLM内部

## 哪些场景需要微调?
1. ***知识灌注***  
	***微调适合让通用模型去适配垂直领域，也是微调的最大适用场景***
	比如法律、医疗、金融、工业、教育等专业领域，模型需要更精准理解专有术语、业务规则和行业表达，从而提高专业问答能力。
	
2. **微调 Embedding 模型**  
	当 RAG 检索效果不理想时，可以微调 Embedding 模型。  这类微调不是为了让生成模型“回答得更好”，而是为了让向量检索更懂某个垂直领域的语义匹配关系，例如医学术语相似度、法律条文关联、企业内部文档召回等
	
3. **微调嵌入式场景下小模型** 
	在一些嵌入式环境或移动设备上，可以对小模型进行微调。  
	这类场景通常受限于算力、内存、延迟和部署成本，微调可以让小模型在特定任务上获得更高性能，例如手机端助手、边缘设备问答、离线质检、机器人控制等。
	
4. **对话风格微调** 
	当模型需要固定语气、礼貌程度和回答方式时，适合做对话风格微调。  
	比如客服、虚拟助理、销售助手、企业内部助手等场景，需要模型保持稳定的人设、话术规范和交互风格，从而实现更个性化、更一致的对话体验
	
5. **推理能力提升** 
	微调可以优化模型对长文本的理解和逻辑推理能力。  
	适用于复杂问答、长文档分析、多步骤推理、数学题、代码推理等任务，目标是增强复杂问题的推理准确性，减少错误。
	
6. **Agent 能力提升**  
	当模型需要更好地进行多任务协作和 API 调用时，可以通过微调提升 Agent 能力。  
	例如 Function Call、工具选择、任务拆解、自动化执行、智能助手、机器人控制等场景，微调可以让模型更稳定地理解任务意图并调用合适工具。

## 哪些场景适合RAG?
1. **动态数据**
	当知识经常变化时，更适合用 RAG。比如政策法规、产品文档、企业内部知识库、新闻资讯、价格信息等。如果直接微调模型，数据一变就要重新训练或增量训练，维护成本较高；而 RAG 可以直接更新知识库，模型查询时实时检索最新内容
    
2. **降低幻觉**  
    RAG 可以让模型回答时基于检索到的外部资料，而不是完全依赖模型参数中的记忆。这样可以减少模型“编答案”的概率，尤其适合法律、医疗、金融、企业制度问答等对准确性要求高的场景。
    
3. **增强可解释性**  
    RAG 可以返回引用来源，例如文档片段、网页链接、知识库条目。用户不仅能看到答案，还能看到答案依据。这一点在审计、合规、客服质检、专业问答系统中非常重要。
    
4. **降低成本**  
    相比微调，RAG 通常不需要重新训练大模型，只需要构建向量库、检索系统和上下文拼接流程。对于中小规模知识接入、频繁更新的业务知识，RAG 的上线和维护成本通常更低。
    
5. **依赖生成能力**  
    如果任务本身仍然需要模型进行总结、改写、归纳、问答、推理式表达，那么 RAG 很适合。RAG 负责“找资料”，LLM 负责“组织语言、理解问题、生成答案”。例如企业知识库问答、合同条款解释、论文助手、客服机器人等。

## 2.2 预训练的两个挑战
大语言模型预训练的两大挑战(从模式 / 参数 / 数据三个角度进行考虑)
1. 显存效率：模型参数量太大，显存不够用
   · 即使目前显存最大的 GPU 也放不下这些大模型的模型参数。
   · 例如：175B 参数量的 GPT-3 模型参数需要占用 700 GB（175B * 4bytes）(float32-->4bytes)的显存。参数梯度是 700GB，Adam 优化器状态需要 1400 GB(其实在梯度这块,使用float16-->2bytes的精度就可以了,不需要用到float32的精度,梯度本身不见得需要这么精确)，共计需要 2.8 TB 的显存

2. 计算效率：训练数据量多，模型参数量大，计算量大，单机训练时间久
   · 即使我们能将大模型放在一张 GPU 上，训练大模型需要的海量计算操作需要耗费很长时间。例如：用英伟达 A100 显卡训练 175B 参数量的 GPT-3 模型大约需要 288 年。

## 2.6 预训练-3D并行
**3D并行是由数据并行（DP）、张量并行（TP）和流水线并行（PP）组成**
DP: Data Parallelism
TP: Tensor Parallelism
PP: Pipeline Parallelism

**数据并行**：计算效率高、实现简单。
- 显存效率：每张卡上都保存了完整的模型、梯度、优化器状态，因此显存效率不高。
- 计算效率：当增加并行度时，单卡的计算量是保持恒定的，可以实现近乎完美的线性扩展。但规约梯度的通信开销，与模型大小成正相关。

**张量并行**：因模型结构而异，实现难度大。
- 显存效率：随着并行度增加，成比例地减少显存占用。是减少单层神经网络中间激活的唯一方法。
- 计算效率：频繁的通信，限制了两个通信阶段之间的计算量，影响了计算效率，计算效率很低。

**流水线并行**：通信成本最低。
- 显存效率：减少的显存与流水线并行度成正比。但流水线并行不会减少每层中间激活的显存占用。
- 计算效率：成本更低的点对点（P2P）通信。通信量与流水线各个阶段边界的激活值大小成正比。

**3D并行实例:**
Bloom-176B 进行预训练时，在 394 张 NVIDIA A100 80GB GPU（48 个节点）上使用了 3D 并行（数据并行、流水线并行、张量并行）策略，针对 350B 个 Token 训练了大约 3.5 个月

# 3.SFT-微调(Supervised Fine-Tuning)
SFT，全称是 **Supervised Fine-Tuning**，中文通常叫 **监督微调**
它是在一个已经完成预训练的语言模型基础上，继续使用人工标注好的 **输入-输出样本** 进行训练，进一步优化模型的表现，以适应特定任务或场景的需求

不同于 RAG 或者 Agent 技术，通过搭建工作流来优化模型表现，微调是通过修改模型参数来优化模型能力，**是一种能够让模型“永久”掌握某种能力的方法**

## 3.1 微调的步骤
1 确定任务目标
2 准备数据       
3 选择预训练模型
4 设置微调环境
5 配置微调参数 
6 训练模型       
7 评估与验证     
8 部署与应用

**1.1 确定任务目标**
	**目标**：明确微调的具体任务和需求
	示例：
	任务类型：文本分类、问答、文本生成等 (格式,数据量都不一样的)
	领域：医疗、法律、金融等
	输出：任务定义文档
	
**1.2 准备数据**
	目标：收集和预处理任务相关数据
	子步骤：
	**数据收集**：
	收集与任务相关的标注数据（如分类标签、问答对）
	途径:开源数据集 / 购买数据集 / 业务部门收集数据集(或自己通过LLM进行构建)  
	示例：医疗问答数据、法律文书数据
	**数据清洗**：
	去除噪声数据（如重复、错误标注）
	**数据划分**：
	将数据划分为训练集、验证集和测试集（如 80%/10%/10%）
	**数据格式化**：
	将数据转换为模型输入格式（如 JSON、CSV）
	输出：高质量的训练数据
	
**1.3 选择预训练模型**
	目标：选择适合任务的预训练模型
	考虑因素：
	模型规模（如参数量）
	模型架构（如 Transformer）
	预训练数据（如通用文本、领域文本）
	示例：
	通用模型：GPT-4、Qwen-Base
	领域模型：BioBERT（生物医学）、Legal-BERT（法律）
	输出：预训练模型文件
	
**1.4 设置微调环境**
	目标：配置硬件和软件环境
	子步骤：
	硬件配置： 
	GPU/TPU：确保有足够的计算资源
	软件配置：
	深度学习框架：如 PyTorch
	模型库：如 Hugging Face Transformers
	依赖安装：
	安装必要的库和工具
	输出：配置好的微调环境
	
**1.5 配置微调参数**
	目标：设置微调的超参数和训练策略。
	关键参数：
	学习率：控制参数更新的步长。
	批量大小（Batch Size）：每次训练的样本数。
	训练轮数（Epochs）：完整遍历训练数据的次数。
	优化器：如 AdamW、SGD。
	输出：微调配置文件。
	
**1.6 训练模型**
	目标：在任务数据上微调模型
	子步骤：
	加载预训练模型：
	加载预训练模型的权重
	加载数据：
	将训练数据加载到模型中
	训练循环：
	前向传播：计算模型输出
	计算损失：比较输出与真实标签
	反向传播：更新模型参数
	保存模型：
	定期保存微调后的模型
	输出：微调后的模型文件
	**通过诸如PEFT库写代码 / LLama factory  等工具 进行操作**
	
**1.7 评估与验证**
	目标：评估微调模型的性能
	子步骤：
	验证集评估：
	使用验证集评估模型性能
	测试集评估：
	使用测试集评估模型的泛化能力
	错误分析：
	分析模型在哪些样本上表现不佳
	调整参数：
	根据评估结果调整超参数
	输出：模型性能报告
	
**1.8 部署与应用**
	目标：将微调后的模型部署到实际场景中
	子步骤：
	模型导出：
	将模型导出为可部署格式
	部署环境配置：
	配置服务器或云环境
	接口开发：
	开发 API 或用户界面
	监控与维护：
	监控模型性能，定期更新模型
	输出：部署好的微调模型
	
## 3.2 微调方案
### 微调方案-全量微调 / 高效微调
而从微调方案的大类上来划分，微调又可以划分为**全量微调**：带入全部数据进行微调，和**高效微调(PEFT: Parameter-Efficient Fine-Tuning)**：只带入部分数据进行微调。毫无疑问，全量微调是一种**算力消耗更大**、但对模型的能力改造更为彻底的方法，而高效微调则更类似一种“四两拨千斤”的方法，通过修改模型部分参数，来调整模型整体能力

### PEFT-高效微调(Parameter-Efficient Fine-Tuning)
在全量微调成本较高的背景下，PEFT 的核心思路是：冻结大部分预训练模型参数，只训练少量新增参数或少部分原有参数，从而以更低成本完成任务适配

PEFT（高效微调）不是某一种单独算法，而是一类“尽量少更新参数、尽量少占显存、尽量保留底座模型能力”的微调方法集合，常见技术大致可以按参数改动方式分为三类：增加额外参数、选择部分参数更新、重参数化
~~~
高效微调 PEFT
├─ 1. 增加额外参数 Additive
│  ├─ Adapter Tuning
│  ├─ Prefix Tuning
│  ├─ Prompt Tuning
│  ├─ P-Tuning
│  └─ IA³ 等
│
├─ 2. 选择部分参数更新 Selective
│  ├─ BitFit
│  ├─ LayerNorm Tuning
│  ├─ Diff-Pruning
│  └─ 只更新部分层 / 部分权重
│
└─ 3. 重参数化 Reparameterization-based
   ├─ LoRA
   ├─ AdaLoRA
   ├─ LoHa / LoKr
   ├─ Compacter
   └─ Intrinsic-SAID 等
~~~
**A:增加额外参数**:这种方法是在原有的预训练模型的基础上增加一些额外的参数
**S:选取一部分参数更新**: 这种方法是在微调过程中只更新模型的一部分参数，而不是所有参数。这可以减少计算量，提高微调效率
**R:引入重参数化**:这种方法是在微调过程中只更新模型的一部分参数，而不是所有参数。这可以减少计算量，提高微调效率

常见的参数高效微调技术有 Prefix Tuning、Prompt Tuning、P-Tuning、Adapter Tuning、LoRA 等

## 3.3 LoRA微调 (重要)
> **LoRA 相比全量微调是工程首选**，训练时间缩短 50%~70%，显存节省 60%~80%，**效果损失通常在 1%~3% 以内**（不好说）
### LoRA原理
#### LoRA核心原理
假设模型有 $W$ 这个权重矩阵：
	$W \in \mathbb{R}^{d \times k}$
全量微调会直接更新：
	$W \leftarrow W + \Delta W$
而 LoRA 的核心想法是：
> 原始大模型不动，只训练一个很小的“补丁矩阵”


在线性层里，原始计算是：
	$y = Wx$
LoRA 不直接改 $W$，而是冻结 $W$，额外加一个可训练的低秩更新：
	$y = Wx + \Delta W x$ 
关键是，LoRA 不直接训练完整的 $\Delta W$，而是把它拆成两个小矩阵：
	$\Delta W = BA$
其中：
	$B \in \mathbb{R}^{d \times r}$     $A \in \mathbb{R}^{r \times k}$   
所以：
	$y = Wx + BAx$
这里的 $r$ 是 **rank: 秩**，通常很小，比如 4、8、16、32
![image.png](https://img.vectorpeak.cn/obsidian/2026/05-06/20260517210449985.png?imageSlim)

#### LoRA 为什么参数少？
如果原始权重矩阵是：
	 $W \in \mathbb{R}^{4096 \times 4096}$
完整微调一个 $\Delta W$，需要训练：
	$4096 \times 4096 = 16,777,216$   约 1677 万个参数
如果用 LoRA，设 $r=8$  (LoRA的低秩分解)，则只需要训练：
	$B: 4096 \times 8$   $A: 8 \times 4096$     
总参数：$8 \times 4096 + 4096 \times 8 = 65,536$
**从 1677 万降到 6.5 万，参数量大幅下降，这就是 LoRA 的核心收益**

#### LoRA 通常插在哪里？
在 Transformer 里，LoRA 常插入注意力层的线性投影矩阵，例如：
- Query projection：$W_q$
- Key projection：$W_k$
- Value projection：$W_v$
- Output projection：$W_o$

很多 LLM 微调实践里，常见配置是只对：$W_q, W_v$ 或者：$W_q, W_k, W_v, W_o$ 加LoRA

## 3.4 QLoRA微调
### QLoRA核心贡献
**QLoRA 可以理解成 LoRA + 4bit 量化, 一种更省显存的大模型微调方法**

**QLoRA做了这三件事, 用来减小模型的内存占用:**
**4bit NormalFloat（NF4）**：对于正态分布权重而言，一种信息理论上最优的新数据类型，该数据类型对正态分布数据产生比 4 bit 整数和 4bit 浮点数更好的实证结果

**双量化**：对第一次量化后的那些常量再进行一次量化，减少存储空间

**分页优化器**：使用 NVIDIA 统一内存特性，该特性可以在 GPU 偶尔 OOM 的情况下，进行 CPU 和 GPU 之间自动分页到分页的传输，以实现无错误的 GPU 处理。该功能的工作方式类似于 CPU 内存和磁盘之间的常规内存分页。使用此功能为优化器状态（Optimizer）分配分页内存，然后在 GPU 内存不足时将其自动卸载到 CPU 内存，并在优化器更新步骤需要时将其加载回 GPU 内存
![image.png](https://img.vectorpeak.cn/obsidian/2026/05-06/20260517200111515.png?imageSlim)


### 4bit NormalFloat(NF4)
4bit 的含义是：每个数只能用 4 个二进制位表示，因此最多只能有$2^4 = 16$个可选档位。**量化的本质，就是把连续的浮点数压缩到有限的离散档位中**

#### 4bit线性量化
**4bit 线性量化**就是把原本连续的浮点数，用 **4 个二进制位** 表示成有限的是 **等间距** 整数数档位. 优点是可以节省内存，模型权重更小，显存压力更低.  原来 FP16 一个数 16bit，经过4bit线性量化后只需要4bit，理论上存储可降到约四分之一.  缺点是**容易被抹平细微差异**：相近的数可能被量化成同一个值
![image.png](https://img.vectorpeak.cn/obsidian/2026/05-06/20260517204509552.png?imageSlim)


#### NF4
NF4 是 QLoRA 中使用的一种 4bit NormalFloat 类型。它不是均匀刻度，而是为近似正态分布的权重设计的一组非均匀档位

## RLHF(Reinforcement Learning from Human Feedback)

传统 RLHF 大致是三步：
1. 先做 SFT：让模型学会基本回答格式。
2. 再训练 Reward Model：给同一个问题的多个回答打分，判断哪个更符合人类偏好。
3. 最后用 PPO 等强化学习算法优化模型，让模型生成高奖励回答。


### RM(Reward Model-奖励模型)
 把“人类觉得哪个回答更好”转化成模型可以优化的一个标量奖励分数



## DPO(Direct Preference Optimization-直接偏好优化)
### DPO要解决什么问题?
传统 RLHF 大致是三步：
1. 先做 SFT：让模型学会基本回答格式。
2. 再训练 Reward Model：给同一个问题的多个回答打分，判断哪个更符合人类偏好。
3. 最后用 PPO 等强化学习算法优化模型，让模型生成高奖励回答。

问题是：这套流程重。奖励模型要单独训练，PPO 训练不稳定，调参复杂，算力成本也高

DPO 的核心思想是：既然训练数据本身已经告诉模型“哪个回答更好”，就没有必要再单独训练奖励模型，也不必使用复杂的强化学习流程。它直接利用偏好数据来优化语言模型。

偏好数据通常写成：



其中：

- $\pi_\theta$ 表示正在训练的新模型；
- $\pi_{ref}$ 表示参考模型，通常是冻结的 SFT 模型；
- $y_w$ 表示人类更偏好的回答；
- $y_l$ 表示人类较不偏好的回答；
- $\sigma$ 是 sigmoid 函数；
- $\beta$ 控制偏好优化的强度。

这个公式的直观含义是：

$$
\log \frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)}
$$

应该变大，表示新模型相对于参考模型，更偏向优胜回答。

同时：

$$
\log \frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)}
$$

应该相对变小，表示新模型不要过度偏向失败回答。

因此，DPO 本质上是在拉开优胜回答和失败回答之间的概率差距。

### 为什么需要参考模型

参考模型 $\pi_{ref}$ 的作用类似一根“安全绳”。

如果只让模型不断提高优胜回答的概率、降低失败回答的概率，模型可能会过拟合偏好数据，甚至破坏原本的语言能力。参考模型可以约束新模型不要偏离原来的 SFT 模型太远。

这和 RLHF 中的 KL 约束思想接近：

$$
D_{KL}(\pi_\theta || \pi_{ref})
$$

也就是让模型朝人类偏好方向移动，但不要移动得过猛。

### $\beta$ 的作用

$\beta$ 可以理解为偏好强化力度。

- $\beta$ 较大时，模型会更激进地拉开优胜回答和失败回答的差距；
- $\beta$ 较小时，模型会更保守，更接近参考模型。

类比来说，$\beta$ 像调味料的用量。太少，模型对齐效果不明显；太多，模型可能变得不稳定，甚至损伤原有能力。

### DPO 的优势

DPO 的主要优势在于流程更简单：

- 不需要单独训练奖励模型；
- 不需要 PPO 这类复杂强化学习算法；
- 训练流程更稳定；
- 数据需求和算力成本通常更低；
- 可以直接使用成对偏好数据进行优化。

不过，“训练速度提升 3 倍”这类说法需要结合具体实验设置来看。DPO 通常比 PPO-style RLHF 更轻量，但具体加速倍数会受到模型规模、数据规模、训练框架和硬件条件影响。
