## 1.论文的重要性?
在大模型这个行业里，和传统开发不一样，即使是开发岗，也需要读论文的。我从三个点来说明为什么：

### 面试确实会问
这点你从我的面试记录中可以看出来，很多公司是直接会问你最近读过什么论文的。这点在传统开发领域几乎不会出现，而在大模型方向，即使是应用岗，该问题出现的频率也很高。从清华大学出版社*大模型工程师面试 - 算法原理、开发实践与系统部署*中也讲了，如果以 3 个维度（刷题，论文和项目）来衡量大模型岗位准备的优先级。推荐顺序为：

1. **校招/初级算法岗**：刷题 > 项目 ≈ 论文
2. **高级算法岗**：论文 > 项目 ≈ 刷题
3. **Agent/Rag 方向开发**：项目 ≈ 论文 > 刷题

算法岗就不说了，肯定要看论文的。但是对于开发岗，书上明确将论文和项目放在了同一等级。至于这个优先级排序，我觉得我们也不用较真，但是可以看出的是，书中明确将论文作为大模型开发方向的一个重要准备方向。和我的个人实践一样，确实是需要准备的

### 论文确实有用
大模型这个行业发展很快。即使是 Agent，有人称 2025 年是 Agent 元年，现在 2026 年，Agent 发展也不过一年。行业中以什么样的形式组织 Agent，什么样的 Agent 框架最好，怎么评估 Agent 这些工程方面的东西，都还在研究和讨论。书和资料更多的是一些基础知识，但是论文上的知识，有的更加先进，代表未来的方向，会给我们不一样的其实。看了很多论文，对我们工程也确实有帮助。比如有的论文写如何选 Agent 架构，项目的 Agents.md 如何组织最好这些东西对我们工程最佳实践也有帮助

### 论文能够提供新的思考框架
另外一点是，在我们面试的时候，论文给了我们很多不一样的启发,可以改变我们看问题的角度:
比如 Agent 的架构，如果你看了谷歌的*Towards a Science of Scaling Agent System*，在回答 Agent 架构，你会有不一样的思维，回答会更出彩。
如果你看了*Are Repository-Level Context Files Helpful for Coding Agents?*，你在回答有哪些写组织提示词的最佳技巧这种问题时，会有新的启发。所以论文对于我们面试问题，确实能带来实打实的帮助。我的亲身经历是，你在面试的时候，如果问题能结合论文回答，回答出一些不一样的东西，甚至面试官会向你请教，面试变得十拿九稳

很多时候，不一定非要把论文从头到尾完整读完。只要能把它的思想融入到面试回答中，就已经很有价值

#### 3. 把论文和面试问题关联起来
前面整理的很多面试问题，可能会引用到相关论文. 当你在某个回答中看到论文引用时，可以回到这里查对应论文的总结，把它当作附录或索引，用来补充背景和细节

#### 4. 一定要准备几篇自己能讲清楚的论文
一定要提前准备几篇自己能够讲清楚的论文

即使没有完整读过原文，也建议至少看一遍总结，知道它解决了什么问题、用了什么方法、带来了什么启发

这样当面试官问：
- “你最近读了什么论文？”
- “这篇论文给你什么启发？”
- “它和你的项目有什么关系？”

你就可以从这个专题里整理出自己的回答，而不是临场硬想

### Agent
#### Evaluating AGENTS.md
**原文**
[Evaluating AGENTS.md - Are Repository-Level Context Files Helpful for Coding Agents?](https://arxiv.org/abs/2602.11988)

**摘要**
越来越多的开发者/公司为代码仓库写一个 AGENTS.md（或 CLAUDE.md）文件，相当于一份“仓库说明书”，告诉 AI 编程智能体：项目结构概览，用哪些工具（如 uv、ruff）、遵守哪些代码风格、如何运行测试等。OpenAI、Anthropic 等厂商强烈推荐这个做法。截至论文写作时（202602），GitHub 上已有超过 6 万个开源仓库包含此类文件。本篇论文就是在研究，这些东西到底对 Agents 有没有用？是写的越详细越好吗？有没有可能反而会带来负面影响？


**实验方法**
论文构建了一个新基准 AGENTBENCH（共 138 个真实 GitHub Issue 任务），选自 12 个小众但有开发者提交的 context file 的 Python 仓库，补充已有的 SWE-bench Lite（300 个任务，这个是目前评测 AI 编程智能体能力的一套测试集）

对 4 个主流 Agent（Claude Code / Codex / Qwen Code）测试了三种设置：
1. None: 不提供任何 Context File（AGENTS.md CLAUDE.md）
2. LLM: 按照厂商推荐的指令，提供使用 AI 自动生成的 Context File
3. Human: 提供开发者手写的 Context File

测试方法是给出对每个 PR，给出一个测试用例：<问题描述，PR 前的代码仓库，测试集，正确答案>（这个测试集和方式也是 SWE-bench Lite 的测试集形式，所以通过这个问题你可以学习了解典型的一种测试 AI Coding 能力的数据集），让 AI 完成这个 PR。比较 AI 完成后的问题的：任务完成率，成本（总 Token 消耗），行为分析（工具调用次数，是否遵循了 Context File，有没有运行更多的上下文搜索，做更多的测试等）


**核心发现**
1. LLM 自动生成的 context file 反而有害。与不提供任何文件相比，LLM 生成版本平均使任务成功率下降约 3%，而且推理成本（token 消耗）增加超过 20%，换用更强的模型（如 GPT-5.2）来生成也没有一致改善

2. 人类手写的 context file 帮助极为有限。仅带来平均约 +4% 的成功率提升，远低于人们预期

3. Agent 确实读取并执行了 context file 中的指令。比如 context file 提到使用 uv，Agent 就会用 uv，说明效果差不是因为 Agent 忽略了说明书，而是说明书本身加重了任务难度

4. Context file 导致 Agent 行为更“臃肿”。Agent 运行更多测试、查看更多文件、写更多文件。GPT-5.2 的推理 token 增加了 22%，说明 Agent 认为任务“变难了”

**核心假设：** context file 中包含了大量“非必要要求”，这些额外约束反而让 Agent 在完成核心任务时分心、出错，相当于给 Agent “加了很多不必要的作业”。



**对面试的帮助**
**Agent 评估：** 看完了本论文，其实你又可以进一步理解 Agents 评估。比如本文对 Coding Agents 的评估维度是：正确率、轨迹路径（调用工具次数，是否遵循了指令，是否阅读了更上下文，做了更多次数），成本（总 Token 消耗，推理 Token 消耗）

**Agent 测试集：** 业内常见的一种对 Coding Agents 评估的测试集：SWE-bench Lite，它是什么，它的思想是什么？

**提示词工程：** 什么该写，什么不该写？为什么有时写多了反而效果变差了。我以阿里夸克千问一面的面试问题（请搜关键词去看对应部分）。在我读完这个论文，总结了这个笔记后，我对于当时面试官的问题：如何做提示词的裁剪压缩，就有了一些新的理解，也去扩展了相关答案。如果现在我去面试，我就会说，我看了 xx 的论文，他说了 xx，所以我们应该 xx。所以其实读完，立刻显现出了它对我们面试带来的收益。

##### 1.背景知识 - ResNet 残差网络
因为这个论文是对残差网络进行了改进，所以我们需要先了解一下残差网络是什么。我们不讲深入，因为这个不是深度学习的章节，尽量让大家都懂，如果你看不懂，可以结合着 AI，搜一下资料理解一下

**1.1 传统残差连接概念**
2015 年由微软研究员何恺明提出，***核心思想极其简单：对任意层的函数变换，无条件地将原始输入加回输出***

直观效果是不管中间的函数 F 做了什么变换，输出至少保留了输入的原始信息，从而极大缓解了梯度消失问题。网络不再学习完整的映射，而是只学习残差（差值），数学直觉告诉我们这里容易学习

这个结构从原始 Transformer 架构至今基本没有变过——Transformer 中每个子层（Multi-Head Attention、Feed Forward）后面的 "Add & Norm" 就是残差连接的体现。（以下这个如雷贯耳的 Transformer 架构图，我们可以看到中间的 Add & Norm 就是残差连接）
![image.png](https://img.vectorpeak.cn/obsidian/2026/05-06/20260510142246922.png?imageSlim)

**1.2 ResNet 解决了什么问题?**
训练深层神经网络存在两大致命问题：梯度消失和梯度爆炸

**梯度消失:** 也可以称为信息稀释，指的是反向传播时，梯度在一层层传回前面网络的过程中不断变小。结果是，靠近输入端的早期层几乎得不到有效更新，模型很难学到低层特征。这就像一句话传了几十个人之后，最后已经听不清最初的信息了

**梯度爆炸:** 也可以称为隐藏状态爆炸, 梯度在反向传播过程中不断放大，导致参数更新幅度过大，训练过程剧烈震荡，甚至直接崩溃。这就像水管中的水压不断升高，最终把管道撑爆。在千亿级参数的大模型训练中，这表现为梯度分布极不均匀，模型极度不稳定，甚至直接崩溃，几乎成了限制大模型往更深层发展的“物理魔咒

残差连接的出现，正是为了缓解深层网络难以训练的问题。它通过一条“捷径”把输入直接加到输出上：
$$
y = F(x) + x
$$
这样一来，信息和梯度都可以沿着这条捷径更顺畅地传播。即使中间层暂时学不好，模型也至少可以保留原始输入，不至于让信息在层层变换中完全丢失


不过，传统残差连接虽然解决了深层网络可训练性问题，但在超大规模 Transformer 中，它也可能带来新的限制：每一层都只是把新信息加回同一条 residual stream，长期堆叠后可能出现信息混杂、表达受限或训练动态不理想的问题。这也是*Attention Residuals*这类论文重新思考残差结构的原因

##### 2.Kimi 的核心创新：Attention Residuals 注意力残差
**2.1 灵感来源——时间与深度的对偶性**

论文中最优美的洞察在于发现了一个对偶关系：

在时间维度（序列建模）上，旧方法是 RNN/LSTM 的逐词累加传递，新方法则是 Transformer 通过注意力机制进行加权求和——这一演进已被证明极其有效。

而在深度维度（层间信息传递）上，情况惊人地相似：旧方法是传统残差的逐层累加传递，Kimi 提出的新方法则是注意力残差——用注意力机制对所有前序层进行加权聚合。同一套“用注意力替代逐步累加”的升级逻辑，在两个不同维度上都产生了巨大的效果提升。


正如论文原文所写：

“时间与深度的对偶性——就像 RNN 在时间维度上的作用一样，残差连接在深度维度上将所有先验信息压成一个单一状态。在序列建模中，Transformer 通过用注意力机制取代循环机制从而改进了 RNN……我们在深度上提出了同样的方法。”

Karpathy 的评价精准概括了这一思想：

“LSTM 就是将 residual 旋转 90° 得到的。事实证明，注意力机制同样是可以旋转 90° 的。我们是不是没有充分理解 ‘Attention is All You Need’？”

（大家理解一下这个旋转90度的概念，其实就是我们把时间上的处理作为横轴，我们引入注意力机制。现在是在网络结构的深度上，我们做手脚，也是引入了类似注意力机制的能力，大家把这个叫做注意力机制的旋转90度）
![image.png](https://img.vectorpeak.cn/obsidian/2026/05-06/20260510142341457.png?imageSlim)


**2.2 方法详解**
在传统 Transformer 中，残差连接通常采用逐层递推的累加方式：
$$
x_{l+1}=x_l+F_l(x_l)
$$
这意味着第 $l+1$ 层主要基于第 $l$ 层的状态继续计算。虽然 $x_l$ 中已经包含了前面层的信息，但这些信息通常以固定方式被不断累加，模型本身缺少一种显式机制来判断：当前层到底应该更多依赖哪一层的表示

Attention Residuals 的核心思路，是把“注意力机制”引入到深度维度上。也就是说，每一层不再只是被动接收上一层传来的累积结果，而是可以主动回看前面若干层的表示，并根据当前输入动态分配权重

可以把它理解为：传统残差连接更像是把前面所有信息一路打包传下来；而 Attention Residuals 则像是在每一层都增加了一个“检索器”，让模型能够从历史层中选择当前最有用的信息

例如，较深的第 50 层在处理某个 token 时，可能发现第 2 层保留的浅层特征仍然非常关键，于是为第 2 层分配更高权重，从而更有效地利用早期表示，缓解信息在深层网络中被稀释的问题
传统残差中，第 $x_l$ 层只能接收第 $l$ 层的输出并简单相加。注意力残差则让每一层拥有“回头看”的能力

• 每一层会发出“寻物启事”，到前面所有层中寻找相关特征
• 通过注意力机制为不同层分配不同权重，选择性地聚合最有用的信息
• 例如：第 50 层可以给第 2 层分配 0.8 的超高权重，越级提取最原始特征，彻底解决信息稀释问题


**2.3 分块策略——解决计算开销**
如果每一层都对前面所有层进行全局注意力计算，通信量和显存开销会迅速增加。为此，Kimi 团队提出了分块策略

如果每一层都对前面所有层做全局注意力，通信量会瞬间爆炸。Kimi 团队设计了分块策略

**具体来说:**
- 模型会被划分为若干个区块-(例如每 8 层作为一个 block)
- **区内**: 即在区块内部，仍然采用类似传统残差连接的逐层累加方式
- **跨区**: 在区块之间，引入全局注意力式的筛选与调度

深层网络因此可以跨区块选择更有价值的历史表示，而不是机械地接收所有累积信息

这种设计在表达能力和计算开销之间取得了折中：既保留了 Attention Residuals 对历史层进行选择性聚合的优势，又避免了对所有层做全局注意力带来的巨大通信成本

根据官方仓库介绍，Block AttnRes 在推理延迟增加不到 2% 的情况下，可以达到与使用 1.25 倍训练计算量的 baseline 相近的 loss。这说明它并不是单纯增加复杂度，而是在更高效地利用已有计算


**2.4 结果**
Kimi 团队在 48B 参数的模型上验证了注意力残差的有效性，结果非常亮眼。在训练效率方面，相比传统残差提升了 1.25 倍，达到同等性能时节约近 20% 的计算量。在能力评测方面，考验逻辑能力的 GPQA 多步推理测试中性能提升了 7.5 分，代码能力也大幅提升。而这些收益的代价极小——推理延迟仅增加不到 2%

对 AI 公司而言，这意味着可直接节省数千万美金的训练成本，训练周期缩短数周到数月，超大规模模型能稳定收敛

**2.5 行业启示**
**回归第一性原理，敢于质疑基础**
残差连接这一沿用十年的“老祖宗”，由于过于基础，渐渐形成了“它就该这样”的固有思维。Kimi 论文证明：最基础的部分往往也是最容易被忽略的部分，**越是“理所当然”的东西，越值得重新审视。越是基础、稳定、被广泛使用的结构，越可能隐藏着长期未被重新审视的改进空间**. 创新不仅在上层应用和花哨技巧上，动地基往往能带来颠覆性突破

**注意力机制已在时间维度（序列建模）上被验证有效，Kimi 团队的洞察在于将同一原理迁移到深度维度**。这种跨领域迁移能力——发现不同场景下公式/结构的相似性——目前仍然是人类独有的直觉优势。正如晓辉博士所评论：“如果你没有对数学公式之间关系的直觉，就没有这样一次突破的机会”

**Q4：能简单介绍一下 Kimi 注意力残差的具体机制吗？**

传统残差连接中，每一层的输出等于这一层变换结果加上这一层输入，信息只能逐层往后传

注意力残差把这个过程改成：每一层生成一组 Query，去和前面所有层的输出做注意力计算，再按相关性分配权重后加权聚合。

这样第 50 层可以直接给第 2 层一个很高的权重，越级提取早期特征。为了控制计算量，论文还用了分块策略：每若干层分成一个区，区内正常累加，跨区才做全局注意力，因此内存开销大幅下降

**最大的启发是“跨维度迁移”这种思维方式**

注意力机制在时间维度上替代 RNN，已经是十年前的事情了。但十年来没人想到把同样的思想用到深度维度上。Kimi 团队发现这两个方向的数学结构几乎一模一样，都是把逐步累加替换为全局加权聚合，然后就做出了突破

这说明很多时候创新不是凭空发明全新的东西，而是发现已有方法在新场景下的适用性。对我自己的工作来说，也提醒我要多关注不同领域之间结构上的相似性，而不是只在一个方向上深挖

**CAREFUL论文应该包装为:**
**CAREFUL：面向可信 AI 数据治理的数据删除通知分发机制**
它本身并不是直接改进模型结构、损失函数或训练策略的算法论文，而是解决 AI 系统在使用用户数据时必须面对的一个前置问题：当用户要求删除个人数据时，系统如何沿着数据流转路径，安全、完整、隐私保护地通知所有相关的数据处理节点

在大模型、推荐系统和广告系统中，用户数据往往不会只停留在一个数据库里，而是会流向特征工程、训练样本构建、日志系统、画像系统、RAG 知识库、模型评估集等多个环节。此时，删除请求不能只作用于源数据库，还必须同步触达所有使用过该数据的下游模块。否则，即使数据库中的原始记录被删除，AI 系统仍然可能继续在训练样本、特征缓存或模型更新流程中使用这些数据

从这个角度看，CAREFUL 研究的“删除通知分发”可以理解为 AI 数据生命周期中的合规传播问题。它把用户删除请求建模为数据流转图上的安全通知过程：节点可以对应数据控制者、AI 数据处理模块或下游业务系统，边则代表数据共享与处理关系。删除通知需要沿着历史数据流转路径传播，同时避免被篡改、丢弃或泄露完整流转路径


**TrustNotify论文应该包装为:**
**TrustNotify：面向可信 AI 数据治理的轻量级删除通知验证框架**
它本身并不是直接优化模型结构、损失函数或训练策略的 AI 算法论文，而是解决 AI 系统在数据生命周期管理中的一个关键工程问题：当用户数据已经进入训练样本、特征工程、日志系统、推荐画像、RAG 知识库或模型评估集之后，系统如何以低成本、可信、可验证的方式通知所有相关模块执行删除

如果说 CAREFUL 更强调“删除通知如何沿着数据流转路径安全传播，并尽量保护路径隐私”，那么 TrustNotify 更强调“删除通知是否完整送达、是否可信执行、是否能够低成本验证”。它关注的不只是通知发出去，而是要确认通知确实到达了所有应该处理该数据的节点，并且能够通过回执机制证明这一过程

从 AI 方向来看，TrustNotify 可以被理解为 AI 数据治理中的可信通知与回执验证机制。它把用户删除请求建模为数据流转图上的通知传播问题：节点可以对应训练数据仓库、特征平台、日志系统、RAG 文档库、模型评估集或下游业务系统；边则对应数据共享、特征同步或样本构建关系。删除通知需要沿着这些边传播，并通过安全验证和回执机制确认每个相关节点都完成了处理

TrustNotify 的核心价值在于轻量化。AI 系统每天可能产生大量数据流转和删除请求，如果每一次删除通知都采用重型加密、全量验证或人工审计，系统成本会很高，业务系统也难以长期接入。因此，TrustNotify 使用 HMAC 逐跳验证、递归回执、概率抽样验证和可信执行环境，使删除通知在保证可信性的同时尽量降低计算和通信开销

在面向 AI 的叙事中，可以把 TrustNotify 包装为连接“用户删除权”和“AI 模型数据清理流程”的中间层。它不直接回答模型参数如何遗忘，而是回答：哪些 AI 数据处理节点需要收到删除指令？这些指令是否真的传到了？回执是否可信？系统能否在高并发删除请求下保持较低开销？
