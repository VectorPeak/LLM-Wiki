10. Langchain里拿到这个流式响应，怎么实现的，使用的一些什么协议？  
     （涉及的点很多：Langchain的runnable, Python的异步迭代器，以及传输协议SSE，Websocket, Http Chunked。如果都弄得很明显需要花一些功夫，时间不够的同学可以先理解掌握本题，具体深入的细节可以先放一放）  
     在 LangChain 中，所有核心组件（模型、链、Agent 等）都被统一封装成一个抽象类 `Runnable`，它定义了标准调用接口，包括 `invoke()`（一次性输出）和 `stream() / astream()`（流式输出）。流式输出的这两个方法通过 Python 的异步迭代器，让开发者可以用 `for` 或 `async for` 循环逐块消费生成的内容，实现“边生成边处理”的能力.

 更底层实现分为两层：  
 • **后端内部**：LangChain 迭代器从底层模型获取 token 或片段，前提是模型本身支持流式生成（如 OpenAI 的 `stream=True`）。  
 • **后端对外传输**：将这些 chunk 推送给客户端，常用三种协议：  
  a. **SSE (Server-Sent Events)**：基于 HTTP 长连接，单向推送，浏览器用 `EventSource` 接收；  
  b. **WebSocket**：双向实时，适合需要交互（如停止生成）；  
  c. **HTTP Chunked**：普通 HTTP 分块传输，前端用 `ReadableStream` 读取.  
 客户端在 Python 或后端服务中消费流式响应时，本质是遵循迭代器语法，用 `for chunk in runnable.stream(...)` 或 `async for chunk in runnable.astream(...)` 来一点一点接收并处理数据。

11. 如何提高获取Agent响应的并发度？  
     （这个问题就很空泛，可以回答的点很多。和大模型无关，是传统开发的并发策略方法。首先针对于服务端/客户端有所不同）  
     客户端的话这里并发大概是同时调用多个Agent API获取结果，是一个I/O操作，不需要密集CPU计算，那么可以从线程/协程/进程/线程池/消息队列 等典型的技术切入和面试官讨论。这里可以看一下这篇文章  
     [https://blog.csdn.net/mo_62006803/article/details/134862520](https://blog.csdn.net/mo_62006803/article/details/134862520)  
     如果是服务端的话，可能要构建一个Agent网关或者平台，需要考虑一些Worker池，负载均衡之类的问题）

---

这里就以客户端来回答。  
在客户端，如果需要同时调用多个 Agent 的 API 来提高并发度，常见的并发方法包括：多线程、多进程、协程、消息队列、线程池等.

由于调用远端 API 属于 I/O 密集型操作，因此：  
• 多进程并不适合这种场景，因为它更适合于 CPU 密集型任务，而且进程切换的开销较大。  
• 如果系统结构简单、并发量不高，可以使用**多线程**来快速实现并发。  
• 如果追求更高的并发效率，推荐使用**协程**，因为协程更加轻量，资源占用更低，适合处理大量并发的 I/O 请求.  
• **消息队列**则更适合用于任务解耦、异步处理和高并发任务调度的场景。它通常用于构建更复杂、可扩展的系统架构。

 **MCP 标准协议本身就定义了两个关键机制**：  
 1. **版本协商（Version Negotiation）**  
  • 在连接初始化阶段，客户端和服务器会交换它们支持的协议版本列表，并选定一个双方都支持的版本。如果没有交集，连接会优雅失败。

 2. **能力协商（Capabilities Negotiation）**  
  • 握手时，双方会声明自己支持的功能（如 `sampling`、`resources.subscribe` 等）。客户端可以根据服务器返回的能力决定启用哪些功能，避免调用未实现的能力。  
  所以，客户端在初始化握手后就能拿到：  
   ○ 协议版本（是否兼容）  
   ○ 支持的能力列表（哪些功能可用）  
  如果发现不匹配，客户端可以：  
   ○ 回退到兼容版本（如果支持多版本）  
   ○ 禁用不支持的功能（避免调用导致错误）  
   ○ 或直接提示用户“当前服务端不兼容，需要升级或切换”

▶ **4. GPT在2025年出了一个叫Response的API，你大概了解吗？**  
（关于GPT的Response API，其实是在25年3月份OPENAI就提出了，希望将llm的工具调用、多轮对话等功能整合到api中，减少Agent开发者的负担。网上资料不多，面试也几乎从未出现。他都推出了大半年了，但是看上去这个思想没有火起来，估计可能也不会火起来了。我是第一次被问到。所以针对于这个知识点，我的建议是：大致了解下，不用花太多功夫。所以我尝试用下面的回答大概简单概括下它的思想，理解这个回答，下次万一真被问到，能说道几句就好了，并且我严重怀疑这个基本面试不会问到）

**Response API 是 OpenAI 设计 LLM 调用接口的从 `Completions` → `Chat Completions` → `Responses` 的过渡**：

- 早期的 `Completions` 只面向纯文本续写；
- 随后 `Chat Completions` 围绕 `messages` 支持对话与函数调用，但状态与工具编排仍主要在客户端；
- 如今 `Responses` 用一个**有状态的统一端点**，把多轮上下文与多步工具调用纳入服务端协议。

**设计思想与理念**：以 **“一次请求里的智能体循环”** 为核心，把 **状态、工具、事件流与结构化输出** 写进端点契约（而非让客户端手工拼装），让模型在单次响应生命周期中：  
→ 按需调用托管/外部工具  
→ 跟踪步骤并输出可验证结果  
→ 从而标准化智能体能力、降低工程胶水、提升可观测性与可扩展性。

▶ **6. 你里面提到的对性能做测评，里面有一些bool类型指标，也有一些过程指标，这分别是怎么评判的？**  
（同学们，这个问题不清楚的话，会去看一下八股部分：**Agent开发框架**，我给大家推荐的这一章的参考资料 _Agentic Design Pattern_，里面就讲了这个问题，Agent评估问题，我推荐的两个资料，都是很有用的哈，这里也验证了，希望你能回到八股部分，看看我的笔记和推荐的资料，以下回答几乎来自于我的参考资料。）

- **对于 bool 类型的结果评估**：要求智能体响应必须**完全匹配预期输出**时才成功（如日期、布尔值、ID等），即“全有或全无”。
- **对于过程指标**：需使用更先进的 NLP 技术：
    - 字符串相似度：Levenshtein（编辑距离）、Jaccard（集合重合度）
    - 关键词分析 + 嵌入模型语义相似性（如 text-embedding-3-small）
    - LLM-as-a-Judge：构建专用 Prompt，由更强模型（如 GPT-4o）判断是否满足业务关键事实（如“Q1目标”“资源分配”）

> 📌 示例：在邮箱项目中，`email_to_event` 功能需从非结构化邮件中提取结构化日程信息。针对不同字段采用分层评估策略：
> 
> 1. **确定性字段**（date, time, participants）→ ISO 标准化后直接比对
> 2. **短文本语义字段**（location, subject）→ Levenshtein + Embedding 阈值（>0.85）
> 3. **长文本与意图字段**（description）→ LLM-as-a-Judge + RAG 相关性指标

 **动态注入的思路是**：所有 Plugin 预先注册到 Kernel 中，但在每次调用 LLM 时，通过**过滤器机制**指定当前允许使用哪些 Plugin. 这样，虽然 Kernel 中存在全部工具，但 LLM 在该次调用中只能感知和使用被筛选出来的工具子集。

 **参考链接**：[https://github.com/fzy2012/rhzl-Agentic-Design-Patterns-cn/blob/feature/interactive-website/13-Chapter-07-Multi-Agent-Collaboration.md](https://github.com/fzy2012/rhzl-Agentic-Design-Patterns-cn/blob/feature/interactive-website/13-Chapter-07-Multi-Agent-Collaboration.md)

 在使用中心化多Agent架构时，系统复杂度会随着 `r`（主管和子Agent的回合数）、`n`（子Agent数量）、`k`（子Agent迭代代数）增大。`n` 增大时，系统复杂度增加，主管Agent确实会成为瓶颈，这也是这种架构的弊端.

 为了解决这个问题，首先我们可以思考：是否一定要用这个架构？  
 • 比如动态环境下（网络导航/检索）→ 更适用去中心化，不存在单点瓶颈。  
 • 强推理、长推理链任务 → 更适用于单Agent。

 若确实需用中心化架构，可采取以下措施缓解单点Agent压力：  
 1. **动态负载均衡**：主管不直接管理所有Agent，而是通过调度池/任务队列分发任务，调度器负责高并发、自动化调度。  
 2. **采用分布式系统**：将Agent跑在多个节点，提高并发与容灾性.  
 3. **混合型架构**：保留中心化骨架，但将顶层Agent压力从子Agent个数 `n` 降级到小组数 `g`（如按职能分组），减缓顶层压力.  
 4. **控制 `k`, `r`**：单点Agent性能瓶颈与 `r`, `n`, `k` 相关，设计时应尽量减少交互轮次与迭代代数。

 最后的总结部分，我也引用了UC Berkeley的论文做总结，丰富了回答，也侧面像面试官展示了我们在关注前沿知识/论文的能力，关于这个论文，可以参考：  
 [https://36kr.com/p/3608851367871752](https://36kr.com/p/3608851367871752)

 **对于这个问题，核心的思想是有两个：提升模型的能力，和降低任务的难度。**

 首先讲**提升模型的能力**，我们可以通过：  
 1. 通过微调，强化学习让模型能够在指定任务上取得更好的效果。  
 2. 降低任务的复杂性。Agent系统中LLM的难度主要体现在3个方面：  
  • **推理维度**：Agent是否可以在部分可观测、信息不完整的环境中，将线索整合并做出正确抉择。  
  • **工具维度**：在众多工具依赖图中，Agent是否可以选择正确工具，使用正确参数。  
  • **交互维度**：在多轮对话中，是否能够主动识别意图，主动澄清问题，维持上下文一致.

 针对于小模型，我们可以通过**任务设计的角度**，比如拆分成任务、限制步骤、减少工具推理难度等方法，去避免这三个维度的难度，让任务复杂性下降，使得使用小模型也能够取得很好的效果。

 值得一提的是，我看过UC Berkeley 25年12月发表的一篇论文，结论是：即使在目前生产级别的Agent项目中，也通常将Agent步骤限制在10步以内，也会使用比如添加抽象层的方法降低工具推理难度，所以这些方法其实不止在小模型上需要使用，在大模型的项目中一样需要。

▶ **8.1 这个冲突怎么能让模型解决的很好，还是说你一股脑塞给模型，都让他来处理呢？怎么样能让模型整体的质量比较高呢？**  
 （其实这个问题的思路和第7个问题有点像，还是可以从Infra提升模型能力以及工程角度，降低模型推理难度这两个角度去回答，另外加了一点提示词工程的回答，其实这问题也没有标准答案，就是思维要灵活，和面试官讨论）

 首先，在不考虑性能、成本的角度，我们先考虑**提升模型的能力**：现在模型的能力越来越强，我们先去选择更强更大，更合适的模型看是否能解决这个问题。或者通过后训练的方式，比如使用SFT，输入一些冲突检测相关的数据，强化模型对这个指定任务的回答准确性能。

 如果模型确实无法处理，从**工程的角度**，我们可以：  
 1. 从LLM层解耦出一些具体的任务。模型任务处理的复杂度主要体现在推理、任务调用、交互的维度。我们可以抽离出一些具体的任务，比如之前判断冲突的时候，是让模型自己去调用距离计算api，然后做时间加减法。这时我们可以抽离这个任务，在语言层面调用工具计算好距离，然后以提示词形式填充进去。降低了模型理解工具的负担，另外模型不擅长做精确的时间加减法，我们也把这一任务交给了语言层面去做，不要把所有压力都给模型，而是让整个系统在人为定义好的任务流程图的格子里去填空。

 2. 从**提示词工程角度**，可以使用COT（Chain of thoughts）来完成，抽象出具体的任务，让模型一步一步思考。

▶ **8.4 什么时候用多Agent去做，什么时候用一个Agent去完成？举具体例子说明。**  
 （这题在八股部分多Agent地方已经总结得很好了，甚至是很多次，我下面回答基于八股部分的知识，我这块不用参考任何资料就能直接回答，如果你觉得这个问题还回答不上来，那就是对应的八股笔记部分的多Agent部分和参考资料没理解好。以下这个回答都是基于我八股的笔记和参考资料。）

 **Single-Agent 的优势**在于：不需要承担Agent之间的协调、通信开销和拓扑传播带来的误差扩散，在顺序调用、工具密集、单Agent基线收益高（达到45%+）时，使用Single-Agent效果会更好。例如典型的“工作流”例子（如邮件→行程→通知），这种情况下使用多Agent性能反而会下降。

 **Multi-Agent 的优势**在于：在复杂的任务中，特别是需要**分工和并行**的问题上，可以突破单Agent的能力边界，提高效率和鲁棒性。比如：  
 • 金融推理任务（需多个专家Agent协同验证）  
 • 网页搜索任务（更适合去中心化的多Agent架构）

▶ **8.8 怎么避免一些极端不合理输出的corner case？**  
 （我不能说这个是标准答案，这个答案是根据学习笔记中llm的参数 和 如何避免Agent出现幻觉 这两章内容总结的。我想说的是理解了这个笔记的很多内容，在遇到一些新问题时我们也能灵活应对）

 模型的不合理输出，通常是因为模型生成的回答置信度比较低导致的. 我们可以通过：  
 • 强化学习的方法，让模型在对于低置信度的时候来进行一个兜底的回答，而不是一个不靠谱的回答；  
 • 或者是通过修改模型的生成参数，降低温度（temperature）、减少topK，减少模型输出极端Case的情况。

 最后其实可以设置LLM生成后的检查，我们对LLM最后生成的结果再进行一次筛查，如果有比如极端言论或者完全不能接受的回答，直接拦截，给用户一个提示，现在很多Agent产品其实也都这么做。

▶ **9. 在项目过程中，怎么去做提示词的压缩和裁剪？**  
 （这个我觉得挺好的，有哪一些工程手段减少提示词，有点像提示词工程，但基本上提示词工程都是讲如何写更有效的提示词而不是怎么压缩。但这个问题说实话还是第一次见到，答案也不难，所以我认为没有很具备典型性，我就不把它抽象到八股笔记中，就把答案放在这里，大家阅读，理解一下就好）

 从工程的角度说，我们需要保持在描述清楚问题的前提下，尽量减少提示词/输入Token的消耗，可以使用以下策略：  
 1. 使用一些**结构化语言**，比如Jason格式、编号格式来简洁分明地描述事实、规则和要求，避免冗长的自然语言赘述。  
 2. 对于一些上下文内容，使用**压缩**、摘要方式保留下文，而不是保留原文.  
 3. 我们在设计提示词的时候，尽量将**静态内容放在最前面**，将变量放在后面。这样可以让kv-cache的机制尽量命中缓存，复用KV块.  
 4. 检查提示词是否有重复、冗余、噪音。只写AI不知道的内容，它推测不出来的内容。比如代码中要用UV不用PIP，而不是写一些AI知道的宽泛内容（如系统目录结构——它自己能推理出），或者请遵循现有Coding风格（AI训练时就知道）。写这些废话和宽泛的话，反而会增加AI的推理难度。  
 （这个观点可以看笔记文档部分：_Evaluating AGENTS.md – Are Repository-Level Context Files Helpful for Coding Agents?_）
 
**9.1 我提到了kv-cache可以避免输入token的重复计算，面试官问你从工程的角度上来说如何能让kv-cache更好的发挥作用？有没有一些可能，你的工程使用不当导致kv-cache并没有生效？**  
 （因为我上面提到了Kv-cache，kv-cache其实有点像是Infra的知识，不过即使做工程，还是建议一下Kv-cache是什么，有啥用，其实也不难。

 回答其实就是上面的第三点，面试官问这个问题原因是因为我提到使用kv-cache，但是没说Kv-cache在工程层面怎么用。

✅ **工程层面关键实践**：

- **序列长度对齐**：避免因padding导致KV缓存碎片化（如batch内序列长度差异过大时，启用`flash attention`的`causal mask`优化）
- **分组注意力（Grouped Query Attention, GQA）**：减少KV缓存显存占用（Q: 32 heads, K/V: 8 heads → 显存降75%）
- **PagedAttention（vLLM核心）**：将KV缓存按物理页管理，支持非连续内存分配，解决长上下文OOM问题
- **缓存压缩**：对KV值进行int4量化（如MLC-LLM方案），牺牲<1%精度换取50%+显存节省
- **过期淘汰策略**：对长时间未访问的KV块执行LRU淘汰（尤其适用于对话场景中历史轮次）

> 📌 面试官真正在考察：你是否理解KV-Cache不是“开了就有效”，而是需配合**推理引擎调度策略**才能发挥价值。

     b. 独立但是需要限流：适用于任务很多，但是不能全部同时跑的（比如因为资源限制，API Rate Limit的场景）比如媒体体理解批量处理, Cron多Job并发执行

     f. 异步父子编排：主任务派生子任务，子任务完成后回调给主任务继续。

### **4. 你们怎么做监控的？**

> **常考问题，必须准备标准答案**  
> 讲Agent评估的理论笔记里有，包括还推荐了论文（美团龙猫的）。监控分**线上监控 + 线下测试集评估**：
> 
> - **线上监控**：推理延迟（P99 < 2s）、工具调用成功率（>99.5%）、幻觉率（通过规则+LLM-as-Judge双校验）
> - **线下测试集评估**：从三个维度设计：
>     1. **推理维度**：部分可观测环境下的决策准确率（如冲突检测正确率）
>     2. **工具调用维度**：工具参数匹配度（Jaccard相似度 >0.95）
>     3. **规划维度**：多步骤任务完成率（End-to-end success rate）

✅ **关键话术**：

> “我经常就用美团龙猫那个方法，从三个维度：推理、工具调用、规划维度上构建标准和面试官讲。这部分检测的部分，笔记里面有，并且面试真题也有，答案很多，大家可以自己看看笔记。”

### ▶ **5. 智能体的就怎么去做优化以及说你们做了哪些优化？**

> 面试官问此题本质是考察**工程闭环能力**，回答需覆盖三层：
> 
> **1. 架构选型**
> 
> - 如何选Agent架构？依据是我给大家推荐的架构论文（八股都有），讲清理论（如单Agent vs 多Agent适用边界）。
> 
> **2. 微调 & 强化学习 & 量化**
> 
> - 微调：LoRA + DPO 对齐用户偏好（如日历助手中的“早睡/晚睡”倾向）
> - 量化：Q4_K_M + FFN层4-bit压缩，输出层6-bit保精度
> - 推理加速：SFT后训练 + KV-Cache优化（见前文9.1）
> 
> **3. 上线迭代：数据飞轮 → 监控 → 再训练**
> 
> - **数据飞轮**：用户行为（接受/拒绝建议）→ 更新提示词 → 提升下次决策准确率

### ▶ **Q1: 子Agent之间怎么通信的？**

> **OpenClaw标准方案**：
> 
> - 不用Handoff（避免状态丢失）
> - Orchestrator spawn子Agent → 子Agent完成后通过**push-based announce**将结果（JSON格式）注入Orchestrator队列
> - **关键约束**：仅传递最终结构化结果（JSON），不传递推理链（防文档膨胀）
> - 控制上下文长度：子Agent私有session，Orchestrator仅保留摘要

> 📌 理论来源：参考八股《多Agent通信机制》

---

### ▶ **Q2: 为什么不让孩子Agent互相通信？**

> **根本原因**：**收益 < 成本**
> 
> - 估值任务中，子Agent职责高度独立（财报/新闻/报告/Call无交叉依赖）
> - 互相通信会引入：
>     - 开销增加（额外RPC调用）
>     - 错误传播（一个Agent出错导致全链路失败）
>     - 调试复杂度飙升
> 
> ✅ 例外场景：网页搜索（需Agent间协商关键词）→ 改用中心化调度。

---

### ▶ **Q3: 为什么选中心化而不是去中心化或单Agent？**

> **数据支撑**（Google论文结论）：
> 
> - 单Agent错误率：4.4×
> - 去中心化错误率：17.2×
> - **中心化错误率最低**（金融场景对准确性要求极高）
> 
> **业务适配**：
> 
> - 估值任务需**统一出口**（最终DCF结果必须由主Agent整合）
> - 子任务间存在**隐式依赖**（如新闻情绪影响财报解读权重）→ 需中心协调
