## 一、企业私有化部署大模型：2026年从选卡到上线的配置清单和避坑实录

去年3月，老板让我搞个”企业内部AI”。

要求很简单：数据不出内网，模型自己跑，钱别花太多。

结果我花了一整年。

我是做电力行业AI落地的，自建GPU集群，2×RTX 4090的服务器跑了快一年。我们这行特殊——完全内网、断网环境，模型要本地跑，硬件要自己买。跟你在家里跑个Ollama完全是两个世界。

今天把这一年的选型经验整理出来，从显卡到网络到模型到框架，踩过的坑全列上。如果你也在做企业私有化部署，可以直接参考。

---
**先说结论：企业私有化部署，不是”买台电脑装个模型”就完事了。**

选卡、服务器、网络、推理框架、模型选型、监控运维、高可用，哪个环节掉链子都会翻车。而且跟个人部署最大的区别是——你的客户在用，挂了你得兜底。

下面我按实际遇到的业务需求来讲，不按预算分档。企业买硬件看的是”我要干什么”，不是”我有多少钱”。

---
**知识库问答——入门级，烂大街了**

90%的企业第一个需求就是这个。把文档喂给大模型，员工问问题，模型回答。在电力行业，这玩意儿已经是烂大街产品了——随便搜搜”电力知识库”能出来几十家。

虽然真做出来还是有很多边界case要踩（我踩了四个大坑：[企业知识库搭建踩了四个大坑](https://www.zhihu.com/question/650032076/answer/2023757075717660994)），但上面已经这么定位了，你也不好拿知识库去报项目。

所以这个场景的策略：**搭起来能用就行，别花太多精力**。真正的价值在后面的场景上。

硬件也简单。一张RTX 4090 24G，配个双路CPU、64G内存、2TB固态，总共3~4.5万搞定。模型用Qwen3.5-27B-Claude-Distilled系列，AWQ量化后不到20G，24G卡稳稳跑。这是用Claude蒸馏出来的27B模型，回答质量比原版Qwen3.5-27B强不少。我生产环境跑的是v2，v3（Qwopus3.5-27B-v3）最近也出了，拿Claude Code写个程序都没问题。还有个更猛的——Qwen3.5-27B-Claude-4.6-Opus-Reasoning-Distilled-v2，直接用Claude 4.6 Opus蒸馏，推理能力又上一截。

但别急着买卡。

先说个我自己的翻车故事。6张魔改48G的4090跑了半年，我写了个回答说”故障率0”。发出去两周，生产环境一张卡直接死了——GPU固件处理器超时崩掉，客户反馈”问任何问题都没有回复”。

详细翻车过程：[魔改4090用了一年，开发真香，生产翻车了](https://zhuanlan.zhihu.com/p/2028142360152817950)

所以如果预算宽裕，直接上PRO6000 96G，6万出头，正规产品有质保。一张卡顶两张魔改卡，96G显存跑27B绰绰有余，还能留大量空间给KV cache做高并发。

---
**然后你会发现知识库不够用**

客户用着用着就提新需求了——”能不能帮我查一下上个月110kV变电站跳闸次数最多的前五个站是哪些？”

这就是Text-to-SQL。用户用自然语言提问，系统翻译成SQL去查数据库，再把结果整理成回答。比知识库重一档，对模型推理能力要求高得多。

27B勉强能跑，但准确率上不去。我之前从60%干到85%花了三个月：[Text-to-SQL准确率从60%干到85%？我踩了三个月的坑](https://zhuanlan.zhihu.com/p/2024359354514556166)

硬件上，两张4090做Tensor Parallelism，vLLM部署，模型切成两半各跑一张卡。服务器配128G内存、万兆网卡，总共7~10万。

但多卡有坑——后面细说。

不想折腾多卡的话，有个更好的选择：**Qwen3.6-35B-A3B**。MoE模型，总参数35B但每次只激活3B，显存占用极低，单卡就能跑。而且它也有Claude蒸馏版，推理质量比27B蒸馏版还强。我服务器上就放着这个和它的蒸馏版，日常推理基本都用它——推理速度极快，性价比碾压所有稠密模型。

所以我的建议是：如果预算够，一张PRO6000 96G比两张4090省心太多。不用折腾多卡通信，不用担心TP崩掉，单卡稳定性碾压。

---
**再往上就是大规模并发了**

给全公司几百人同时用，或者对外提供API。这种场景最吃硬件。

这时候就不是买一台机器的事了。3台服务器，每台2×4090 48G（或者2台×2×PRO6000），加上128~256G内存，配200G甚至400G InfiniBand网络，机柜配电UPS全套。总共30~50万。

对，网络可能是最贵的那个。

说句实话，100G也未必够。多机多卡推理，模型参数要在GPU之间来回搬，100G只是”能用”的级别。预算够的话200G/400G InfiniBand才是正解。我用100G纯属……预算超了。（网络这块真的别省，后面越加机器越痛苦）

我从万兆升级到100G的翻车故事：[从万兆到10万兆：GPU集群网络升级翻车+救火实录](https://zhuanlan.zhihu.com/p/2023715937958766315)

---
**GPU一定会挂——高可用不是可选项**

这是很多教程不告诉你的事。GPU故障不是概率问题，是时间问题。

魔改卡跑了半年”零故障”，说死就死。正规卡也一样——消费级4090没有ECC显存，显存位翻转虽然罕见但不是零。风扇停转、散热硅脂老化、电源波动，都能让一张卡突然不响应。

**单卡部署就是单点故障。** 你要上生产，必须做冗余。

思路很简单：Data Parallelism。同一份模型部署在多张卡上，前面挂Nginx做负载均衡加health check，一张卡死了自动摘除，流量切到其他卡，用户无感。这就是N+1冗余。

举个例子：27B模型需要1张卡，那我部署2份，分别在不同的物理机上（避免一台机器挂了两个实例一起完蛋）。前面的Nginx做round-robin，加上健康检查。可用性要求更高的上N+2。

热备还是冷备？我们两种都试过。热备就是冗余实例一直跑着，零中断。冷备是备卡不加载模型，故障时自动拉起，中间有30秒到1分钟的模型加载时间。客户能接受冷备的话，省一半显卡开销。

监控也得提前建好。Prometheus抓GPU指标——温度、显存、功耗、Xid错误码，Grafana看面板，AlertManager配告警。GPU挂了：检测→摘除故障节点→拉起备用→通知运维。别等客户凌晨三点打电话说”系统坏了”你才发现。

---
**然后说说踩过的坑**

PCIe带宽这个我踩得最惨。三台服务器，6张4090全部运行在PCIe 1.0，带宽只有额定的8%。我跑了半年benchmark，一直在”残血”状态下测数据，还纳闷怎么性能跟别人差这么多。原因特蠢——BIOS里PCIe Link Speed被设成了Gen1。（服务器出厂默认值，谁能想到呢）改完以后模型加载速度直接快了10倍。[详细排查过程在这](https://zhuanlan.zhihu.com/p/2023279846856427316)

多卡推理的坑也说一下。Tensor Parallelism不是你想切几份就切几份——模型的注意力头数必须能被GPU数量整除。之前6张卡做TP=6直接崩了，64个注意力头不能被6整除（64÷6≈10.67）。改成TP=2或TP=4才行。（vLLM报错倒是挺明确的，但文档里确实没提这个限制）这种坑官方文档里找不到，全是跑上去报错了才知道。

推理框架的话，快速出原型用Ollama，一行命令拉起来。但生产环境别用。出问题了看不到日志，GPU挂了不会自动恢复也不会告警。我现在全换vLLM了。日志详细、GPU状态可监控、社区活跃。TensorRT-LLM理论上性能更好，但编译太慢、版本绑死，90%的场景vLLM够用。

---
**模型怎么选？别听博主吹，去看hf-mirror热榜**

模型更新太快了，我现在写的推荐两个月后大概率过时。所以与其听各个博主吹（包括我），不如自己去 [hf-mirror.com](https://link.zhihu.com/?target=https%3A//hf-mirror.com/) 看热榜。HuggingFace国内镜像，首页Trending Models榜单，大家用什么、下载数多少，一目了然——用脚投票的结果，比任何评测都真实。

我写文章这会儿刚去看了一眼，MiniMax 2.7排在榜一。更新太快了。

我选模型的逻辑：先上hf-mirror看热榜前10，圈出参数量在你显卡能跑的范围内的，然后去试几个，看回答质量和速度哪个更适合你的场景。

2026年4月企业私有化部署关注度最高的几个：Qwen3.5-27B-Claude-Distilled系列（v2/v3，我生产环境跑的）；Qwen3.6-35B-A3B（MoE，35B总参只激活3B，性价比之王）；Gemma4 27B（Google新出的，24G显存，猛但稳定性待观察）；Seed-OSS-36B（字节新出的，可以关注）；还有榜一的MiniMax 2.7，我还没来得及试。

**把hf-mirror加到书签栏，每个月看一次，比看评测文章管用。**

---
**掏心窝说几句**

渐进式投入，别一步到位。这是我踩过最大的认知坑——总想着一次性把硬件配齐，结果项目还在报批阶段就因为预算太大被砍了。现在我做配置一律用4卡或8卡机箱，先配1-2块卡，快速把业务搭起来，让领导看到效果。后面随着业务深入再加卡，决策快、迭代迅速。机箱和电源按满配选好，留够升级空间，卡可以慢慢加。

先想清楚业务场景再买硬件。别上来就买最贵的卡，知识库问答一张4090就够了，买A100是浪费。

稳定性大于性能大于价格。开发时图便宜用魔改卡，生产时翻车了再补PRO6000的钱，还不如一开始就买对。

网络别省。万兆勉强够，100G能用，200G/400G才是王道。我选100G纯粹因为预算超了，你别学我。

监控提前做，冗余提前备。部署完第一时间接Prometheus+Grafana，生产环境至少N+1冗余。单卡部署就是给自己埋雷。

数据安全是底线。电力、金融、政务这些行业，数据不出内网是硬性要求，私有化部署不是可选项，是唯一选项。

## 二、一行@Transactional锁了28万行数据14小时，差点让运维背锅

>  声明：本文基于真实生产事故改编，为脱敏需要，对业务场景、数据规模、时间线等细节做了调整。但事件性质、故障根源、技术原理完全真实。

### 系统挂了14个小时，运维差点背锅
去年我在一家电商公司做后端开发，年底大促前两天，系统挂了。

不是那种”响应变慢”的挂，是**彻底瘫痪**——接口全超时，定时任务全报错，最后连注册中心都只剩心跳在蹦。

运维同学凌晨被叫起来重启服务，好了半小时又挂。反反复复折腾了快14个小时，领导差点以为运维操作失误。

结果呢？不是运维的锅，不是数据库的锅，不是Redis的锅。

是**一行`@Transactional`注解**的锅。

准确说，是一个`@Transactional`加在了不该加的地方，配合CGLIB代理的机制，把一个本应毫秒级的小事务，膨胀成了一个锁28万行、持续十几秒的巨型事务。然后这个巨型事务被定时任务反复重试了10次。

14小时。28万行锁。

说实话查出来的时候我自己都愣了。

（后面我把完整排查过程写出来，这个坑太隐蔽了，不写出来对不起那14个小时）

---
### 先说背景
那年我们做年终大促活动，有个批量更新用户订单状态的功能。业务逻辑大概是这样——先更新一批上游状态码，然后根据状态码更新订单表的统计字段，最后再批量更新用户维度的聚合数据。

代码结构长这样：
```
@Transactional
public void refreshOrderStats() {
    // 第1步：更新上游状态码（编程式事务，自己控制提交）
    updateStatusCode();

    // 第2步：更新订单关联字段
    updateOrderRelations();

    // 第3步：批量更新用户维度的统计
    batchUpdateUserStats();  // 这个方法会更新28万行
}
```
看起来没啥问题对吧？

第1步里我们用了编程式事务（`TransactionTemplate`），自己控制了事务的开启和提交。第3步也是分批的，每批300行，心想这样应该不会有长事务。

**但实际跑起来完全不是这么回事。**

---
### 事故是怎么发生的
大促前一天晚上8点，定时任务触发`refreshOrderStats()`。

这个方法的第3步`batchUpdateUserStats()`要更新28万多行用户统计数据。代码里确实做了分批，每批300行，每条SQL大概长这样：
```python
UPDATE user_order_stats
SET order_count = CASE user_id WHEN ? THEN ? WHEN ? THEN ? ... END,
    last_active = CASE user_id WHEN ? THEN ? WHEN ? THEN ? ... END
WHERE user_id IN (?, ?, ?, ... /* 300个值 */)
```
一条SQL 300个`CASE WHEN`分支，总共约1375条SQL。

分批了啊！300行一批，一批一批来，有什么问题？

问题在于——**这些SQL全在同一个Spring事务里**。

### 为啥？因为CGLIB代理
`refreshOrderStats()`方法上有`@Transactional`注解。Spring用的是CGLIB代理（因为方法是public的，类没实现接口），所以实际调用链是这样的：
```
定时任务触发
  → 代理类.refreshOrderStats()   ← Spring事务从这里开始！
    → 真实对象.refreshOrderStats()
      → updateStatusCode()        ← 编程式事务提交了，但外层事务没提交
      → updateOrderRelations()
      → batchUpdateUserStats()    ← 1375条SQL，全部在外层事务里
  → 代理类提交事务               ← 到这里才真正释放所有行锁！
```
你看到没有？虽然第1步里编程式事务自己提交了，但**外层的CGLIB代理事务还没提交**。第3步的1375条SQL全在这个外层事务里，从第一条SQL执行到最后一条SQL执行完，所有被更新的行都持有行锁。

而且这个外层事务只有在`refreshOrderStats()`整个方法返回后，代理类才会提交。

28万行锁，从第一条UPDATE开始一直持有到最后一条UPDATE执行完。整个过程持续了大约37秒。

37秒看着不长？但在一个高并发的系统里，37秒锁住28万行，足够把所有涉及这张表的查询全部阻塞了。

---
### 连锁崩溃的真实过程
说实话，单纯一次51秒的事务虽然离谱，但不至于让系统挂掉。真正要命的是后面发生的事。

**第一次锁超时之后，没人处理，然后它又来了一次。**

这个批量更新是通过定时任务触发的。第一次（晚上8点18分）锁超时失败后，定时任务在第二天早上6点多又触发了。又失败了。然后7点又来。9点多又来。

**8次。同一个操作被触发了8次。**

每次都是28万行的更新，每次都锁超时。没有重试上限，没有熔断，没有告警。8次失败累计大约300秒的锁竞争，在14小时内持续给数据库加压。

（你问我为什么没有告警？因为当时告警规则只监控CPU和内存，没人想到要监控锁等待时间。教训啊。）

到了第二天下午5点左右，数据库已经被这个毒事务持续折腾了快14个小时，响应速度已经明显变慢。然后——

下午5点，好几个重量级定时任务同时启动了。有的读数据库写缓存，有的做数据聚合，有的跑统计报表。平时没问题，因为数据库是健康的。但这时候数据库已经在锁竞争里挣扎了，这些任务一叠加：

- **17:50**：Redis 4个节点全部超时（30秒超时）。缓存不可用，所有请求直接打到数据库。
- **17:55**：OOM。`GC overhead limit exceeded`。JVM疯狂GC但回收不了内存。
- **18:16**：连接池耗尽。Druid连接池里所有连接都被占用，新的请求拿不到连接。
- **18:22**：大面积NullPointerException。36个线程同时在等连接，线程号已经排到300多号了。

服务彻底瘫痪。

直到第二天凌晨，服务只剩Eureka心跳还在蹦——每5分钟一条，证明这个服务还活着，但干不了任何事。

---
### 怎么查出来的
说实话排查过程挺曲折的。

一开始看到OOM和连接池耗尽，以为是内存泄漏。花了好几个小时翻GC日志和堆转储，发现确实内存不够用了，但不是内存泄漏——是几个定时任务同时加载了大量数据到内存，叠加Redis超时导致线程阻塞，内存释放不了。

然后开始往回翻日志，发现下午5点之后才OOM的，但数据库的异常从前一天晚上就有了。

翻到前一天晚上8点的日志，发现第一条关键信息：
```
Lock wait timeout exceeded; try restarting transaction
```
然后发现这条错误从晚上8点到第二天下午2点半，重复出现了8次。每次都是同一张表，同一个方法。

顺藤摸瓜看了调用栈，发现了`$$EnhancerBySpringCGLIB$$`这个类。哦，CGLIB代理。然后看到方法上有`@Transactional`。

**到这时候才反应过来——事务边界被代理类撑大了。**

写代码的人以为第3步是分批更新、每批独立提交。但实际上因为外层`@Transactional`的存在，所有批次都在同一个事务里。分批形同虚设。

当时的心态……怎么说呢，就是那种”原来是你小子”的感觉。一行注解，28万行锁。

---
### 怎么修的
修复方案其实不复杂，但搞清楚原因花了大半天。

核心就一件事：**把事务边界缩小到最小粒度**。
```python
// 修之前：整个方法一个Spring事务
@Transactional
public void refreshOrderStats() {
    updateStatusCode();
    updateOrderRelations();
    batchUpdateUserStats();  // 28万行全在一个事务里
}

// 修之后：去掉类级别的@Transactional，每批独立事务
public void refreshOrderStats() {
    // 第1步：独立事务，做完就提交
    transactionTemplate.execute(status -> {
        updateStatusCode();
        return null;
    });
    // 此时锁已释放

    // 第2步：独立事务
    transactionTemplate.execute(status -> {
        updateOrderRelations();
        return null;
    });

    // 第3步：每批50行，独立事务
    for (List<User> batch : Lists.partition(allUsers, 50)) {
        transactionTemplate.execute(status -> {
            batchUpdateUserStats(batch);
            return null;
        });
        // 50行的锁立即释放
    }
}
```
改完之后行锁持有时间从37秒降到**毫秒级**。28万行的更新变成了5000多个小事务，每个事务只锁50行，执行完立即释放。

同时加了几个配套措施：

- 重试上限3次（之前没有上限，反复重试10次）
- 锁超时第一次就触发告警（之前完全没监控）
- 重量级定时任务错开执行（之前全挤在同一个时间段）

---
### 我后来总结的排查方法论
说实话这个事故让我对Spring事务有了全新的认识。后来我整理了一套排查事务问题的套路，分享给大家。

**第一步：看InnoDB锁信息。**
```
-- 看当前正在等待的锁
SELECT * FROM information_schema.INNODB_LOCK_WAITS;

-- 看当前持有锁的事务
SELECT * FROM information_schema.INNODB_TRX
ORDER BY trx_started ASC;
```
如果看到一个事务从几分钟甚至几小时前就开始了还没结束，十有八九是长事务。

**第二步：看事务里执行了什么。**
```
-- 看当前正在执行的SQL
SELECT trx_query, trx_started, trx_tables_locked, trx_rows_locked
FROM information_schema.INNODB_TRX;
```
`trx_rows_locked`这个字段特别有用——如果看到几十万行甚至更多，那就是批量更新在长事务里了。

**第三步：看调用栈，找CGLIB代理。**

在日志里搜索`$$EnhancerBySpringCGLIB$$`或者`$$FastClassBySpringCGLIB$$`。如果看到这些类名，说明方法被CGLIB代理了，`@Transactional`注解在代理层创建了一个外层事务。

**第四步：画事务边界图。**

拿张纸，把方法调用链画出来，标清楚哪些地方开了事务、哪些地方提交了事务。很多时候你以为事务已经提交了，但外层代理的事务还没提交——这种”嵌套事务幻觉”是最容易踩的坑。

---
### 最后说几句
这个事故折腾了快两天才彻底解决。事后复盘，最让我后怕的不是技术问题——CGLIB代理的行为是Spring文档里写清楚的，学学就会了。让我后怕的是**排查的时间成本**。

14个小时的连锁崩溃，日志从几万行翻到十几万行。各种表象——OOM、连接池耗尽、Redis超时、NullPointerException——全都是”果”，不是”因”。真正的”因”藏在一行不起眼的`@Transactional`注解里。

如果没有系统地往回追溯时间线，如果只看最后几个小时的日志，你可能会得出”加内存就行”或者”Redis不稳定”的结论。然后加完内存过几天又挂了。

所以我现在有个习惯：每次遇到连锁故障，不管表象是什么，**先把时间线拉出来从最早的异常开始看**。最先出现的异常往往是根因。

还有就是——没事翻翻自己代码里的`@Transactional`注解，看看它到底管了多大范围。你可能会发现一些”惊喜”。

别问我怎么知道的。

## 四、Text-to-SQL准确率从60%干到85%？我踩了三个月的坑

Text-to-SQL准确率从60%干到85%？我踩了三个月的坑

我们公司去年上了个智能问数系统。领导的原话是”让业务人员用自然语言查数据,不用学SQL了”。

听起来很美好对吧？

三个月后,我看着后台数据:准确率60%。

是的,10个问题有4个答非所问。用户问”上个月各区域电费收入”,系统给他查成了”各区域用户数”。

(别问我为什么,大模型觉得这俩是同一回事)

---
### 第一个坑:表太多,模型懵了
我们的数据库有200多张表。

把200张表的schema全塞给大模型让它自己选,结果就是——它选不对。

客户说”查变压器负载率”,模型选成了”变电站信息表”。看起来沾边,但不是一回事。

**解决方法:加个表选择器**

不是让模型从200张表里盲选,而是先做个检索,把可能相关的5-10张表筛出来,再让模型选。

怎么筛?用向量相似度检索,把用户问题跟每张表的说明文字做个匹配。

效果立竿见影.准确率从60%提到了72%。

(别高兴太早,后面还有坑)

---
### 第二个坑:没示例,模型瞎写
用户问”查询功率因数低于0.9的变电站”,模型写出来的SQL是这样的:
```
SELECT * FROM station WHERE pf < 0.9
```
没毛病对吧?但我们数据库里功率因数字段叫`power_factor`,不叫`pf`。

模型怎么知道你字段叫啥?它又没见过你的数据库。

**解决方法:few-shot示例**

给模型塞几个同类查询的示例:
```
问题:查询电压异常的变电站
SQL:SELECT station_name FROM substation WHERE voltage_status='异常'

问题:统计各区域变压器数量
SQL:SELECT area,COUNT(*) FROM transformer GROUP BY area
```
让模型照猫画虎,准确率从72%提到了80%。

这招的难点是:你得整理示例.没捷径,人肉活。

(我当时整理了50个,眼睛都看花了)

---
### 第三个坑:SQL写对了,但执行报错
这个问题最坑。

模型写出来的SQL语法没问题,但业务逻辑有问题.比如:

- 用户问”昨天”,模型写的是`DATE='2024-01-15'`(硬编码日期)
- 用户问”北京区域”,模型写了`region='北京'`,但表里存的是`region_code='BJ'`

这种错误,语法检查发现不了,只有执行才知道。

**解决方法:SQL校验+重试**

加一层校验逻辑:

- **执行前检查**:关键字段有没有用错(日期字段是不是用了硬编码)
- **执行后检查**:有没有报错,返回结果是不是空

如果发现问题,把错误信息塞回给模型让它自己修。
```
你写的SQL执行报错:column "pf" does not exist
请检查字段名是否正确,已知功率因数字段为power_factor
```
模型一看,哦字段名错了,自己改过来。

这招加上,准确率干到了85%。

---
### 最后说两句
三个坑加起来,简单查询85%+,复杂查询60%。

复杂查询为啥还是60%?因为用户的问题本身就很模糊。

“帮我分析一下最近的负荷变化趋势”——这问题你让人来写SQL也得问清楚什么叫”分析”、什么叫”趋势”。

所以我们的做法是:简单查询自动化,复杂查询半自动化(机器出初稿,人工确认后执行).

Text-to-SQL不是银弹,但三板斧加上去,能用.
