# 来源与联网核对规则

本知识库的目标是减少幻觉，而不是把模型记忆包装成知识。Codex 在回答和写 Wiki 页面时必须区分：本地来源、外部来源、推断、假设。

## 必须联网核对的情况

以下信息输出前应联网核对：

- 当前版本、最新发布、API 行为、模型名称、软件依赖
- 新闻、政策、法律、价格、市场数据
- 项目是否仍维护、仓库状态、issue/PR 状态
- 论文发表状态、作者信息、会议或期刊信息
- 公司、组织、人物的当前职务或状态
- 使用者明确要求 Google、搜索、联网、核对

## 来源优先级

从强到弱：
1. 官方文档、项目仓库、论文原文、标准文本
2. 作者博客、发布说明、技术报告
3. 多个独立来源一致的二手资料
4. 单个二手资料
5. Codex 根据上下文做出的推断

推断必须明确标注，不能伪装成来源事实

## 派生文本不是来源

PDF 抽取文本、OCR 结果、网页正文缓存、分块文本和 embedding 输入都只是读取辅助物。它们可以帮助 Codex 处理资料，但不能取代原始来源。

- `source_path` 必须指向 `RAW` 中的原始文件或使用者提供的原始 URL。
- 缓存路径不得写入 `sources` 字段。
- 如果缓存文本与原始文件不一致，以原始文件为准。
- 缓存统一放在 `D:\LLMWiki\LLMWiki\Clippings\.llmwiki-cache\`，不得放入 `RAW`。
- `Clippings/.llmwiki-cache` 虽在 `Clippings` 下，但不是剪藏来源，不应作为 `RAW` 之外的事实来源。

## 本地来源与外部来源冲突

当 `Wiki` 中已有说法与新联网资料冲突时：

1. 不直接覆盖旧说法。
2. 记录冲突双方。
3. 比较来源日期、来源权威性、定义范围和语境。
4. 必要时在相关页面加入 contradiction callout。

## 链接规则

- 本地文件链接必须能直接点击到达。
- 外部网址必须使用完整 URL。
- Wiki 内部页面优先使用 Obsidian wikilink。
- 输出中引用外部资料时，应给出可点击来源。

## 自笔记的特殊地位

`RAW/02.Logs` 和 `RAW/03.Self-Notes` 都包含使用者自己的原始观察和想法。它们可以作为高优先级研究线索，但不是外部事实来源。写入 Wiki 时应区分：

- 使用者观察。
- 使用者假设。
- 已被外部来源支持的事实。
- 仍待验证的问题。

## 公司项目资料

历史说明：当前 RAW 分区已不再使用 `RAW/08.Business`；现行 `RAW/08` 为 `RAW/08.Leetcode`。旧业务页如仍存在，只能按历史 Wiki 页面处理，不能作为当前 RAW 分区规则。

`RAW/08.Business` 是旧结构中的公司项目资料。写入 Wiki 时应保持业务上下文、责任边界和决策链路：

- 不把会议记录直接等同于最终决策。
- 决策必须记录日期、背景、选项和理由。
- 需求必须尽量写清目标、范围、验收标准和依赖。
- 涉及公司内部敏感信息时，不主动联网搜索或外发内容，除非使用者明确要求。

## GitHub 与 GitMD-style 来源规则

GitHub 项目的权威来源始终是 GitHub 原仓库。Codex 本地生成的 GitMD-style 输出可以作为 `RAW/04.GitHub` 的规范化采集件，但当 GitMD-style 内容与 GitHub 原仓库冲突时，以 GitHub 原仓库为准。

写入 `Wiki` 时：

- `source_path` 可以指向 `RAW/04.GitHub/<owner>__<repo>.gitmd.md`。
- 重要事实仍应保留 `repo_url` 或来源摘要页中的 GitHub 原始链接。
- 涉及维护状态、license、release、issue、默认分支、commit 等易变化信息时，输出前必须联网核对。

## Leetcode 与 Wechat 资料

`RAW/08.Leetcode` 是算法训练资料。写入 Wiki 时应区分题目事实、解法推导、复杂度结论和个人总结；复杂度、边界条件和算法正确性应能从题目约束或标准算法分析中推出。

`RAW/09.Wechat` 是微信文章和中文技术长文资料。写入 Wiki 时应区分作者观点、面试话术、工程经验和可验证事实。涉及当前模型、工具、项目状态、版本、性能数据或行业判断时，必须联网核对，不能直接把公众号说法写成客观事实。
